## 一、网络分层、TCP/UDP 与传输层基础

### 1. 网络的五层/四层结构是什么？每层的作用和协议？

**答案：**

**一句话总结：**

- 五层模型里应用层管应用协议，传输层管端到端通信，网络层管路由寻址，链路层管相邻节点传输，物理层管比特流传输。

**五层模型（教学模型）：**

| 层次        | 作用     | 协议/设备             |
| --------- | ------ | ----------------- |
| **应用层**   | 提供应用服务 | HTTP、DNS、FTP、SMTP |
| **传输层**   | 端到端通信  | TCP、UDP           |
| **网络层**   | 路由寻址   | IP、ICMP、路由器       |
| **数据链路层** | 相邻节点传输 | Ethernet、ARP、交换机  |
| **物理层**   | 比特传输   | 网线、光纤、集线器         |

**四层模型（TCP/IP模型）：**

```
┌─────────────┐
│   应用层    │ ← HTTP, DNS, FTP
├─────────────┤
│   传输层    │ ← TCP, UDP
├─────────────┤
│   网络层    │ ← IP, ICMP
├─────────────┤
│ 网络接口层  │ ← Ethernet, ARP
└─────────────┘
```

**数据封装过程：**

```
应用层数据 → [TCP头|数据] → [IP头|TCP头|数据] → [MAC头|IP头|TCP头|数据|FCS]
```

**这些常见缩写分别是什么：**

- `HTTP`：`HyperText Transfer Protocol`，超文本传输协议
- `DNS`：`Domain Name System`，域名系统
- `FTP`：`File Transfer Protocol`，文件传输协议
- `SMTP`：`Simple Mail Transfer Protocol`，简单邮件传输协议
- `TCP`：`Transmission Control Protocol`，传输控制协议
- `UDP`：`User Datagram Protocol`，用户数据报协议
- `IP`：`Internet Protocol`，网际协议
- `ICMP`：`Internet Control Message Protocol`，互联网控制报文协议
- `ARP`：`Address Resolution Protocol`，地址解析协议
- `MAC`：`Media Access Control`，媒体访问控制地址

### 2. TCP和UDP的区别是什么？

**答案：**

**一句话总结：**

- TCP 面向连接、可靠、有序，适合对正确性要求高的场景；UDP 无连接、开销小、实时性好，适合对时延敏感的场景。

| 特性       | TCP         | UDP        |
| -------- | ----------- | ---------- |
| **连接**   | 面向连接（三次握手）  | 无连接        |
| **可靠性**  | 可靠传输（确认、重传） | 不可靠（尽最大努力） |
| **有序性**  | 保证顺序        | 不保证        |
| **流量控制** | 有（滑动窗口）     | 无          |
| **拥塞控制** | 有           | 无          |
| **传输效率** | 较低          | 较高         |
| **头部大小** | 20-60字节     | 8字节        |
| **传输方式** | 字节流         | 数据报        |

**适用场景：**

- **TCP**：HTTP、文件传输、邮件、数据库连接
- **UDP**：DNS、视频流、游戏、`VoIP`（`Voice over IP`，网络语音）、直播

**为什么UDP适合实时应用：**

1. 无需建立连接，延迟低
2. 头部开销小
3. 丢包不重传，避免累积延迟
4. 可以在应用层自定义可靠性

### 3. TCP三次握手和四次挥手的过程？

**答案：**

**一句话总结：**

- 三次握手是双方确认收发能力并同步初始序列号，四次挥手是因为 TCP 全双工，双方关闭发送通道要分开进行。

**三次握手（建立连接）：**

```
Client                              Server
  |                                    |
  |──────── SYN (seq=x) ──────────────>|  1. 客户端发送SYN
  |                                    |
  |<─────── SYN+ACK (seq=y, ack=x+1) ──|  2. 服务端发送SYN+ACK
  |                                    |
  |──────── ACK (ack=y+1) ─────────────>|  3. 客户端发送ACK
  |                                    |
  |        连接建立，可以传输数据        |
```

**四次挥手（断开连接）：**

```
Client                              Server
  |                                    |
  |──────── FIN (seq=u) ──────────────>|  1. 客户端发送FIN
  |                                    |
  |<─────── ACK (ack=u+1) ─────────────|  2. 服务端确认
  |                                    |
  |         （服务端继续发送剩余数据）    |
  |                                    |
  |<─────── FIN (seq=w) ──────────────-|  3. 服务端发送FIN
  |                                    |
  |──────── ACK (ack=w+1) ─────────────>|  4. 客户端确认
  |                                    |
  |    客户端等待2MSL后关闭连接         |
```

**MSL / 2MSL 是什么：**

- `MSL`（Maximum Segment Lifetime）是**报文段最大生存时间**，表示一个 TCP 报文在网络中最多还能存活多久；超过这个时间，就应该认为它已经从网络里消失了。
- `2MSL` 就是两个 `MSL` 的时间。主动关闭方发送最后一个 `ACK` 后，不会立刻关闭，而是进入 `TIME_WAIT`，等待 `2MSL`。
- 为什么要等 `2MSL`：
  1. **确保最后一个 ACK 能被对方收到**。如果这个 ACK 丢了，被动关闭方会重发 `FIN`，主动关闭方在 `TIME_WAIT` 期间还能再次回 ACK。
  2. **让旧连接中的延迟报文自然消失**。这样同样四元组的新连接建立时，不会被上一个连接残留的旧报文干扰。
- 面试里可以直接说：`MSL` 是“一个 TCP 报文在网络中的最长存活时间”，`2MSL` 是主动关闭方在 `TIME_WAIT` 状态里的等待时长。
- `2MSL` **不是所有系统都固定 60 秒**。教材和面试里常用 “`MSL≈30s`，所以 `2MSL≈60s`” 来帮助理解，但实际时长由操作系统实现决定。

**状态变化：**

- 握手：CLOSED → SYN_SENT → ESTABLISHED
- 挥手：ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED

### 4. 为什么挥手是四次而不是三次？

**答案：**

**一句话总结：**

- 挥手通常是四次，因为收到 `FIN` 的一方可能还有数据没发完，所以 `ACK` 和 `FIN` 往往不能合并。

**原因：TCP是全双工通信，双方需要独立关闭各自的发送通道**

```
三次握手合并：
- SYN和ACK可以合并，因为建立连接时双方同时准备好

四次挥手不能合并：
- 服务端收到FIN时，可能还有数据要发送
- 服务端的ACK和FIN不能合并发送
```

**详细过程：**

```
1. 客户端发FIN：我不再发送数据了
2. 服务端发ACK：收到，但我可能还有数据要发
   （服务端继续发送剩余数据）
3. 服务端发FIN：我也不发了
4. 客户端发ACK：好的，连接关闭
```

**特殊情况（三次挥手）：**

- 如果服务端没有数据要发，可以将ACK和FIN合并
- 开启TCP延迟确认时可能出现

### 5. TCP半连接是什么状态？

**答案：**

**一句话总结：**

- 半连接就是服务端收到了 `SYN` 但还没收到最后 `ACK` 的状态，这类连接会占用半连接队列，过多时容易引发 SYN Flood 问题。

**半连接（SYN_RCVD状态）：**
服务端收到SYN但还没收到ACK，连接还未完全建立。

```
Client                    Server
  |──── SYN ─────────────>|
  |                       | SYN_RCVD（半连接状态）
  |<──── SYN+ACK ────────-|
  |                       | 等待ACK...
```

**半连接队列（SYN Queue）：**

- 存放处于SYN_RCVD状态的连接
- 默认大小：`net.ipv4.tcp_max_syn_backlog`

**SYN Flood攻击：**

```
攻击者发送大量SYN，不回复ACK
→ 半连接队列被占满
→ 正常用户无法建立连接
```

**防御措施：**

1. **SYN Cookie**：不保存连接状态，将状态编码在seq中
2. 增大半连接队列
3. 缩短SYN超时时间
4. 防火墙限流

### 6. TIME_WAIT状态是什么？为什么会出现大量TIME_WAIT？

**答案：**

**一句话总结：**

- `TIME_WAIT` 是主动关闭方为保证最后 `ACK` 可达并清理旧报文而保留的一段等待时间，大量短连接场景下很容易堆积。

**TIME_WAIT：** 主动关闭方发送最后一个 `ACK` 后进入的状态，持续 `2MSL`。也就是说，**等待 2MSL 的是主动关闭连接的一方**。面试里常按“约 60 秒”理解，但实际时长取决于操作系统实现。

**作用：**

1. **确保最后的ACK到达**：如果丢失，对方会重发FIN
2. **让旧连接的数据包消失**：防止影响新连接

**为什么会出现大量TIME_WAIT：**

- 服务器主动关闭大量短连接（如HTTP短连接）
- 高并发场景下频繁创建/关闭连接

**问题：**

- 占用端口资源（端口号只有65535个）
- 占用内存

**解决方案：**

```bash
# 1. 允许复用TIME_WAIT连接
net.ipv4.tcp_tw_reuse = 1

# 2. 快速回收（不推荐，可能有问题）
net.ipv4.tcp_tw_recycle = 1  # 4.12内核后移除

# 3. 使用长连接
Keep-Alive

# 4. 让客户端主动关闭
```

### 7. TCP 是如何保证可靠传输的？

**答案：**

**一句话总结：**

- TCP 的可靠传输不是靠单点机制，而是靠序列号、确认应答、重传、校验、滑动窗口和拥塞控制这一整套组合拳。

- 面试官您好，TCP 保证可靠传输并不是依靠单一的技术，而是通过一套**从数据本身、到收发双方、再到全局网络**的多维兜底机制来实现的。我主要从四个维度来汇报：
  
  **第一层：基础的数据完整与有序（解决乱序和损坏）**
  
  - **序列号与确认应答（Seq & ACK）：** 这是最核心的机制。TCP 会给每个字节的数据打上连续的编号（序列号）。接收端收到后，会回复一个确认号（ACK），告诉发送端“这个编号之前的数据我全收到了，接下来请发下一个”。这就保证了数据不重、不漏，并且能在接收端按序重组。
  
  - **校验和（Checksum）：** TCP 报文头部有校验和字段，如果数据在传输过程中发生比特位翻转或损坏，接收端校验失败会直接丢弃，等待重传。
  
  **第二层：异常情况的兜底（解决丢包问题）**
  
  - **超时重传：** 如果发送端发了包，但在一个动态计算的 RTT（往返时间，RTO）内没有收到 ACK 确认，就会认为包丢了，强制重新发送。
  
  - **快速重传：** 为了解决超时等待太久的问题，如果发送端连续收到 3 个相同的重复 ACK（说明某个特定的包丢失，而后面的包到了），就会立刻触发重传，不需要死等超时。
  
  **第三层：保护接收端（解决发得太快，接收端处理不过来的问题）**
  
  - **滑动窗口与流量控制：** 接收端在每次回复 ACK 时，会在 TCP 头部的 `Window` 字段带上自己当前“接收缓冲区的可用空间大小”。发送端会根据这个窗口大小来动态调整自己的发送速度，绝对不超过接收端能承受的极限。
  
  **第四层：保护整个网络（解决网络极度拥堵的问题）**
  
  - **拥塞控制：** 如果仅仅考虑收发双方，当整个网络链路已经严重拥堵时还拼命发包，会导致网络瘫痪。所以 TCP 维护了一个“拥塞窗口（cwnd）”，通过**慢启动、拥塞避免、拥塞发生（快重传/快恢复）**这几个核心算法，动态试探并适应当前网络的真实承载能力。”

### 8. TCP的拥塞控制算法有哪些？

**答案：**

先记一句最关键的话：

> **TCP 拥塞控制不是为了保护某一台机器，而是为了避免“整个网络被发爆”。它的核心思路是：先小心试探，再逐步放量，一旦发现丢包或拥堵迹象，就立刻收缩。**

---

#### 先把几个核心名词讲清楚

- `cwnd`：`congestion window`，**拥塞窗口**  
  发送端自己维护的一个窗口，用来控制“现在最多可以往网络里放多少未确认数据”。

- `rwnd`：`receive window`，**接收窗口**  
  接收方告诉发送方：“我这边还能接多少。”

- `ssthresh`：`slow start threshold`，**慢启动阈值**  
  用来决定当前是处于“慢启动”还是进入“拥塞避免”。

- `MSS`：`Maximum Segment Size`，**最大报文段长度**  
  可以理解成 TCP 一次正常发送的数据块大小单位。

真正发送时，发送窗口常常可以粗略理解成：

```text
发送窗口 ≈ min(cwnd, rwnd)
```

也就是说：

- 接收方太慢，要受 `rwnd` 限制
- 网络太堵，要受 `cwnd` 限制

---

#### TCP 为什么要有拥塞控制？

因为网络不是无限宽的。  
如果发送方只顾着疯狂发包：

- 路由器缓冲区可能被打满
- 丢包会变多
- 重传会变多
- 整个网络反而更慢

所以 TCP 不会一上来就把链路打满，而是会：

1. 先谨慎发一点
2. 看网络扛不扛得住
3. 如果没问题，再慢慢加大发送量
4. 一旦怀疑拥塞，就缩回去

这就是拥塞控制的本质。

---

#### 四个经典阶段到底在干什么？

#### 1）慢启动（Slow Start）

名字叫“慢启动”，但增长其实**一点都不慢**。  
它的特点是：

> **指数增长**

假设一开始：

```text
cwnd = 1 MSS
```

那么大致会这样涨：

```text
1 -> 2 -> 4 -> 8 -> 16 ...
```

为什么？

因为每收到一个 ACK，`cwnd` 都会增长；从 RTT 这个粒度看，就近似成“每经过一个 RTT 翻倍”。

#### 为什么要这样？

因为连接刚建立时，发送方根本不知道网络当前容量有多大。  
所以它先快速试探网络上限。

#### 什么时候结束慢启动？

常见两种情况：

1. `cwnd` 涨到了 `ssthresh`
2. 或者中间发生了丢包 / 超时

---

#### 2）拥塞避免（Congestion Avoidance）

当 `cwnd` 到了 `ssthresh` 之后，TCP 就不敢再指数翻倍了，而是改成：

> **线性增长**

大致可以理解成：

```text
每个 RTT 大约增加 1 MSS
```

比如：

```text
16 -> 17 -> 18 -> 19 ...
```

#### 为什么这时候变慢？

因为说明前面已经把网络探测到一个相对高的水位了，再指数涨就太激进，容易把网络打爆。

所以进入拥塞避免阶段后，TCP 会“更保守地继续加速”。

---

#### 3）快重传（Fast Retransmit）

如果发送方连续收到 **3 个重复 ACK**，通常意味着：

> 某个中间报文很可能丢了，但后面的报文已经到达接收方了。

这时 TCP 会认为：

- 链路还没完全断
- 只是某个包丢了
- 没必要傻等超时

于是它会：

> **立即重传那个疑似丢失的报文**

这就叫快重传。

#### 为什么是 3 个重复 ACK？

因为：

- 1~2 个重复 ACK 可能只是乱序
- 连续 3 个重复 ACK 更像是真的有包丢了

---

#### 4）快恢复（Fast Recovery）

如果已经触发了快重传，说明网络出现问题，但还没差到“完全发不动”。  
所以这时 TCP 不会像超时那样直接把 `cwnd` 打回 1，而通常会：

1. 把 `ssthresh` 调成当前 `cwnd` 的一半
2. 把 `cwnd` 也降下来（常见理解：减半附近）
3. 然后继续进入拥塞避免，线性增长

所以快恢复的思想是：

> **收缩，但别一下子退回解放前。**

这比“超时后从头开始”更高效。

---

#### 如果发生超时，会怎么样？

超时比“3 个重复 ACK”更严重。通常意味着：

- 网络堵得更厉害
- 或者包丢得更严重

这时 TCP 会更激进地收缩：

- `ssthresh = cwnd / 2`
- `cwnd` 直接退回很小（经典理解就是回到 `1 MSS`）
- 然后重新走慢启动

所以你可以这样记：

- **3 个重复 ACK**：说明还能抢救，走快重传 / 快恢复
- **超时**：说明更严重，直接大幅退让，重新慢启动

---

#### 用一个完整例子串起来

假设：

```text
初始 cwnd = 1 MSS
ssthresh = 16 MSS
```

那么可能的过程是：

```text
慢启动：
1 -> 2 -> 4 -> 8 -> 16

到达 ssthresh 后进入拥塞避免：
16 -> 17 -> 18 -> 19 -> 20

这时如果收到 3 个重复 ACK：
触发快重传 + 快恢复
ssthresh = 10
cwnd 大致降到 10 左右
然后继续线性增长：
10 -> 11 -> 12 -> 13 ...

如果不是 3 个重复 ACK，而是直接超时：
ssthresh = 10
cwnd 回到 1
重新慢启动
```

你真正要记住的不是每个实现的绝对数值，而是这个**趋势**：

> **先指数探测，后线性增长；轻度拥塞就减半继续走，严重拥塞就退回去重新试。**

#### 常见拥塞控制算法

##### 1）Reno

- 经典教材算法
- 重点就是你上面看到的：
  - 慢启动
  - 拥塞避免
  - 快重传
  - 快恢复

##### 2）Cubic

- Linux 上非常常见
- 更适合高带宽、长距离网络
- 它不是简单线性增长，而是用三次函数控制窗口增长速度
- 面试里知道它是 Linux 常见默认算法、适合高带宽场景，通常就够了

##### 3）BBR

- Google 提出的算法
- 不再只盯“丢包”来判断网络状况
- 更关注：
  - 带宽（Bandwidth）
  - RTT（Round-Trip Time，往返时延）
- 核心思路是尽量把发送速率控制在“接近链路瓶颈带宽、同时不过度排队”的位置

所以面试里可以简单说：

> Reno 更经典，Cubic 更适合现代高带宽网络，BBR 更强调基于带宽和 RTT 建模，而不是单纯把丢包当成拥塞信号。

---

#### 面试里推荐这样答

> TCP 拥塞控制的核心目标是避免把网络打爆。发送方会维护一个拥塞窗口 `cwnd`，先通过慢启动快速试探网络容量，再在拥塞避免阶段线性增长；如果收到 3 个重复 ACK，就触发快重传和快恢复，说明出现了丢包但网络还没彻底堵死；如果发生超时，则说明拥塞更严重，会把窗口大幅缩小并重新慢启动。经典算法是 Reno，现代 Linux 常见的是 Cubic，另外还有更关注带宽和 RTT 的 BBR。

**一句话总结：**

> TCP 拥塞控制本质上就是：**先试探、再放量；一旦怀疑拥塞，就立刻收缩。**

---

### 9. RTT 是什么？它对网络性能有什么影响？

**答案：**

**一句话总结：**

- `RTT`（`Round-Trip Time`，往返时延）反映的是一次请求来回的时间，它越大，握手、重传和同步请求的整体延迟就越高。

- RTT 就是一个数据包从发送端出去，再收到对端响应所花的往返时间。

- 它直接影响握手耗时、重传超时估计，以及很多同步请求的整体延迟。

- 所以带宽高不代表一定快：如果 RTT 很大，小请求一来一回还是会很慢。

- 对后端系统来说，RTT 高时常见优化思路是减少往返次数，比如连接复用、批量发送、流水线化，而不是只盯着带宽。

### 10. 如果一次通信本身就要 100ms，从建连到服务端第一次收到真正有用的数据，最少要多久？

**答案：**

**最终答案：`150ms`，也就是 `1.5 RTT`。**

这题的关键是先搞清楚：

- 题目里的 `100ms` 通常指的是 **RTT（Round-Trip Time，往返时延）**
- 如果 `RTT = 100ms`，那么**单程时延**就是：

```text
100ms / 2 = 50ms
```

而“服务端第一次收到真正有用的数据”，指的是：**服务端真正收到业务数据包的时间点**，不是客户端开始发数据的时间点。

#### 时间线拆解

```text
T = 0ms      客户端发送 SYN
T = 50ms     服务端收到 SYN
T = 100ms    客户端收到 SYN + ACK
T = 100ms    客户端发送 ACK，并且可以携带业务数据
T = 150ms    服务端收到 ACK + 业务数据
```

所以：

- **客户端角度**：到 `100ms` 时，它已经可以把第一个业务数据发出去了
- **服务端角度**：到 `150ms` 时，它才第一次真正收到有用的数据

因此标准答案应该是：

```text
服务端第一次收到真正有用的数据，最少需要 1.5 RTT = 150ms
```

#### 为什么不是 100ms？

因为 `100ms` 只是 **一个往返**：

- 第一次握手 `SYN` 到服务端：`0.5 RTT`
- 第二次握手 `SYN+ACK` 回客户端：再 `0.5 RTT`

到这里一共才 **1 RTT = 100ms**，此时只是客户端确认“连接建立了”，它这时才开始把真正的业务数据发出去。  
而业务数据再到服务端，还要再走 **0.5 RTT = 50ms**。

所以总时间是：

```text
1 RTT（握手确认到客户端） + 0.5 RTT（数据到服务端） = 1.5 RTT
```

#### RTT 是不是等于“前两次握手”？

**不完全等于，但在三次握手这个场景里，你可以近似这么理解。**

更准确地说：

- `RTT` 是一个**网络时延概念**，表示一个报文从发送到收到对端响应的往返时间
- 在 TCP 三次握手里：
  - 客户端发 `SYN`
  - 服务端回 `SYN+ACK`
- 这两步合起来，耗时刚好是 **1 个 RTT**

所以大家常说：

> TCP 建立连接本身就要先消耗掉 1 个 RTT。

但要注意，`RTT` 不是“握手专属概念”，它是所有网络交互都适用的通用时延指标。

#### 面试里推荐这样回答

> 如果题目里的 100ms 指 RTT，那么从客户端发起 TCP 建连，到服务端第一次收到真正业务数据，最少需要 1.5 RTT，也就是 150ms。因为前两次握手一共消耗 1 个 RTT，客户端在收到 SYN+ACK 之后才能把第三次 ACK 和业务数据一起发出，业务数据再到服务端还需要半个 RTT。

#### 额外加分点

如果面试官继续追问，你可以再补一句：

- **TCP 裸连**下，这题答案是 `1.5 RTT`
- **如果是 HTTPS**，还要再加上 `TLS` 握手开销
  - `TLS 1.2` 往往还要额外多个 RTT
  - `TLS 1.3` 会更快

### 11. 什么是 IO 多路复用？`select`、`poll`、`epoll` 有什么区别？`epoll` 是怎么实现的？

**答案：**

**一句话总结：**

- IO 多路复用就是让一个线程同时监听多个 socket 的可读可写事件；`epoll` 相比 `select/poll` 的核心优势是把“监听集合维护”和“就绪事件返回”分开了，避免每次全量遍历。

**先说 IO 多路复用是什么：**

如果服务端要同时处理很多连接，最笨的办法是：

- 一个连接一个线程
- 或者一个连接一个阻塞读

这样连接一多，线程数和上下文切换成本会很高。

IO 多路复用的思路是：

- 一个线程里同时监听很多 fd
- 哪些 fd 就绪了，内核通知你
- 应用只处理就绪的那些连接

所以它解决的是：

- 高并发连接下线程模型太重的问题
- 大量连接里只有少量活跃时的资源浪费问题

**`select` / `poll` / `epoll` 区别：**

| 机制       | 核心特点                                      | 问题                  |
| -------- | ----------------------------------------- | ------------------- |
| `select` | 用位图保存 fd 集合，每次调用把集合从用户态拷到内核态，返回后还要遍历全部 fd | fd 数量有限制，且每次都全量扫描   |
| `poll`   | 用数组保存 fd，没有 `select` 的固定上限                | 还是每次都要把集合拷进内核，并全量遍历 |
| `epoll`  | 先把关心的 fd 注册到内核，事件发生后只返回就绪 fd              | 更适合高并发、大量长连接场景      |

**为什么 `epoll` 更快：**

传统 `select/poll` 的问题是：

1. 每次调用都要把监听的 fd 集合从用户态拷到内核态
2. 内核返回后，应用还要再遍历一遍所有 fd，找谁就绪了

`epoll` 把这两件事拆开了：

1. `epoll_ctl`
   把你关心的 fd 先注册到内核
2. `epoll_wait`
   只等就绪事件，直接返回已经就绪的 fd 列表

所以 `epoll` 的优势不是一句“时间复杂度 O(1)”就能解释完，面试里更稳的说法是：

> `epoll` 避免了每次调用都把整个 fd 集合来回复制，也避免了每次都从头遍历所有 fd；它更像是把感兴趣的连接长期挂在内核里，等事件发生时只把就绪结果通知给用户态。

**`epoll` 的实现可以怎么理解：**

你可以从两个核心数据结构去记：

1. **红黑树**
   
   - 用来存“我关心哪些 fd”
   - `epoll_ctl` 本质上是在往这棵树里增删改监听项

2. **就绪链表**
   
   - 当某个 fd 真正就绪时，内核把它挂到 ready list 上
   - `epoll_wait` 只需要从 ready list 里取就绪事件返回

所以 `epoll` 的大致流程是：

```text
应用调用 epoll_create 创建 epoll 实例
应用调用 epoll_ctl 注册/修改/删除要监听的 fd
fd 就绪时，内核把对应事件挂到 ready list
应用调用 epoll_wait，只拿已经就绪的事件
```

**LT 和 ET：**

- `LT`（水平触发）
  只要 fd 里还有数据没读完，下次还会继续通知
- `ET`（边沿触发）
  只有状态从“无”变“有”时通知一次，要求应用一次尽量读干净

面试里一般可以这样说：

> 默认更容易写对的是 LT；ET 性能潜力更高，但更容易因为没读干净或没写完导致逻辑 bug。

**面试可直接背的版本：**

> IO 多路复用就是一个线程同时监听多个连接的 IO 事件，避免一个连接一个线程。`select` 和 `poll` 的问题是每次都要把监听集合传给内核，并且返回后还要遍历全部 fd；`epoll` 则把监听集合长期维护在内核里，通过 `epoll_ctl` 注册 fd，等事件发生时由 `epoll_wait` 直接返回就绪事件。它内部可以理解成“红黑树管理关注的 fd，ready list 管理已经就绪的 fd”，所以更适合高并发长连接场景。

---

### 12. 为什么很多人说 epoll 不是异步 IO，而仍然是同步 IO？

**答案：**

这题很容易和 “IO 多路复用” 混在一起。先记一句话：

> `epoll` 解决的是“一个线程如何高效等待很多 fd 的就绪事件”，但真正的数据读写，通常还是要线程自己去 `read` / `write`，所以它一般仍然属于**同步 IO**。

#### 为什么说它是同步的？

因为在典型 `epoll` 模型里：

1. 线程调用 `epoll_wait`，等待哪些 fd 就绪
2. 内核返回“某些 fd 已经可读 / 可写”
3. 线程自己再去调用 `read` / `write` 把数据拷到用户态或写出去

也就是说：

- `epoll_wait` 只是帮你等“事件通知”
- 真正的数据搬运和后续处理，还是当前线程自己做

所以它不是“数据自己异步完成了再通知你”，而是“内核告诉你现在可以干了，你自己去干”。

#### 那什么更接近异步 IO？

更严格意义上的异步 IO 是：

> 你把 IO 请求交给内核后，内核把等待 + 数据准备 + 数据拷贝这些事都完成，最后再通知你“整个 IO 已经完成了”。

而 `epoll` 只是把“等待很多 fd”的问题高效化了，它没有把“后续读写动作”也替你做完。

#### 面试里怎么答

> epoll 本质上是 IO 多路复用机制，它让一个线程可以同时监听很多 fd 的就绪事件，但线程在收到事件后，通常还是要自己调用 read/write 去完成真正的数据读写，所以它一般仍然被归类为同步 IO，而不是严格意义上的异步 IO。

### 13. TCP粘包问题是什么？如何解决？

**答案：**

**一句话总结：**

- TCP 是字节流没有消息边界，粘包的根因不是 TCP 出错，而是应用层没有自己定义消息边界。

**什么是粘包：**
TCP是字节流协议，没有消息边界，多个消息可能被合并接收。

```
发送方发送：[消息A][消息B][消息C]
接收方收到：[消息A消息B][消息C] 或 [消息A消][息B消息C]
```

**产生原因：**

1. **发送方**：Nagle算法合并小包
2. **接收方**：应用层读取不及时，多个包在缓冲区
3. **网络层**：MTU限制导致分片重组

**解决方案：**

| 方案       | 实现           | 适用场景 |
| -------- | ------------ | ---- |
| **固定长度** | 每条消息固定N字节    | 简单场景 |
| **分隔符**  | 用特殊字符分隔（如\n） | 文本协议 |
| **长度前缀** | 消息头包含长度信息    | 最常用  |

**长度前缀示例：**

```go
// 发送
func Send(conn net.Conn, data []byte) {
    length := uint32(len(data))
    binary.Write(conn, binary.BigEndian, length)
    conn.Write(data)
}

// 接收
func Recv(conn net.Conn) []byte {
    var length uint32
    binary.Read(conn, binary.BigEndian, &length)
    data := make([]byte, length)
    io.ReadFull(conn, data)
    return data
}
```

## 二、HTTP / HTTPS / TLS

### 14. HTTP和HTTPS的区别是什么？

**答案：**

**一句话总结：**

- HTTP 是明文传输，HTTPS 是在 HTTP 外面加了一层 `TLS`（`Transport Layer Security`，传输层安全协议），能同时提供机密性、完整性和身份认证。

| 特性      | HTTP | HTTPS             |
| ------- | ---- | ----------------- |
| **协议**  | HTTP | HTTP over TLS/SSL |
| **端口**  | 80   | 443               |
| **安全性** | 明文传输 | 加密传输              |
| **证书**  | 不需要  | 需要CA证书            |
| **性能**  | 较快   | 较慢（握手开销）          |
| **SEO** | 无加分  | 搜索引擎优先            |

**HTTPS提供的安全保障：**

1. **机密性**：数据加密，防止窃听
2. **完整性**：`MAC`（`Message Authentication Code`，消息认证码）校验，防止篡改
3. **身份认证**：证书验证服务器身份

**加密方式：**

- **非对称加密**：`RSA` / `ECDHE`（`Elliptic Curve Diffie-Hellman Ephemeral`，椭圆曲线临时 Diffie-Hellman 密钥交换），用于密钥交换
- **对称加密**：`AES`（`Advanced Encryption Standard`，高级加密标准），用于数据传输
- **哈希算法**：`SHA`（`Secure Hash Algorithm`，安全哈希算法），用于完整性校验
- **证书机构**：`CA`（`Certificate Authority`，证书颁发机构）

---

### 15. HTTPS请求的整个过程？TLS四次握手具体流程？

**答案：**

**一句话总结：**

- HTTPS 的核心流程就是先 TCP 建连，再完成 TLS 握手协商密钥，最后通过加密通道传输 HTTP 数据。

**整体过程：**

1. DNS 解析域名，拿到服务端 IP
2. TCP 三次握手建立连接
3. TLS 握手：协商版本、加密套件，验证证书，协商会话密钥
4. 后续 HTTP 请求和响应都走加密通道
5. 通信结束后，再进行 TCP 四次挥手

**TLS 握手怎么理解：**

- `TLS` 全称是 `Transport Layer Security`，中文通常叫**传输层安全协议**。
- 严格来说，TLS 不像 TCP 那样有固定“第几次握手”的官方定义。
- 面试里常说的“TLS 四次握手”，通常是指 **TLS 1.2 的经典握手流程按 4 个阶段来记**。
- 如果补充新版本，可以顺带说一句：**TLS 1.3 已经简化到通常 1 个 RTT，不太适合再机械地叫四次握手。**

**TLS 1.2 经典流程：**

```
Client                                    Server
  |                                          |
  |──── ClientHello ─────────────────────────>|
  |     (TLS版本、支持的密码套件、客户端随机数) |
  |                                          |
  |<──── ServerHello ────────────────────────-|
  |     (确认版本、密码套件、服务端随机数)     |
  |<──── Certificate ────────────────────────-|
  |     (服务端证书、公钥)                    |
  |<──── ServerKeyExchange ──────────────────-|
  |<──── ServerHelloDone ────────────────────-|
  |                                          |
  |──── ClientKeyExchange ───────────────────>|
  |     (客户端生成预主密钥并发给服务端)       |
  |──── ChangeCipherSpec ────────────────────>|
  |──── Finished ────────────────────────────>|
  |                                          |
  |<──── ChangeCipherSpec ───────────────────-|
  |<──── Finished ───────────────────────────-|
  |                                          |
  |        加密通信开始                        |
```

**每一步在干什么：**

1. `ClientHello`
   客户端告诉服务端：我支持哪些 TLS 版本和加密套件，这是我的随机数。
2. `ServerHello + Certificate`
   服务端选定一套加密方案，返回自己的随机数和证书，让客户端验证身份。
3. `ClientKeyExchange`
   客户端验证证书后，生成预主密钥，发送给服务端。双方基于客户端随机数、服务端随机数和预主密钥推导出本次会话密钥。
4. `Finished`
   双方都用协商好的密钥校验握手消息，确认后续开始用对称加密传输 HTTP 数据。

**为什么最终用对称加密传数据：**

- 非对称加密适合做身份认证和密钥交换，但性能较低
- 对称加密速度快，更适合传输大量业务数据

**面试回答模板：**

- HTTPS 的过程可以分成三段：先 DNS 解析，然后 TCP 三次握手建连，再在 TCP 连接之上做 TLS 握手。以 TLS 1.2 为例，客户端先发 `ClientHello`，带上支持的版本、加密套件和客户端随机数；服务端回 `ServerHello`、证书和服务端随机数；客户端验证证书后生成预主密钥并发给服务端；随后双方基于这些材料推导出会话密钥，并通过 `Finished` 报文确认握手成功。握手完成后，后续 HTTP 数据就通过对称加密通道传输。这样设计的原因是，非对称加密适合做身份认证和密钥交换，但效率较低，真正传业务数据更适合用对称加密。补充一点，TLS 1.3 已经把流程进一步简化了，通常 1 个 RTT 就能完成握手。
- 客户端先发 ClientHello，服务端返回 ServerHello、证书和服务端随机数。客户端验证证书后，自己生成一个 Pre-Master Secret，用服务端公钥加密发过去，服务端再用私钥解密拿到它。随后双方基于 Client Random、Server Random 和 Pre-Master Secret 通过相同的密钥派生算法计算出同一个 Master Secret，再进一步派生出真正用于对称加密的数据传输密钥。后续 HTTP 数据就是用这些会话密钥加密传输的

---

### 16. HTTP怎么保持长连接？HTTP 1.0和1.1的区别？HTTP/2 又改进了什么？

**答案：**

先记一句最核心的话：

> **HTTP/1.0 默认短连接，HTTP/1.1 默认长连接；HTTP/2 不是简单“更快一点”，而是通过二进制分帧、多路复用、头部压缩和服务器推送，显著提高了一条连接的利用率。**

---

#### 1. HTTP 怎么保持长连接？

##### HTTP/1.0

默认是**短连接**。  
也就是说：

- 发一个请求
- 收一个响应
- TCP 连接就可能被关闭

如果想复用连接，需要显式声明：

```http
Connection: keep-alive
```

##### HTTP/1.1

默认就是**长连接（persistent connection）**。  
也就是说：

- 一条 TCP 连接上可以连续承载多个 HTTP 请求 / 响应
- 不用每次都重新 TCP 三次握手

如果想显式关闭，才写：

```http
Connection: close
```

所以它节省的主要是：

- 重复建连成本
- 重复握手延迟
- 频繁创建销毁连接的开销

---

#### 2. HTTP/1.0 和 HTTP/1.1 的核心区别

| 特性     | HTTP/1.0              | HTTP/1.1                 |
| ------ | --------------------- | ------------------------ |
| 连接     | 默认短连接                 | 默认长连接                    |
| Host 头 | 可选                    | 必须                       |
| 缓存控制   | 较弱，主要靠 `Expires`      | 更完整，支持 `Cache-Control`   |
| 断点续传   | 不完善                   | 更好支持 `Range`             |
| 传输方式   | 一般依赖已知 Content-Length | 支持分块传输（chunked transfer） |
| 管道化    | 不支持                   | 支持（但实际上用得不多）             |

#### HTTP/1.1 的几个关键增强点

1. **持久连接**：减少重复 TCP 建连
2. **Host 头必须带**：支持同 IP 多虚拟主机
3. **分块传输编码**：内容长度不固定时可以边生成边发
4. **缓存控制更细**：`Cache-Control` 比单纯 `Expires` 更灵活
5. **支持 Range**：便于断点续传、大文件下载

---

#### 3. 那 HTTP/2 到底改进了什么？

这部分非常值得细讲，因为面试里常常会追问：

> 你说 HTTP/2 更快，那到底快在哪？

HTTP/2 最核心的几个改进是：

1. **二进制分帧（Binary Framing）**
2. **多路复用（Multiplexing）**
3. **头部压缩（HPACK Header Compression）**
4. **服务器推送（Server Push）**

下面分别讲。

---

#### 3.1 二进制分帧（Binary Framing）

HTTP/1.1 本质上是**文本协议**。  
也就是说，请求和响应长得像这样：

```http
GET /index.html HTTP/1.1
Host: example.com
User-Agent: xxx
```

这种方式的缺点是：

- 解析依赖文本规则
- 协议扩展性一般
- 不够紧凑

HTTP/2 做了一件事：

> **把原来的 HTTP 消息拆成更小的二进制帧（frame）来传。**

你可以理解成：

- HTTP/1.1：传整段文本
- HTTP/2：把消息切成很多标准化小块，每块都有明确类型和长度

这样带来的好处是：

- 更容易高效解析
- 更适合并发调度
- 为多路复用打基础

一句话：

> **二进制分帧是 HTTP/2 的底层传输组织方式，它让协议更规整、更高效，也为多路复用提供了基础。**

---

#### 3.2 多路复用（Multiplexing）

这是 HTTP/2 最核心的改进之一。

#### HTTP/1.1 的问题

虽然 HTTP/1.1 有长连接，但一条连接上请求处理还是容易互相影响。  
即便有管道化，也会遇到经典问题：

> **队头阻塞（Head-of-Line Blocking）**

比如：

- 你发了请求 A、B、C
- A 响应特别慢
- 后面的 B、C 即使早就准备好了，也容易被前面卡住

#### HTTP/2 的做法

HTTP/2 在**一条 TCP 连接**里，允许同时跑多个独立的 stream（流）。

你可以理解成：

```text
一条 TCP 连接
  ├── stream 1：请求 /a
  ├── stream 2：请求 /b
  ├── stream 3：请求 /c
```

每个 stream 的帧可以交错发送，比如：

```text
A1 B1 C1 A2 B2 C2 ...
```

这样就不用像 HTTP/1.1 那样“一条连接里基本排队等前面”。

#### 它的价值是什么？

- 一条连接能承载更多并发请求
- 减少浏览器为了并发下载资源而建立很多 TCP 连接
- 连接利用率更高
- 页面加载通常更快

一句话：

> **多路复用解决的是“一条连接里多个请求怎么并发传”的问题。**

---

#### 3.3 头部压缩（HPACK Header Compression）

HTTP 请求头和响应头里往往有很多重复内容，比如：

- Cookie
- User-Agent
- Accept
- Host
- Authorization

在 HTTP/1.1 里，每次请求很多头都会原样重复发，非常浪费带宽。

HTTP/2 引入了：

> **HPACK（Header Compression for HTTP/2）**

它会通过：

- 静态表
- 动态表
- 索引引用

来避免反复发送完全一样的头部字段。

你可以粗略理解成：

- 第一次发完整头部
- 后面相似的头部不再全文重发，而是“引用之前的表项”

这对什么最有帮助？

- 小请求很多的场景
- 接口调用频繁的场景
- Header 很长（比如 Cookie 很重）的场景

一句话：

> **HPACK 解决的是“HTTP 头重复发送太浪费”的问题。**

---

#### 3.4 服务器推送（Server Push）

HTTP/2 还支持一个特性：

> 服务端在客户端还没明确请求某个资源时，主动把它推过去。

比如客户端请求 HTML 页面，服务端知道这个页面肯定还要用到：

- CSS
- JS
- 图片

于是服务端可以主动推送这些资源。

不过这个特性在现实里没有想象中那么常用，因为：

- 容易推错资源
- 和浏览器缓存策略配合复杂
- 实际收益不一定稳定

所以面试里知道它的存在即可，不用把它讲成 HTTP/2 的最大亮点。

---

#### 4. HTTP/2 就完全没有问题了吗？

也不是。  
HTTP/2 虽然解决了 HTTP 层的多路复用问题，但它底层仍然跑在 **TCP** 上。  
所以如果 TCP 层某个包丢了，还是会影响这条连接上的多个 stream。

这也是为什么后来 HTTP/3 改走 QUIC / UDP，一个核心目标就是进一步缓解这种底层队头阻塞问题。

---

#### 5. 面试里推荐这样答

> HTTP/1.0 默认短连接，HTTP/1.1 默认长连接，并补充了 Host 头、分块传输、缓存控制和断点续传等能力。HTTP/2 的关键改进不只是“更快”，而是通过二进制分帧让协议更规整，通过多路复用让一条 TCP 连接承载多个并发请求，通过 HPACK 头部压缩减少重复头部开销，并引入服务器推送能力。所以 HTTP/2 提升的核心是连接利用率和并发效率。但因为它底层仍然基于 TCP，所以在更低层面仍然会受到 TCP 丢包带来的影响。

**一句话总结：**

> HTTP/2 的核心价值就是：**把“一条连接只能低效串行跑请求”升级成“单连接高效并发传输请求”。**

### 17. HTTP状态码502和503分别代表什么？

**答案：**

**一句话总结：**

- `502` 更像是上游返回有问题，`503` 更像是当前服务自己忙不过来或暂时不可用。

| 状态码                         | 含义    | 场景             |
| --------------------------- | ----- | -------------- |
| **502 Bad Gateway**         | 网关错误  | 上游服务无响应或返回无效响应 |
| **503 Service Unavailable** | 服务不可用 | 服务器过载或维护中      |

**502产生原因：**

1. 上游服务器宕机
2. 上游服务器响应超时
3. 上游返回格式错误

**503产生原因：**

1. 服务器过载
2. 服务器正在维护
3. 限流触发

**其他常见状态码：**

```
200 OK - 成功
301 Moved Permanently - 永久重定向
302 Found - 临时重定向
304 Not Modified - 缓存未修改
400 Bad Request - 请求错误
401 Unauthorized - 未认证
403 Forbidden - 禁止访问
404 Not Found - 资源不存在
500 Internal Server Error - 服务器内部错误
504 Gateway Timeout - 网关超时
```

---

### 18. 502 和 504 的区别是什么？

**答案：**

- 两者都常见于网关或代理层，但语义不同。
- `502 Bad Gateway` 更像是：代理拿到了一个**非法或异常的下游响应**。
- `504 Gateway Timeout` 更像是：代理等待下游响应时**超时了**。
- 简单记：
  - `502`：下游回了，但回得不对
  - `504`：下游没及时回

---

### 19. GET、POST、PUT 的区别？

**答案：**

**一句话总结：**

- 这三个方法的核心区别在于 **语义**、**幂等性** 和 **典型使用场景**，不是“谁底层更高级”。

| 方法       | 语义          | 幂等性 | 典型场景              |
| -------- | ----------- | --- | ----------------- |
| **GET**  | 获取资源        | 幂等  | 查询详情、列表、搜索        |
| **POST** | 新建资源 / 提交动作 | 非幂等 | 创建订单、提交表单、触发一次性操作 |
| **PUT**  | 整体更新资源      | 幂等  | 全量更新用户信息、覆盖式修改    |

**怎么理解这三个方法：**

1. **GET**
   
   - 语义是“取数据”，不应该对服务端状态产生副作用。
   - 多次调用结果通常一致，所以是幂等的。
   - 常见例子：查用户信息、查商品详情、查列表。

2. **POST**
   
   - 语义是“提交数据”或者“发起一个动作”。
   - 它通常不是幂等的，因为连续提交两次，往往会创建两份数据或执行两次动作。
   - 常见例子：创建订单、提交评论、发起支付。

3. **PUT**
   
   - 语义更偏“对一个已知资源做整体覆盖式更新”。
   - 它通常是幂等的，因为同样的数据反复更新多次，结果应该一样。
   - 常见例子：把某个用户资料整体更新成一份新内容。

**一个面试里很容易加分的点：**

- `POST` 和 `PUT` 最大的区别，不是都能“传 body”，而是：
  - `POST` 更偏创建 / 提交动作
  - `PUT` 更偏对目标资源做确定性的覆盖更新

**技术层面要说清：**

- 三者本质上都是 HTTP 方法，底层都可以跑在同样的 TCP 连接上。
- 它们的区别主要是协议语义、缓存语义和接口设计约定，不是 TCP 层差异。
- `GET` 理论上也不是绝对不能带 body，但工程上通常不这么设计，因为兼容性和语义都不好。

**面试里推荐这样答：**

> GET、POST、PUT 的核心区别在语义和幂等性。GET 用来获取资源，应该是幂等的；POST 更适合创建资源或提交一次性动作，通常非幂等；PUT 更适合对一个确定资源做整体更新，通常是幂等的。它们底层都还是 HTTP 请求，差别主要在接口语义，而不是 TCP 层。

**实际应用：**

```http
# GET - 查询
GET /users?id=123 HTTP/1.1

# POST - 提交
POST /users HTTP/1.1
Content-Type: application/json

{"name": "Tom", "age": 20}
```

### 20. HTTPS 会加密 URL 吗？为什么敏感信息不建议放在 URL 里？

**答案：**

- 会，但要分情况说。
- **HTTPS 建立成功之后，请求里的 `path + query`**，比如 `/user?id=123&token=xxx`，这些内容是在 TLS 加密通道里的，传输过程中普通中间人看不到。
- 但**不是整个 URL 的所有信息都完全隐藏**：
  - 目标 **IP 和端口** 对网络层是可见的。
  - **域名** 在很多场景下也可能暴露，比如传统 DNS 查询、TLS 握手里的 SNI；只有在 DoH/DoT、ECH 这类机制下才会进一步隐藏。

所以高分回答一般会说：

- **HTTPS 会加密 HTTP 请求内容，包括 URL 里的路径和查询参数；**
- **但域名、IP、端口这些信息不一定完全隐藏。**

为什么敏感信息仍然不建议放 URL：

- URL 很容易进入**浏览器历史记录**、**收藏夹**、**服务器访问日志**、**代理日志**、**CDN 日志**。
- 页面跳转时还可能通过 **Referer** 带到别的系统。
- 即使传输链路被 HTTPS 加密了，**链路外的存储和日志侧仍然可能泄露**。

所以像密码、验证码、长期 token、身份证号这类敏感数据，面试里更稳的回答是：

- **不要放 URL query 里；**
- 更适合放在 **请求体** 或 **Authorization Header**；
- 同时配合 HTTPS 传输、服务端脱敏和日志治理。

一句话版：

- **HTTPS 会加密请求中的 path 和 query，但不会天然隐藏所有元信息；敏感数据不该放 URL，不是因为 HTTPS 不加密，而是因为 URL 太容易被历史记录、日志和 Referer 泄露。**

---

### 21. 为什么 HTTP/2 已经多路复用了，还要 HTTP/3？

**答案：**

先记一句最关键的话：

> **HTTP/2 解决了 HTTP 层“一条连接里多个请求怎么并发传”的问题，但它底层仍然跑在 TCP 上；而 HTTP/3 的核心目标，是进一步解决 TCP 层丢包带来的队头阻塞问题。**

#### 为什么 HTTP/2 还不够？

HTTP/2 确实已经做了多路复用：

- 一条 TCP 连接里可以有多个 stream
- 多个请求 / 响应可以交错传输
- 不再像 HTTP/1.1 那样主要靠多开连接提高并发

但问题在于：

> **这些 stream 虽然在 HTTP 层是多路复用的，但底层仍然共用同一条 TCP 连接。**

而 TCP 有一个特点：

> **TCP 提供的是“整条连接上的全局有序字节流”。**

也就是说：

- TCP 只关心整条连接的数据顺序
- 它并不理解“stream A”和“stream B”是彼此独立的
- 如果前面的某个 TCP 包丢了，后面的数据即使已经到了，也得先等这个丢包重传成功

这就意味着：

- HTTP/2 在应用层解决了一部分阻塞
- 但在更低层的 TCP 上，仍然会被丢包拖住

这就是为什么大家会说：

> **HTTP/2 解决了 HTTP 层的队头阻塞，但没有彻底解决传输层的队头阻塞。**

#### HTTP/3 到底是怎么做的？

HTTP/3 不再基于 TCP，而是基于：

- `QUIC`（跑在 UDP 之上）

关键不只是“它用了 UDP”，而是：

> **QUIC 在传输层原生支持多个独立 stream，每个 stream 只需要保证自己内部有序，不要求整条连接上所有数据统一按序交付。**

这和 TCP 非常不一样：

- **TCP**：整条连接是一个全局有序字节流
- **QUIC**：连接里可以有多个独立 stream，每个 stream 各自保证顺序

所以如果某个 stream 对应的包丢了：

- 在 TCP 里，后面的数据可能一起被卡住
- 在 QUIC 里，通常只影响这个 stream，本连接里的其他 stream 不一定要一起等

#### 一个最容易懂的例子

假设浏览器加载一个页面时，要同时拿：

- HTML
- CSS
- JS
- 图片

##### HTTP/2 + TCP

如果承载 CSS 的某个 TCP 包丢了：

- TCP 会卡住后续交付
- JS、图片所在 stream 也可能一起受影响

##### HTTP/3 + QUIC

如果 CSS 那个 stream 丢包：

- 只影响 CSS 这个 stream
- JS、图片 stream 不一定要一起等它

所以页面整体体验更稳，特别是弱网、丢包环境下更明显。

#### 你可以怎么理解两者关系？

- **HTTP/2**：
  
  - 重点解决“HTTP/1.1 一条连接里请求并发效率低”的问题
  - 手段是：二进制分帧 + 多路复用 + 头部压缩

- **HTTP/3**：
  
  - 重点解决“HTTP/2 虽然多路复用了，但 TCP 丢包还是会拖慢整条连接”的问题
  - 手段是：基于 QUIC / UDP，把 stream 独立性做得更强

#### 面试里推荐这样答

> HTTP/2 确实已经通过多路复用提高了一条连接的并发利用率，但它底层仍然基于 TCP。TCP 提供的是整条连接上的全局有序字节流，所以一旦某个包丢失，后续数据即使到了，也可能被卡住，这会影响同一条连接上的多个 stream。HTTP/3 改走基于 UDP 的 QUIC，它在传输层原生支持多个独立 stream，每个 stream 只保证自己内部有序，不要求整个连接上的所有数据统一按序交付。所以某个 stream 丢包时，通常只阻塞这个 stream，不会把其他 stream 一起拖慢，这就是它缓解队头阻塞的核心机制。

**一句话总结：**

> **HTTP/2 已经让“一条连接里多个请求能并发传”，HTTP/3 则继续解决“底层 TCP 丢包会不会把这些并发请求一起拖慢”。**

### 22. 从输入URL到页面渲染的整个过程？

**答案：**

**一句话总结：**

- 从输入 URL 到页面渲染，主链路就是 DNS 解析、TCP/TLS 建连、HTTP 请求响应，再到浏览器解析和渲染页面。

```
1. URL解析
   ↓
2. DNS解析（域名→IP）
   ↓
3. TCP三次握手
   ↓
4. TLS握手（HTTPS）
   ↓
5. 发送HTTP请求
   ↓
6. 服务器处理请求
   ↓
7. 返回HTTP响应
   ↓
8. 浏览器解析HTML
   ↓
9. 构建DOM树
   ↓
10. 构建CSSOM树
    ↓
11. 合并为渲染树
    ↓
12. 布局（Layout）
    ↓
13. 绘制（Paint）
    ↓
14. 合成（Composite）
```

**DNS解析过程：**

1. 浏览器缓存
2. 系统缓存
3. 路由器缓存
4. ISP DNS服务器
5. 根DNS → 顶级域DNS → 权威DNS

**详细说明：**

- **TCP**：建立可靠连接
- **TLS**：加密传输
- **HTTP**：请求/响应
- **DOM**：文档对象模型
- **CSSOM**：样式对象模型
- **Layout**：计算元素位置大小
- **Paint**：绘制像素
- **Composite**：图层合成

**面试补充：浏览器访问网站时，本机和服务端分别用什么端口？**

这个问题很容易混淆，核心结论是：

- `80` 和 `443` 通常是**服务端监听端口**
- 浏览器所在的**客户端本机一般不会固定使用 80 或 443 作为源端口**
- 客户端发起连接时，操作系统通常会分配一个**随机临时端口**

也就是说，当你在浏览器里访问一个 HTTPS 网站，比如 `https://www.bilibili.com`，更像是下面这样一条连接：

```text
本机IP:52341  ->  B站服务器IP:443
```

而不是：

```text
本机IP:443  ->  服务器IP:443
```

所以面试里可以直接说：

> 访问 HTTP 网站时，浏览器通常连接的是服务端的 80 端口；访问 HTTPS 网站时，浏览器通常连接的是服务端的 443 端口。客户端自己一般使用操作系统分配的临时端口，不会固定从 80 或 443 发起连接。

**一条 TCP 连接怎么唯一确定？**

不是只看端口，而是看**四元组**：

1. 源 IP
2. 源端口
3. 目的 IP
4. 目的端口

这四个值只要有一个不同，通常就是不同的 TCP 连接。

比如下面这两条：

```text
192.168.1.10:52341 -> 1.2.3.4:443
192.168.1.10:52342 -> 1.2.3.4:443
```

虽然访问的是同一个服务端 IP 和同一个 443 端口，但因为源端口不同，所以它们是两条不同的 TCP 连接。

**用浏览器访问 Bilibili 的典型流程，可以这样理解：**

1. 你在浏览器输入 `https://www.bilibili.com`
2. 浏览器看到是 `https`，默认目标端口就是 `443`
3. 浏览器先做 DNS 解析，拿到 B 站某个服务节点的 IP
4. 操作系统给你的浏览器分配一个本地临时端口，比如 `52341`
5. 浏览器发起 TCP 连接：`本机IP:52341 -> B站IP:443`
6. TCP 三次握手成功后，再进行 TLS 握手
7. TLS 建立完成后，浏览器通过这条连接发送 HTTP 请求
8. 服务端返回 HTML、JSON、图片、视频流地址等响应内容
9. 浏览器继续解析页面，如果还需要更多资源，可能会继续复用现有连接，或者再建立新的连接

**为什么有时你感觉“访问的是同一个网站，却不止一条连接”？**

因为现实里经常会有这些情况：

- 一个页面会请求很多资源，比如 HTML、JS、CSS、图片、接口数据
- 浏览器可能会建立多条连接并发下载资源
- 同一个域名背后可能对应多个 IP
- HTTP/2 可能会在一条 TCP 连接里复用多个请求

所以更准确的说法是：

> 访问一个网站，不是简单地“我和这个网站只有一条 TCP 连接”。是否是同一条连接，要看四元组是否一致；而且浏览器还可能做连接复用。

**再补一个容易被追问的点：**

现在很多网站还会使用 `HTTP/3`，它走的是 `QUIC over UDP 443`，这时就不再是 TCP 了。  
不过面试里如果没有特别强调版本，默认按 **HTTPS 基于 TCP 443** 来回答就可以。

**这道题面试时推荐这样答：**

> 浏览器访问网站时，80 和 443 通常是服务器的监听端口，不是客户端固定使用的端口。客户端一般会使用操作系统分配的临时端口。比如我访问一个 HTTPS 网站，本机会先做 DNS 解析，然后以 `本机IP:随机端口 -> 服务端IP:443` 的形式发起 TCP 连接。TCP 连接是由源 IP、源端口、目的 IP、目的端口这四元组唯一确定的，所以即使访问同一个网站、同一个 443 端口，只要源端口不同，也可能是不同的连接。

## 三、DNS、CDN 与网络排障

### 23. CDN 的工作原理是什么？为什么它能加速访问？

**答案：**

- `CDN`（`Content Delivery Network`，内容分发网络）本质上是把内容缓存到离用户更近的边缘节点，让用户优先从最近节点获取资源。
- 这样做的核心价值有两个：
  - 减少跨地域、跨运营商访问带来的网络时延；
  - 降低源站压力，避免所有请求都直接打到主站。
- 一个典型流程是：
  - 用户访问域名
  - DNS 调度到合适的 CDN 节点
  - 节点命中缓存就直接返回
  - 未命中再回源站拉取并缓存
- 面试里最好补一句：CDN 不只是“缓存图片”，也能缓存静态页面、视频分发内容，部分动态加速场景也会用到。

### 24. DNS 解析过程是怎样的？递归查询和迭代查询有什么区别？根域名服务器和权威 DNS 服务器又是什么？

**答案：**

**一句话总结：**

- `DNS`（`Domain Name System`，域名系统）的本质是把域名解析成 IP；完整过程通常是“本机缓存 -> 本地 DNS 服务器 -> 根域名服务器 -> 顶级域服务器 -> 权威 DNS 服务器”。

#### 典型解析流程

1. 浏览器先查自己的 DNS 缓存
2. 操作系统查本机缓存和 `hosts`
3. 如果还没有，就把请求发给本地 DNS 服务器（通常是运营商 DNS 或公司内网 DNS）
4. 本地 DNS 如果也没缓存，就开始往外问：
   - 先问根域名服务器
   - 再问顶级域服务器，比如 `.com`
   - 再问权威 DNS 服务器
5. 最后拿到域名对应的 IP，返回给客户端

#### 根域名服务器是什么？

根域名服务器（Root DNS Server）不是保存全世界所有域名 IP 的地方，它更像是：

> **DNS 世界的“总导航台”。**

它主要负责告诉你：

- “这个域名属于哪个顶级域（TLD）”
- “你下一步应该去问哪一组顶级域服务器”

比如你要查：

```text
www.example.com
```

根域名服务器不会直接告诉你最终 IP，通常会告诉你：

> `.com` 这件事你去问 `.com` 顶级域服务器。

所以它负责的是**指路**，不是直接给最终答案。

#### 权威 DNS 服务器是什么？

权威 DNS 服务器（Authoritative DNS Server）可以理解成：

> **真正掌握某个域名最终记录的人。**

它保存的是该域名的正式记录，比如：

- `A` 记录（域名 -> IPv4）
- `AAAA` 记录（域名 -> IPv6）
- `CNAME` 记录（别名）
- `MX` 记录（邮件服务器）

比如查 `www.example.com`，最终真正能告诉你“它对应哪个 IP”的，就是负责 `example.com` 这个域名的权威 DNS 服务器。

所以：

- **根域名服务器**：告诉你下一步问谁
- **权威 DNS 服务器**：告诉你最终答案是什么

#### 递归查询和迭代查询的区别

- **递归查询**
  
  - 你问一个 DNS 服务器，它要么直接给你最终答案，要么替你继续问到底
  - 对客户端更省事

- **迭代查询**
  
  - 你问一个 DNS 服务器，它如果不知道最终答案，就告诉你“下一步去问谁”
  - 你再自己继续问下一个

#### 在真实场景里通常怎么配合

- 客户端到本地 DNS：更像递归查询
- 本地 DNS 到根 / TLD / 权威 DNS：更像迭代查询

#### 为什么 DNS 很重要？

- 域名访问的第一步就是它
- DNS 慢，整个请求链路就慢
- 所以面试里也常会追问：
  - DNS 缓存有哪些层级
  - DNS 劫持是什么
  - DNS 为什么会影响首包时间

#### 面试里可以这样总结

> DNS 解析就是把域名翻译成 IP。客户端通常先查本地缓存，再问本地 DNS；本地 DNS 如果没有缓存，就按根域、顶级域、权威 DNS 的链路继续查。根域名服务器更像总导航台，只负责告诉你下一步该问哪个顶级域服务器；真正保存某个域名最终记录的是权威 DNS 服务器。客户端和本地 DNS 之间更像递归查询，而本地 DNS 往外查通常更像迭代查询。

### 25. 如何查看网络连接状态？

**答案：**

**一句话总结：**

- 排查网络问题时，先看端口有没有监听、连接处于什么 TCP 状态，再结合 `ss/netstat` 判断问题卡在建连、传输还是关闭阶段。

**netstat命令：**

```bash
# 查看所有TCP连接
netstat -ant

# 查看监听端口
netstat -tlnp

# 统计各状态连接数
netstat -n | awk '/^tcp/ {++state[$NF]} END {for(key in state) print key,state[key]}'
```

**ss命令（更快）：**

```bash
# 查看所有TCP连接
ss -ant

# 查看监听端口
ss -tlnp

# 查看ESTABLISHED连接
ss -ant state established
```

**常见连接状态：**

```
LISTEN      - 监听中
SYN_SENT    - 已发送SYN
SYN_RCVD    - 已收到SYN
ESTABLISHED - 已建立连接
FIN_WAIT_1  - 已发送FIN
FIN_WAIT_2  - 已收到ACK
TIME_WAIT   - 等待2MSL
CLOSE_WAIT  - 收到FIN，等待关闭
LAST_ACK    - 等待最后的ACK
CLOSED      - 已关闭
```

### 26. 客户端连不上服务端，如何从网络角度排查？

**答案：**

排查这类问题，核心思路是：

> **先确认“服务端有没有监听”，再确认“链路能不能到”，最后确认“协议和应用层有没有拦住”。**

#### 一个比较稳的排查顺序

1. **先看服务端端口是否真的在监听**
   
   - `ss -tlnp`
   - `netstat -tlnp`
   - 如果服务根本没监听，这就不是网络传输问题，而是服务启动或绑定问题

2. **确认客户端能不能到服务端 IP**
   
   - `ping` 看连通性（注意有些机器禁 ping）
   - `traceroute` / `tracepath` 看路径
   - 如果是域名访问，先确认 `DNS` 解析是否正常

3. **确认中间有没有被防火墙 / 安全组 / ACL 拦住**
   
   - 云环境里要检查安全组
   - 宿主机要看 iptables / nftables / firewalld
   - 公司内网还可能有网段隔离或策略路由

4. **看 TCP 到底卡在哪个阶段**
   
   - `SYN_SENT` 很多：常见是对端没通、被防火墙丢包、目标不回应
   - `SYN_RCVD` 很多：说明服务端收到了 SYN，但连接没完全建立，可能有半连接积压
   - `ESTABLISHED` 建好了但业务不通：就要继续查应用协议层

5. **最后看应用层问题**
   
   - HTTPS 证书问题
   - HTTP Host / 路由不匹配
   - 服务端返回 4xx / 5xx
   - 上游代理、网关、Ingress 转发错误

#### 面试里推荐这样答

> 网络排查我一般按“监听 -> 连通性 -> 防火墙/安全组 -> TCP 状态 -> 应用层”这个顺序来查。先确认服务端端口有没有监听，再确认客户端到服务端 IP 和端口是否可达；如果链路层面没问题，就看 TCP 是卡在握手还是已经建立连接；如果连接已经建立但请求仍然失败，就继续往 HTTP、TLS、网关转发和业务日志去排查。

## 四、实时通信与会话机制

### 27. WebSocket 是什么？为什么它适合实时场景？

**答案：**

- 先把三者关系说清楚：
  
  - `TCP` 是**传输层协议**，负责可靠传输、顺序、重传、流量控制这些能力。
  - `HTTP` 是**应用层协议**，通常跑在 TCP 之上，核心是“请求 - 响应”。
  - `WebSocket` 也是**应用层协议**，它通常先借助一次 HTTP Upgrade 握手建立连接，握手成功后再切换成 WebSocket 协议，本质上仍然跑在 TCP 上。

- 所以它们的关系可以这样记：

```text
WebSocket / HTTP
        ↓
       TCP
        ↓
        IP
```

- 再说“单工 / 半双工 / 全双工”：
  
  - `TCP` 是**全双工**，客户端和服务端都可以同时发送和接收数据。
  - `WebSocket` 也是**全双工**，因为它建立在 TCP 之上，而且协议语义上就允许双方随时主动发消息。
  - `HTTP` 严格说是请求响应式应用层协议，不完全等同于链路层里的“半双工”概念；但在面试语境里，通常可以把它理解成**偏半双工/请求响应模型**：
    - 一般由客户端先发请求
    - 服务端再返回响应
    - 服务端不能像 WebSocket 那样在同一条连接上随时主动双向推送消息

- WebSocket 适合实时场景，核心原因有四个：
  
  - 建连一次后可以长期保持，不需要频繁重复握手
  - 服务端可以主动推送消息
  - 客户端也可以随时主动发消息
  - 协议头相对更轻，实时交互开销更低

- 所以聊天室、实时通知、弹幕、在线协同这类场景，WebSocket 会比 HTTP 轮询更自然。

- 面试里推荐这样总结：

> TCP 是传输层的全双工可靠传输协议；HTTP 是跑在 TCP 之上的请求响应式应用层协议；WebSocket 也是跑在 TCP 之上的应用层协议，但它通过一次 HTTP Upgrade 建连后，变成了真正适合实时双向通信的全双工协议。所以实时场景通常优先考虑 WebSocket，而不是传统 HTTP 轮询。

### 28. RTMP 这类流媒体协议一般用在什么场景？为什么直播场景会提到它？

**答案：**

- `RTMP`（`Real-Time Messaging Protocol`，实时消息传输协议）是一种面向实时音视频传输的协议，历史上在直播推流链路里非常常见。
- 在典型直播系统里，主播端常用 `RTMP` 推流到服务器，服务器再做转码、分发，观众端则可能用 `HLS`（`HTTP Live Streaming`，HTTP 实时流）、`FLV`（`Flash Video`）或 `WebRTC`（`Web Real-Time Communication`，网页实时通信）等不同协议拉流。
- 面试里不一定要求你背 RTMP 的报文细节，更重要的是说清它在链路里的位置：
  - **推流协议**
  - **低延迟传输**
  - **适合直播源站接入**
- 如果继续追问，可以补一句：现在很多实时性要求极高的直播互动场景也会更多考虑 WebRTC。

### 29. SSE 是什么？它和 WebSocket 有什么区别？

**答案：**

- `SSE` 全称是 `Server-Sent Events`，可以理解成一种基于 HTTP 的**服务端单向推送**机制。

- 它的特点是：
  
  - 客户端通过普通 HTTP 请求建立连接
  - 服务端把响应保持为长连接
  - 后续持续往这个响应流里推送文本事件

- 所以 `SSE` 本质上还是跑在 HTTP 之上的，没有切换成像 WebSocket 那样的独立双向协议。

- 它和 WebSocket 的核心区别，可以从三个维度记：

| 维度   | SSE               | WebSocket                       |
| ---- | ----------------- | ------------------------------- |
| 通信方向 | **单向**，服务端 -> 客户端 | **双向**，客户端 <-> 服务端              |
| 协议基础 | 基于 HTTP 长连接响应流    | 先 HTTP Upgrade，再切到 WebSocket 协议 |
| 适合场景 | 消息推送、流式输出、进度通知    | 聊天室、实时协作、双向互动                   |

- 如果再展开一点：
  
  - `SSE` 更轻，接入简单，天然适合“服务端持续往前端推文本消息”的场景，比如 AI 流式回答、通知流、任务进度流。
  - `WebSocket` 更强，因为它支持双向主动通信，适合聊天室、在线协作、游戏、弹幕这种双方都要频繁发消息的场景。

- 还可以从“谁更像传统 Web”这个角度理解：
  
  - `SSE` 更像“HTTP 没结束，服务端不断往下写数据”
  - `WebSocket` 更像“升级成了专门做实时通信的独立通道”

- 面试里推荐这样总结：

> SSE 和 WebSocket 都能做实时推送，但 SSE 是基于 HTTP 的单向服务端推送，更轻、更简单，适合流式文本输出和通知；WebSocket 是建立在 TCP 上的全双工应用层协议，支持客户端和服务端双向主动通信，更适合聊天、协作和实时互动场景。

### 30. JWT Token、Session、Cookie 有什么区别？

**答案：**

- 这题最容易混的点是：**Cookie、Session、JWT 不是同一层概念。**
- 最稳的答法是先拆定义，再讲它们之间怎么配合。

**1. Cookie 是什么？**

- `Cookie` 是浏览器保存的一小段数据
- 服务端通过 `Set-Cookie` 下发，浏览器后续请求会自动携带
- 它本质上是一个**客户端存储和传输机制**
- Cookie 里可以放：
  - `session_id`
  - 登录态标识
  - 一些偏好设置

**2. Session 是什么？**

- `Session` 是服务端保存的会话状态
- 用户登录后，服务端生成一条会话记录，比如：
  - 用户 ID
  - 登录时间
  - 权限信息
- 然后给客户端一个 `session_id`
- 客户端通常把这个 `session_id` 放在 Cookie 里，后续请求带回来
- 服务端再根据 `session_id` 找到对应会话

所以常见的 **Session 方案** 实际是：

```text
浏览器 --(Cookie: session_id)--> 服务端
服务端 --(session_id 查 Session 存储)--> 找到登录态
```

**3. JWT Token 是什么？**

- `JWT`（`JSON Web Token`）是一种 token 格式，本质上是一段签名后的字符串
- 里面通常会带：
  - 用户标识
  - 过期时间
  - 签发时间
  - 业务自定义声明
- 客户端拿到后自己保存，之后请求时主动带上，一般放在：
  - `Authorization: Bearer <token>`
- 服务端收到后校验签名，如果签名合法且没过期，就认为身份有效

所以 JWT 更偏：

```text
客户端自己带 token -> 服务端校验签名 -> 识别用户身份
```

**核心区别可以这样记：**

| 维度        | Cookie     | Session                      | JWT Token     |
| --------- | ---------- | ---------------------------- | ------------- |
| 本质        | 浏览器存储机制    | 服务端会话机制                      | 一种无状态身份凭证     |
| 存储位置      | 客户端        | 服务端                          | 通常客户端保存       |
| 服务端是否保存状态 | 不一定        | **要保存**                      | 通常**不用保存**    |
| 客户端是否自动携带 | 浏览器下通常自动带  | 通常借助 Cookie 自动带 `session_id` | 一般手动放请求头      |
| 适合场景      | Web 站点状态传递 | 传统登录会话                       | 前后端分离、微服务、移动端 |

**面试里常见的配合关系：**

- `Cookie + Session`
  
  - 浏览器自动带 Cookie
  - Cookie 里放 `session_id`
  - 服务端查 Session
  - 优点是容易做失效控制、踢人下线、权限收回
  - 缺点是分布式场景下要共享 Session，常要配 Redis

- `JWT Token`
  
  - 客户端自己保存 token
  - 请求时主动带 token
  - 服务端只校验签名，不一定查库
  - 优点是更适合前后端分离和微服务
  - 缺点是 token 一旦签发，在过期前通常不容易立刻失效，除非加黑名单、刷新机制

**一句话总结：**

- `Cookie` 是浏览器存储数据的方式
- `Session` 是服务端保存登录状态的方案
- `JWT` 是客户端持有、服务端校验的无状态认证凭证

**如果面试官追问“JWT 和 Session 怎么选”可以这样答：**

- 如果是传统 Web、服务端渲染、强调强会话控制，`Session` 更直接
- 如果是前后端分离、移动端、微服务，`JWT` 更常见
- 但 `JWT` 不是一定比 `Session` 更高级，只是更适合某些架构场景

### 31. 服务端和用户端一般怎么通信？有哪些常见方式？

**答案：**

- 这题不要只答 HTTP，后端更稳的回答方式是按场景拆：
1. **请求-响应型**
   
   - HTTP / HTTPS
   - 最常见，适合 CRUD、页面请求、接口调用

2. **长连接实时通信**
   
   - WebSocket
   - 适合聊天室、实时通知、协同编辑、弹幕

3. **服务端单向推送**
   
   - SSE
   - 适合流式文本输出、进度通知

4. **音视频 / 推流拉流**
   
   - RTMP、WebRTC、HLS 等

**如果面试官追问“有没有更轻一点的方式”，可以这样答：**

- 如果只是服务端往前端持续推文本，不一定非要上 WebSocket
- SSE 会更轻
- 如果只是普通接口交互，HTTP 其实最简单也最稳

**面试里推荐这样答：**

> 服务端和用户端通信最常见还是 HTTP/HTTPS，请求响应式最简单稳妥；如果需要双向实时通信，就用 WebSocket；如果只是服务端单向持续推送文本，可以考虑 SSE，会更轻。到底选哪种，不是看协议新旧，而是看交互模式是单向推送、双向实时，还是普通请求响应。
