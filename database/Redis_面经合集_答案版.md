## 一、基础原理与数据结构

### 1. Redis为什么快？是单线程还是多线程？多线程体现在哪里？

**答案：**

**为什么快：**

1. **纯内存操作**：避免磁盘IO
2. **单线程模型**：避免上下文切换和锁竞争
3. **IO多路复用**：epoll处理大量连接
4. **高效数据结构**：专门优化的数据结构
5. **简单协议**：RESP协议解析简单

**单线程 vs 多线程：**

| 版本        | 网络IO | 命令执行 | 说明               |
| --------- | ---- | ---- | ---------------- |
| **6.0之前** | 单线程  | 单线程  | 完全单线程            |
| **6.0之后** | 多线程  | 单线程  | 网络IO多线程，命令执行仍单线程 |

**Redis 6.0多线程：**

- 多线程只用于网络IO（读请求、写响应）
- 命令执行仍是单线程（保证原子性）
- 需要配置开启：`io-threads 4`

```
网络IO多线程:                  命令执行单线程:
  ┌──────┐                      
  │Thread│ → 读Socket            ┌────────┐
  └──────┘                       │Main    │
  ┌──────┐                       │Thread  │ → 执行命令
  │Thread│ → 读Socket     ----→  └────────┘
  └──────┘                       
  ┌──────┐                       
  │Thread│ → 读Socket            
  └──────┘                       
```

### 2. Redis有哪些数据结构？各自的使用场景？

**答案：**

| 类型              | 底层实现       | 常用场景          |
| --------------- | ---------- | ------------- |
| **String**      | SDS        | 缓存、计数器、分布式锁   |
| **Hash**        | 哈希表/压缩列表   | 对象存储、用户信息     |
| **List**        | 双向链表/压缩列表  | 消息队列、最新列表     |
| **Set**         | 哈希表/整数集合   | 标签、共同好友、去重    |
| **ZSet**        | 跳表+哈希表     | 排行榜、延迟队列      |
| **Bitmap**      | String     | 签到、在线状态       |
| **HyperLogLog** | 概率算法       | UV统计（误差0.81%） |
| **GEO**         | ZSet       | 地理位置、附近的人     |
| **Stream**      | Radix Tree | 消息队列（5.0+）    |

**常用命令示例（含原始格式）：**

```bash
# String
SET key value EX 3600 NX  # 带过期时间的原子设置（分布式锁）
INCR counter              # 原子自增

# Hash
HSET user:1001 name "Tom" age 20
HGETALL user:1001

# ZSet排行榜
ZADD leaderboard 100 "player1"
ZREVRANGE leaderboard 0 9 WITHSCORES  # Top 10
```

### 3. Redis 各数据结构在参数结构上有什么区别？为什么 ZSet 会多一个 score？

**答案：**

先记一句：

> **所有 Redis 数据结构都有顶层 `key`，但 key 下面装的数据组织方式不一样。只有 `ZSet` 额外引入了 `score`，因为它不仅要存成员，还要排序。**

你可以这样理解：

- `String`：`key -> value`
- `Hash`：`key -> field -> value`
- `List`：`key -> [element, element, ...]`
- `Set`：`key -> {member, member, ...}`
- `ZSet`：`key -> {(member, score), (member, score), ...}`
- `Stream`：`key -> [(id, field-value map), ...]`

所以 `ZADD key score member` 看起来比别的命令多一个参数，是因为：

- `key`：这整个有序集合的名字
- `member`：集合里的成员
- `score`：成员的排序依据

#### 为什么 Set 不需要 score，而 ZSet 需要？

- `Set` 只关心“元素在不在”，不关心顺序
- `ZSet` 关心“元素在不在”，还关心“谁排前面、谁排后面”

所以：

> **Set 解决存在性，ZSet 解决存在性 + 有序性。**

#### 面试里推荐这样答

> Redis 所有数据结构都有顶层 key，但 key 下面的数据组织方式不一样。String 是 key-value，Hash 是 key-field-value，List 是 key 下的一串 element，Set 是 key 下的一组 member，而 ZSet 是 key 下的一组 member 加 score。因为 ZSet 需要按分数排序，所以它比 Set 多了一个 score 字段。

### 4. Redis 的 Set 和 ZSet 有什么区别？什么时候该选 ZSet？

**答案：**

- `Set` 只关心“元素在不在”，不关心顺序，典型场景是去重、标签集合、交并差操作。
- `ZSet` 在 member 的基础上多了一个 score，所以它既能查成员，也能按分数排序、排名、范围查询。
- 如果你的场景需要：
  - 排行榜
  - 按时间排序
  - 按分值做区间查询
  - 取 TopN
    那通常就应该想到 `ZSet`。
- 一句话记：**Set 解决存在性，ZSet 解决“存在性 + 有序性”。**

### 5. Redis 的 ZSet 底层实现是什么？插入和查找过程是怎样的？

**答案：**

- `ZSet` 不是只靠一种结构，它通常同时维护：
  - `dict`：`member -> score`，用来按成员快速查
  - `skiplist`：按 `score` 有序，分数相同时再按 `member` 排序
- 这样设计的原因很直接：
  - 只用哈希表，范围查询和排名很差；
  - 只用跳表，按 member 精确查找又不够快。

**插入流程：**

1. 先在 `dict` 里看 member 是否已经存在
2. 如果不存在：
   - 写入 `dict`
   - 按 score 把节点插入 `skiplist`
3. 如果已存在但 score 变了：
   - 先从跳表删除旧节点
   - 更新 `dict` 中的 score
   - 再按新 score 重新插入跳表

**查找流程：**

- `ZSCORE key member` / 判断成员是否存在：
  - 直接查 `dict`，时间复杂度接近 `O(1)`
- `ZRANGE` / `ZREVRANGE` / 按 score 范围查：
  - 在 `skiplist` 里定位起点，再顺着多层索引和链表往后遍历
- 查排名：
  - Redis 会在跳表节点上维护跨度信息，用于辅助计算 rank

**为什么面试爱问跳表：**

- 跳表实现比平衡树更简单；
- 支持有序范围遍历；
- 插入、删除、查找平均复杂度都是 `O(logN)`；
- 工程上非常适合 Redis 这种以内存操作为主、强调实现简洁的系统。

### 6. Redis 的 ZSet 里，如果两个成员分数相同，怎么让“后来的排前面”？

**答案：**

先给结论：

> Redis 的 ZSet 默认**不会按插入时间来给同分成员排序**。同分时，它会再按 **member 的字典序** 排。

也就是说，如果你只是：

```bash
ZADD rank 100 userA
ZADD rank 100 userB
```

那最终谁在前面，不看“谁先来谁后到”，而是看：

- `userA`
- `userB`

谁的 member 字典序更靠前。

所以如果面试官问：

> 第一个人上午考了 100 分，第二个人下午也考了 100 分，想让第二个排前面，怎么做？

最稳的答法是：

#### 方案 1：把“分数 + 时间”一起编码进 score

如果业务允许 score 不是纯整数分数，而是一个复合排序值，可以把它设计成：

```text
综合分 = 分数主权重 + 时间次权重
```

比如：

- 分数仍然是主要排序维度
- 同分时，用更晚的时间戳让 score 稍微更大

这样：

- 100 分的人整体仍然排在 99 分前面
- 同样 100 分时，下午来的那个 score 会略大一点，所以排前面

#### 方案 2：score 只存分数，member 里编码一个可比较的“逆序时间”

因为 ZSet 同分时会按 member 字典序排序，所以也可以把 member 设计成：

```text
逆序时间戳:用户ID
```

比如你想让“后来的排前面”，就不要直接放正常时间，而是放一个能让“时间越晚，字典序越靠前”的辅助字段。

但这个方案实现上会更绕，代码可读性也差一些，所以工程里更常见的是第一种：

> **直接把 tie-break 信息编码进 score。**

#### 方案 3：业务层显式维护第二排序键

如果你不想污染 score，也不想让 member 变复杂，那就可以在业务层自己维护：

- `score`
- `createTime`

Redis 只负责主排序，真正返回给前端前再做一次业务排序。

不过这种方式的代价是：

- 不能完全依赖 ZSet 一步出最终序
- 应用层逻辑更重

**面试里推荐回答：**

> ZSet 默认同分不是按插入顺序排，而是按 member 字典序排。所以如果题目要求“同分时后来的排前面”，不能直接依赖默认行为。更稳的做法是把时间作为第二排序维度编码进去，最常见是把“分数 + 时间”一起编码到 score 里，让分数决定主排序、时间决定同分下的先后。

### 7. Redis 为什么 List 用 quicklist，ZSet 要用跳表？

**答案：**

- Redis 选底层结构不是为了“某个数据结构最帅”，而是为了在时间复杂度和内存占用之间做平衡。
- List 既要支持两端高效插入删除，又不能像传统链表那样太浪费内存，所以现在更偏向用一种“链表 + 紧凑连续小块”的思路，也就是 quicklist。
- ZSet 既要按成员快速查，也要按分数有序范围查，所以它通常不是单靠一种结构，而是同时利用哈希表和跳表：
  - 哈希表负责按 member 快速定位；
- 跳表负责按 score 排序、范围查询、排名查询。
- 所以一句话总结：List 重点是节省内存并兼顾两端操作，ZSet 重点是“既能快查，又能有序”。

### 7A. Redis 的跳表原理是什么？为什么 ZSet 不直接用红黑树？

**答案：**

- 跳表本质上可以理解成：
  - 最底层是一条有序链表
  - 上面再加多层“快捷索引”
- 查找时不是一格一格往后走，而是先在高层快速跳，跳过大片区间，再逐层往下收敛，所以平均时间复杂度能做到 `O(logN)`。
- 跳表的层数通常靠随机函数决定，常见理解是：

> **每个节点都有概率继续升一层，所以整体会形成“下层密、上层稀”的结构。**

- 为什么 Redis 在 ZSet 上更偏向跳表而不是红黑树：
  1. 范围查询自然：跳表底层本来就是链表，顺着往后扫区间很方便
  2. 实现简单：插入删除不需要像红黑树那样做复杂旋转
  3. 工程可维护性更好：Redis 更偏实用主义，跳表已经足够满足排序和范围查需求

**一句话记忆：**

> 跳表 = 有序链表 + 多层索引；Redis 选它主要是因为实现简单、范围查询自然、工程上够用。

### 7B. 什么是 ziplist / listpack？为什么说它省内存但不适合大数据量？

**答案：**

- `ziplist`（压缩列表）和现在更新版本里的 `listpack`，本质上都是一种**紧凑连续内存结构**。
- 它们的特点是：
  - 元素挨着存，没有传统链表那种大量指针开销
  - 元素少、值短时非常省内存
- 代价是：
  - 修改时可能触发大块内存移动
  - 某些场景还可能出现连锁更新
  - 不适合元素很多、频繁修改的场景

所以 Redis 的思路通常是：

- 小对象 / 小集合时，用紧凑结构省内存
- 超过阈值后，再自动切换到更适合大数据量的结构

**面试里推荐回答：**

> ziplist/listpack 是 Redis 为了节省内存使用的一种紧凑存储结构，元素连续存放，没有多余指针，所以小数据量场景特别省空间；但它的缺点是插入、删除和修改可能触发较大的内存搬移，甚至连锁更新，所以不适合大数据量和高频修改场景。Redis 的策略通常是小对象先用紧凑结构，超过阈值再切到 hashtable、skiplist 这类更适合大规模操作的结构。

## 二、消息队列、延迟任务与去重

### 8. Redis 怎么实现消息队列？Redis 怎么做定时任务 / 延迟任务？

**答案：**

先给结论：

> Redis 能做消息队列，也能做定时 / 延迟任务，但它更适合**轻量、低复杂度、对消息语义要求没那么极致**的场景；如果你要很强的投递保障、复杂消费组治理、海量堆积和回溯能力，通常还是会优先考虑 Kafka / RocketMQ / RabbitMQ 这类专门 MQ（`Message Queue`，消息队列）系统。

---

#### 一、Redis 实现消息队列，常见有哪几种方式？

#### 方式 1：List

最经典的是：

- 生产者：`LPUSH` / `RPUSH`
- 消费者：`LPOP` / `RPOP`
- 阻塞消费：`BLPOP` / `BRPOP`

对应原始格式：

```bash
LPUSH key element [element ...]
RPUSH key element [element ...]
LPOP key
RPOP key
BLPOP key [key ...] timeout
BRPOP key [key ...] timeout
```

例如：

```bash
# 生产者写入消息
RPUSH order_queue "order_1001"

# 消费者阻塞式消费
BLPOP order_queue 0
```

#### 它的优点

- 简单
- 上手快
- 对轻量异步任务很好用

#### 它的问题

- 消费确认能力弱
- 一个消费者取走消息后，如果它处理到一半挂了，这条消息可能就丢了
- 不适合复杂消费组场景

所以 List 更适合：

- 简单异步任务队列
- 内部轻量削峰
- 对消息丢失容忍度稍高的场景

---

#### 方式 2：Pub/Sub

`Pub/Sub` 全称是 `Publish / Subscribe`，中文就是**发布订阅**。

对应原始格式：

```bash
PUBLISH channel message
SUBSCRIBE channel [channel ...]
PSUBSCRIBE pattern [pattern ...]
```

例如：

```bash
# 发布
PUBLISH news_channel "hello"

# 订阅
SUBSCRIBE news_channel
```

#### 它的优点

- 实时广播非常方便
- 适合通知类场景

#### 它的问题

- **消息不落盘**
- 消费者如果当时不在线，就收不到历史消息
- 不适合拿来做可靠消息队列

所以 `Pub/Sub` 更适合：

- 实时通知
- 聊天室广播
- 配置变更广播

而不适合：

- 订单消息
- 支付消息
- 必须可靠消费的任务流

---

#### 方式 3：Stream

`Stream` 是 Redis 5.0 引入的数据结构，很多人会把它理解成 Redis 里更像“正经 MQ”的方案。

常见操作：

- `XADD`：写消息
- `XREAD`：读消息
- `XGROUP`：消费组
- `XREADGROUP`：消费组读取
- `XACK`：确认消息

对应原始格式：

```bash
XADD key * field value [field value ...]
XREAD [COUNT count] [BLOCK milliseconds] STREAMS key [key ...] id [id ...]
XGROUP CREATE key groupname id [MKSTREAM]
XREADGROUP GROUP group consumer [COUNT count] [BLOCK milliseconds] STREAMS key [key ...] id [id ...]
XACK key group id [id ...]
XPENDING key group
```

例如：

```bash
# 写消息
XADD order_stream * orderId 1001 userId 2001

# 创建消费组
XGROUP CREATE order_stream g1 0 MKSTREAM

# 消费组读取
XREADGROUP GROUP g1 c1 COUNT 1 STREAMS order_stream >

# 确认消费
XACK order_stream g1 1712345678901-0
```

#### 它的优点

- 支持消费组
- 支持 ACK
- 支持未确认消息追踪（Pending List）
- 比 List / PubSub 更像成熟队列

#### 它的问题

- 复杂度更高
- 生态和运维成熟度通常不如专业 MQ
- 大规模消息堆积、长时间保留、跨机房流式链路，通常不如 Kafka 这类系统自然

所以如果面试官问：

> Redis 最像 MQ 的能力是什么？

比较稳的回答是：

> **Redis Stream 最像 MQ，因为它有消费组、ACK 和待确认消息管理。**

---

#### 二、Redis 做定时任务 / 延迟任务，最常见怎么做？

最常见方案是：

> **用 ZSet（`Sorted Set`，有序集合）做延迟队列。**

#### 核心思路

- `score`：存任务应该执行的时间戳
- `member`：存任务内容 / 任务 ID

对应原始格式：

```bash
ZADD key score member [score member ...]
ZRANGEBYSCORE key min max [WITHSCORES] [LIMIT offset count]
ZREM key member [member ...]
```

这里一定要记住：

- `delay_queue` 是 Redis 的 **key**
- `1710000010` 是 **score**
- `task_1001` 是 **member**

例如：

```bash
# 当前时间假设是 1710000000
# 10 秒后执行任务
ZADD delay_queue 1710000010 "task_1001"
```

消费者轮询：

1. 取 `score <= now` 的任务
2. 取出来后执行
3. 删除已执行任务

例如：

```bash
ZRANGEBYSCORE delay_queue -inf 1710000010 LIMIT 0 1
ZREM delay_queue task_1001
```

---

#### 为什么 ZSet 适合做延迟任务？

因为它天然按 `score` 有序。

你只要把：

- 执行时间
- 过期时间
- 调度时间

编码进 `score`，就能很方便地找到“现在到期该执行的任务”。

这比 List 更适合“未来某个时间点再执行”的需求。

---

#### 三、Redis 延迟任务有什么坑？

#### 1）轮询会浪费资源

如果消费者一直轮询：

```bash
ZRANGEBYSCORE ...
```

可能会有空转问题。

常见优化：

- 降低轮询频率
- 批量拉取
- 配合本地时间轮 / 调度线程

#### 2）取出和删除要注意原子性

如果你：

1. 先查出任务
2. 再删任务

中间可能被别的消费者抢走。

所以更稳的做法是：

- 用 Lua 脚本把“取出 + 删除”做成原子操作
- 或者先抢占任务状态，再执行

#### 3）任务执行失败怎么办？

这时要考虑：

- 重试次数
- 死信队列
- 幂等处理

否则任务可能反复执行或悄悄丢失。

---

#### 四、消息队列和延迟任务分别怎么回答更稳？

#### 如果面试官问：Redis 怎么实现消息队列？

你可以这样答：

> Redis 做消息队列常见有三种方式：List、Pub/Sub 和 Stream。List 适合轻量异步队列，Pub/Sub 更适合实时广播通知，Stream 更像完整 MQ，支持消费组、ACK 和 Pending 管理。如果只是简单异步削峰，List 就够；如果需要可靠消费和消费组能力，我会优先考虑 Stream。再复杂的大规模消息场景，通常还是用 Kafka 或 RocketMQ 这种专业 MQ 更稳。

#### 如果面试官问：Redis 怎么实现定时任务 / 延迟任务？

你可以这样答：

> 最常见的是用 ZSet 做延迟队列，把任务执行时间戳放进 score，任务内容或任务 ID 放进 member。消费者定期扫描 score 小于等于当前时间的任务并执行。这个方案实现简单，适合中小规模延迟任务；但要注意轮询开销、并发抢任务的原子性，以及失败重试和幂等问题。真正复杂的定时调度系统，通常还会在 Redis 之上叠加业务调度框架。

---

#### 五、一句话总结

> Redis 能做消息队列和延迟任务，但它更像“轻量级方案”：消息队列优先看 List / Stream，定时任务优先看 ZSet；如果追求更强的可靠性、吞吐和治理能力，还是要考虑专业 MQ 或专门调度系统。

### 9. 如果要做 10 亿消息去重，为什么会想到 Redis？有哪些方案？

**答案：**

- 因为去重的核心动作就是“快速判断一个 ID 是否出现过”，Redis 天然适合做高 QPS 的存在性判断。
- 如果要求绝对准确，可以用 `Set/Hash`，但 10 亿级别会非常吃内存，通常要分片。
- 如果 ID 是连续数字，位图很省内存；如果量特别大、允许极低概率误判，可以用布隆过滤器，把空间成本再压下去。
- 高分回答一般会顺手补一句：去重方案没有绝对统一答案，要看“是否允许误判、ID 是否连续、数据保留多久、内存预算多大”。

### 10. 布隆过滤器怎么初始化？它的实现原理是什么？为什么总说它用了多个哈希函数？

**答案：**

先记一句：

> **布隆过滤器（Bloom Filter）的本质，是用一个位数组（bit array）和多个哈希函数，换空间极省、但允许小概率误判的存在性判断结构。**

#### 初始化前要先确定什么？

上线前至少要先估算两个参数：

1. **预计数据量 `n`**
2. **可接受误判率 `p`**

根据这两个参数，才能反推出：

- 位数组大小 `m`
- 哈希函数个数 `k`

#### 它的原理为什么总会讲“多个哈希函数”？

因为布隆过滤器的核心做法就是：

- 对同一个元素，使用 **多个哈希函数**
- 每个哈希函数都会算出一个位置
- 然后把位数组里这些位置都置为 `1`

查询时再用同样的多个哈希函数去检查这些位置：

- 只要有一个位置是 `0`，这个元素就**一定不存在**
- 如果这些位置全是 `1`，它就**可能存在**，但不保证百分之百存在

这就是为什么它会有小概率误判（false positive，假阳性）。

#### 上线时要注意什么？

1. 预估量别太离谱，否则误判率会明显升高
2. 数据持续增长时要考虑重建或分层
3. 普通布隆过滤器不适合精确删除
4. 它不能替代数据库做精确校验，只适合挡掉“一定不存在”的请求

#### 面试里推荐这样答

> 布隆过滤器初始化前，要先根据预计数据量和可接受误判率，规划位数组大小和哈希函数个数。它的底层原理是：对同一个元素使用多个哈希函数，把位数组中的多个位置置为 1；查询时再用同样的哈希函数去检查这些位置。只要有一个位置是 0，就说明这个元素一定不存在；如果这些位置全是 1，则说明它可能存在，所以布隆过滤器会有小概率误判。之所以常说它用了多个哈希函数，是因为这样可以降低冲突和误判率，但同时也要控制计算成本。

## 三、持久化、内存与性能

### 11. Redis的持久化方式有哪些？RDB和AOF的区别？

**答案：**

| 特性        | RDB             | AOF                |
| --------- | --------------- | ------------------ |
| **原理**    | 快照（fork子进程）     | 日志追加               |
| **触发**    | save配置/手动BGSAVE | 每次写命令              |
| **文件**    | dump.rdb（二进制）   | appendonly.aof（文本） |
| **恢复速度**  | 快               | 慢（需重放命令）           |
| **数据完整性** | 可能丢失            | 更完整（取决于fsync策略）    |
| **文件大小**  | 小（压缩）           | 大（可重写）             |

**AOF fsync策略：**

```
appendfsync always    # 每次写入都fsync，最安全最慢
appendfsync everysec  # 每秒fsync，推荐
appendfsync no        # 由OS决定，最快但可能丢数据
```

**Redis 4.0+ 混合持久化：**

```
aof-use-rdb-preamble yes
# AOF文件 = RDB格式的全量数据 + AOF格式的增量数据
# 结合了RDB的快速加载和AOF的数据完整性
```

### 12. Redis的内存淘汰策略有哪些？删除策略有哪些？

**答案：**

**内存淘汰策略（maxmemory-policy）：**

| 策略                  | 说明                |
| ------------------- | ----------------- |
| **noeviction**      | 不淘汰，内存满时写入报错      |
| **allkeys-lru**     | 所有key中LRU淘汰       |
| **allkeys-lfu**     | 所有key中LFU淘汰（4.0+） |
| **allkeys-random**  | 所有key随机淘汰         |
| **volatile-lru**    | 有过期时间的key中LRU     |
| **volatile-lfu**    | 有过期时间的key中LFU     |
| **volatile-random** | 有过期时间的key随机淘汰     |
| **volatile-ttl**    | 淘汰TTL最短的key       |

**过期键删除策略：**

| 策略       | 说明          | Redis使用    |
| -------- | ----------- | ---------- |
| **定时删除** | 到期立即删除（定时器） | 不用（CPU消耗大） |
| **惰性删除** | 访问时检查删除     | ✓ 使用       |
| **定期删除** | 周期性随机检查删除   | ✓ 使用       |

**Redis采用：惰性删除 + 定期删除**

### 13. Redis 内存不够了，应该怎么定位和处理？

**答案：**

- 先别急着加机器，先定位“内存到底被谁吃掉了”。
- 常见排查步骤：
  - 看整体内存使用、淘汰策略是否生效
  - 用 `INFO memory` 看总体情况
  - 用 `MEMORY USAGE key`、`MEMORY STATS` 或采样分析大 key / 热 key
  - 排查是不是 TTL 没设、缓存失控增长、big key、碎片率过高
- 处理方向通常有：
  - 设好 TTL
  - 拆 big key
  - 调整淘汰策略
  - 控制缓存写入量
  - 扩容或分片
- 面试里推荐这样答：**Redis 内存不够不是单一问题，先分清是业务数据涨了、key 设计有问题、big key、还是淘汰策略配置不合理。**

### 14. Redis 慢查询怎么排查？

**答案：**

- Redis 的慢，先别默认是“命令本身慢”，要分成三层看：
  - 命令执行慢
  - 网络慢
  - 阻塞或资源争用导致看起来慢
- 常见排查顺序：
  1. 看 `SLOWLOG GET`，确认是不是慢命令、慢在哪些命令
  2. 看 `INFO commandstats`、`INFO stats`，确认热点命令和整体吞吐
  3. 排查是否有 `big key`、大范围扫描、Lua 脚本、一次性返回太多数据
  4. 看 CPU、内存、碎片率、网络带宽、主从复制积压
  5. 确认是不是持久化、AOF rewrite、RDB fork 抖动带来的毛刺
- 高频根因通常有：
  - `KEYS *`、大范围 `ZRANGE`、大 `HGETALL`
  - big key 导致单次序列化/网络传输过大
  - Lua 脚本执行时间长，阻塞主线程
  - 客户端连接池或网络抖动，不一定是 Redis 本身算得慢
- 处理方向一般是：
  - 把全量扫描改成游标扫描，比如 `SCAN`
  - 拆 big key
  - 控制单次返回量
  - 缩短 Lua 脚本执行时间
  - 优化持久化和实例资源配置
- 面试里可以一句话总结：**先用 `slowlog` 定位慢命令，再区分是命令复杂度、数据模型问题，还是网络/持久化抖动。**

## 四、缓存、热 key 与一致性

### 15. 如何保证本地缓存、Redis 缓存和 MySQL 的一致性？请按难度从小到大讲一遍。

**答案：**

先记一句总纲：

> **缓存一致性很难做到“绝对强一致”，工程上更常见的是按复杂度分层：先做到可用，再做到最终一致，再做到链路统一治理。**

如果按难度从小到大，最常见可以这样讲：

---

#### 方案 1：Cache Aside（旁路缓存模式）——最基础、最常用

这是最经典的模式。

##### 读流程

1. 先查 Redis
2. Redis miss，再查 MySQL
3. 查到后回填 Redis

##### 写流程

1. 先更新 MySQL
2. 再删除 Redis 缓存

你可以记成：

```text
读：cache -> db -> 回填 cache
写：update db -> delete cache
```

#### 为什么不是“先更新 DB 再更新缓存”？

因为更新缓存看起来直接，实际更容易出问题：

- 并发下容易被旧值覆盖
- 多服务场景更难统一治理
- 业务改动多时，更新逻辑分散

所以工程上更常见的是：

> **更新数据库，删除缓存；下次读再自然回填。**

#### 这个方案的优点

- 最简单
- 成本最低
- 大多数中小业务都能落地

#### 它的问题

- 删缓存失败会不一致
- 在高并发下，可能出现短时间脏读

所以这通常是**第一层**方案。

---

#### 方案 2：延迟双删——在 Cache Aside 上做增强

它的思路是：

1. 先删一次缓存
2. 更新 MySQL
3. 过一小段时间，再删一次缓存

例如：

```go
func UpdateData(id string, data Data) error {
    redis.Del(id)

    if err := db.Update(id, data); err != nil {
        return err
    }

    go func() {
        time.Sleep(500 * time.Millisecond)
        redis.Del(id)
    }()

    return nil
}
```

#### 为什么要删两次？

因为可能出现这种并发：

1. 线程 A 更新数据库
2. 线程 B 正好缓存 miss，去查数据库
3. B 查到的是旧值或时机不巧的数据
4. B 又把旧值写回 Redis

延迟再删一次，就是为了尽量把这类“并发回填旧值”再清掉。

#### 优点

- 比单纯 Cache Aside 更稳一点
- 实现仍然不复杂

#### 缺点

- 延迟多久没有标准答案
- 仍然不是强一致
- 对复杂链路帮助有限

所以延迟双删一般适合：

> **单体 / 中小服务，能接受短暂不一致，但又想比普通 Cache Aside 更稳一点。**

---

#### 方案 3：Pub/Sub（发布订阅）——轻量广播失效通知

如果不只是一个实例，而是：

- 每台机器都有本地缓存
- Redis 上还有分布式缓存

那更新数据库后，除了删 Redis，还可能要通知所有实例把本地缓存一起失效。  
这时就会有人想到 Redis 的 `Pub/Sub`。

思路大概是：

1. 更新 MySQL
2. 删除 Redis
3. 发布一个失效消息
4. 所有应用实例订阅这个频道，收到后删除本地缓存

```text
update db -> delete redis -> publish invalidation event -> 各实例清本地缓存
```

#### 优点

- 比纯轮询优雅
- 实现轻量
- 很适合做“缓存失效广播”

#### 缺点

- `Pub/Sub` 本身不可靠，不保证离线消费者补收
- 订阅者挂掉时，消息可能丢
- 更适合通知，不适合强一致链路

所以你可以这样理解：

> **Pub/Sub 更像本地缓存失效广播方案，不是强可靠最终一致方案。**

---

#### 方案 4：MySQL binlog + Canal / CDC + Kafka——更工程化的最终一致方案

这是更常见的大厂方案。

核心思路是：

1. 业务只更新 MySQL
2. MySQL 把变更写入 `binlog`（binary log，二进制日志）
3. Canal / CDC（Change Data Capture，变更数据捕获）订阅 binlog
4. 变更事件写入 Kafka
5. 下游消费者统一删除 / 更新 Redis，或者通知各实例清本地缓存

链路可以理解成：

```text
MySQL -> binlog -> Canal / CDC -> Kafka -> Redis / 本地缓存失效
```

#### 为什么这套更强？

因为它把“缓存更新”从业务代码里解耦出来了：

- 业务代码只负责写数据库
- 数据变更统一从 binlog 出来
- 下游可以统一处理 Redis、本地缓存、搜索索引、ES 等多个消费者

#### 优点

- 更适合多服务、多消费者场景
- 统一治理双写问题
- 更适合大规模系统演进

#### 缺点

- 链路复杂
- 运维成本更高
- 仍然是最终一致，不是写完立刻全局绝对同步

所以这种方案更适合：

> **多服务、大流量、需要统一治理缓存与下游同步链路的大型系统。**

---

#### 本地缓存 + Redis + MySQL 三层怎么一起看？

很多面试不是只问 “MySQL 和 Redis 一致性”，而是会继续追问：

> 如果我还有本地缓存怎么办？

这时你要分三层理解：

1. **MySQL**：事实来源（source of truth）
2. **Redis**：分布式共享缓存
3. **本地缓存**：实例内热点缓存

一致性越往上，通常越弱：

```text
MySQL 最准
Redis 追求最终一致
本地缓存通常是最快但最容易脏的一层
```

所以常见思路是：

- 更新 MySQL
- 失效 Redis
- 再通过 Pub/Sub / MQ / 订阅事件 失效本地缓存

---

#### 面试里推荐这样答

> 如果按难度从小到大讲缓存一致性，我一般会这样回答：最基础的是 Cache Aside，也就是读走缓存旁路，写时先更新 MySQL 再删除 Redis；如果想进一步降低并发脏读风险，可以用延迟双删；如果系统里还有本地缓存，可以用 Redis Pub/Sub 做轻量失效广播；再复杂一点的大型系统，通常会走 MySQL binlog + Canal/CDC + Kafka 这条链路，把数据库变更统一分发给 Redis、本地缓存或其他下游系统做最终一致同步。总体上，简单方案成本低，复杂方案治理能力更强，但一致性语义和系统复杂度也会一起上升。

**一句话总结：**

> **缓存一致性没有银弹，常见演进路径就是：Cache Aside -> 延迟双删 -> 本地缓存广播失效 -> binlog/CDC/Kafka 统一治理。**

---

### 16. 什么是缓存穿透、缓存击穿、缓存雪崩？如何解决？

**答案：**

| 问题       | 描述                | 解决方案                            |
| -------- | ----------------- | ------------------------------- |
| **缓存穿透** | 查询不存在的数据，每次都打到DB  | 1.布隆过滤器 2.缓存空值                  |
| **缓存击穿** | 热点key过期，大量请求打到DB  | 1.热点key不过期 2.互斥锁 3.singleflight |
| **缓存雪崩** | 大量key同时过期/Redis宕机 | 1.过期时间加随机值 2.多级缓存 3.集群          |

**缓存穿透解决：**

```go
// 布隆过滤器
if !bloomFilter.Exists(id) {
    return nil // 一定不存在
}

// 缓存空值
data := redis.Get(id)
if data == "NULL" {
    return nil
}
if data == "" {
    dbData := db.Query(id)
    if dbData == nil {
        redis.SetEX(id, "NULL", 60) // 缓存空值60秒
    }
}
```

**缓存击穿解决：**

```go
// singleflight
var g singleflight.Group
data, err, _ := g.Do(key, func() (interface{}, error) {
    return db.Query(key)
})
```

**缓存雪崩解决：**

```go
// 过期时间加随机值
expiration := baseExpire + rand.Intn(300) // 基础+0~300秒随机
redis.SetEX(key, value, expiration)
```

### 17. 项目里 Redis 一般存什么？key 设计要注意什么？

**答案：**

- 这个问题没有标准唯一答案，但可以按类型回答：
  - 热点业务数据缓存
  - 会话 / token
  - 计数器
  - 排行榜
  - 去重状态
  - 分布式锁
- key 设计上我一般会强调四点：
  - 语义清晰：`业务名:对象类型:对象ID`
  - 便于排查：不要全是无意义短串
  - 控制长度：别把 key 写得过长
  - 规划 TTL：不是所有 key 都该永久存在
- 例如：
  - `user:profile:123`
  - `post:like_count:456`
  - `lock:order:789`

### 18. Redis 和本地缓存怎么配合？

**答案：**

- Redis 更适合做分布式共享缓存，本地缓存更适合做单机极低延迟热点缓存。
- 常见链路是：
  - 先查本地缓存
  - 本地 miss 再查 Redis
  - Redis miss 再回源 DB
- 这样做的好处是热点请求不一定每次都穿透到 Redis，延迟也更低。
- 难点主要有两个：
  - 一致性
  - 命中率
- 所以这题别只答“多加一层缓存”，要补一句：**多级缓存的收益和代价是一起来的。**

### 19. 本地缓存如何固定实例访问？为什么需要这样做？

**答案：**

- 本地缓存是实例内缓存，如果请求每次随机打到不同实例，本地缓存命中率会很差。
- 所以热点业务里经常会通过：
  - 一致性哈希
  - 会话亲和
  - 按用户 ID / 业务 key 路由
    让同一类请求尽量落到固定实例。
- 这样做的目标不是“绝对固定”，而是提高局部热点命中率，减少 Redis 和数据库压力。
- 面试里推荐这样答：**本地缓存要发挥价值，通常要配合路由策略；否则请求被打散，每台机器都只命中一点点，本地缓存收益会很有限。**

### 20. Redis 和 MySQL 做缓存一致性时，延迟双删和 Canal / CDC 分别适合什么场景？

**答案：**

- 两种方案本质上都不是强一致，而是在不同复杂度下追求更稳的最终一致。

| 方案              | 核心思路                      | 优点               | 缺点               | 适用场景                  |
| --------------- | ------------------------- | ---------------- | ---------------- | --------------------- |
| **延迟双删**        | 删缓存 -> 更新 DB -> 延迟再删一次缓存  | 实现简单，接入成本低       | 延迟时间难拿捏，仍可能有短暂脏读 | 单体或中小规模业务，能接受短暂不一致    |
| **Canal / CDC** | 业务只写 DB，订阅 binlog 异步删/更缓存 | 业务代码解耦，可统一治理双写问题 | 链路更复杂，有消费延迟和补偿成本 | 多服务、多数据消费者、需要统一数据同步链路 |

- 如果面试官追问“该怎么选”，可以这样答：
  - 业务简单、写流量不高、团队想快速落地，先用 `cache aside + 删除缓存`，必要时补 `延迟双删`。
  - 如果已经有 `binlog -> MQ / Canal / CDC` 这类基础设施，或者一个数据库变更要驱动多个下游系统，那更适合走 `CDC`。
- 但无论哪种方案，都要补兜底：
  - TTL
  - 删除失败重试
  - 幂等消费
  - 监控和补偿任务
- 高频答法就是：**延迟双删胜在简单，Canal / CDC 胜在解耦和统一治理；选型看一致性要求、链路复杂度和团队基础设施。**

### 21. Redis 集群里为什么还会有热 key 问题？怎么治理？

**答案：**

- 很多人会误以为“上了 Redis Cluster，热点就自然分散了”，其实不对。
- `Cluster` 只是把不同 key 分散到不同槽位、不同节点上，但**单个 key 永远只会落到一个槽位、一个主节点上**。
- 所以如果某一个 key 的访问量特别大，比如首页配置、爆款商品库存、热门榜单，这个 key 还是会把对应分片打爆，这就是热 key 问题。

**为什么集群也扛不住单个热 key：**

- 一个 key 只能属于一个 slot
- 一个 slot 只归某个 master 管
- 大量请求会集中打到这一个分片
- 结果可能是：
  - 单分片 CPU 飙高
  - 网络带宽打满
  - 响应变慢
  - 甚至把对应下游 DB 一起拖垮

**常见场景：**

- 秒杀库存 key
- 热门商品详情 key
- 热榜 / TopN key
- 全站配置 key
- 某个超高频计数器

**怎么排查：**

- 看业务监控，确认是不是某类请求异常集中
- 看 `INFO commandstats`、慢日志、实例 QPS
- 抽样分析访问最频繁的 key
- 看是不是某个分片负载明显高于其他分片

**治理思路一般分四类：**

**1. 本地缓存 / 多级缓存**

- 对特别热、变化不频繁的数据，在应用层再加一层本地缓存
- 让大量读请求先命中本地，不直接打 Redis
- 这对配置类、详情类热点最常见

**2. 热 key 拆分**

- 如果业务允许，可以把一个热点 key 拆成多个子 key
- 例如把计数器拆成：
  - `counter:hot:1`
  - `counter:hot:2`
  - `counter:hot:3`
- 写入时分摊到多个 key，读取时再聚合
- 这样能把压力从一个分片摊到多个分片

**3. 热点隔离**

- 把特别热的数据单独放到专门实例或专门分片
- 不要让它和普通业务 key 混在一起
- 这样即使热点被打爆，也不会拖垮整个缓存集群

**4. 限流 / 降级 / 预热**

- 热点流量过高时要配合限流
- 某些非核心数据可以直接降级返回兜底值
- 活动开始前提前预热缓存，避免瞬时回源

**如果是秒杀库存这类强热点，还要注意：**

- 不要每次都查数据库
- 通常是 Redis 原子扣减 + MQ 异步落库
- 必要时再加 Lua、限流、资格校验，避免所有请求都直接冲击同一个库存 key

**如果面试官继续追问“具体怎么选治理方案”，可以这样答：**

#### 场景 1：读多写少的热点

比如：

- 首页配置
- 热门商品详情
- 公共字典数据

这类数据的特点是：

- 读非常多
- 写很少
- 通常可以接受秒级更新延迟

更让面试官满意的答法是：

> 我会优先上**本地缓存 + Redis 二级缓存**。因为这类热点的核心问题是读流量过大，而不是写冲突。只要数据不是频繁变化，让大量请求先命中本地缓存，收益最高，也最直接。

#### 场景 2：写多读多的热点计数

比如：

- 点赞数
- 浏览数
- 热榜分值

这类热点的特点是：

- 不只是读高
- 写也很集中
- 单 key 很容易成为热点瓶颈

更稳的答法是：

> 这类场景我会优先考虑**拆 key / 分片计数**。因为只靠本地缓存解决不了写热点，本质还是得把一个热 key 的压力拆散到多个 key 或多个分片上，最后异步聚合结果。

#### 场景 3：强一致热点

比如：

- 库存
- 余额
- 优惠券余量

这类场景的高分答法不是“继续猛打 Redis”，而是：

> 强一致热点不要把 Redis 当成最终真相源。Redis 可以做前置拦截和原子扣减，但最终一致性还是要靠数据库事务、MQ 补偿、幂等校验来兜底。也就是说，Redis 负责扛流量，数据库负责兜正确性。

#### 场景 4：活动型瞬时热点

比如：

- 秒杀开始瞬间
- 热门直播间
- 大促首页流量

这类问题很讲究组合拳，面试里推荐这样答：

> 我会用“预热 + 限流 + 降级 + 隔离”一起做。提前把热点数据预热进缓存，入口做限流，非核心请求能降级就降级，同时把热点业务和普通业务隔离到不同实例或不同分片，避免热点把全局拖垮。

**一个更像大厂面试的总结方式：**

- 读热点：优先本地缓存 / 多级缓存
- 写热点：优先拆 key / 分片计数
- 强一致热点：Redis 抗流量，DB 保正确
- 活动热点：预热、限流、降级、隔离一起上

如果面试官问“为什么不是一种方案打天下”，你可以直接说：

> 因为热 key 不是单一问题，关键要先看它是读热点、写热点，还是强一致热点。不同热点的瓶颈不一样，治理手段也不能一把梭。 

**一句话总结：**

> Redis Cluster 解决的是“多 key 的横向扩容”，不是“单 key 的无限扩容”。热 key 的本质是单点过热，所以治理重点是多级缓存、拆 key、热点隔离和流量控制。

## 五、锁、事务与并发控制

### 22. Redis分布式锁的流程是什么？

**答案：**

先记一句最核心的话：

> **它之所以叫分布式锁，是因为这把锁不是只在单机进程里可见，而是借助 Redis 这种所有服务实例都能访问到的共享存储，来实现“跨机器、跨进程”的互斥。**

也就是说：

- 普通锁（如 `synchronized`、`Mutex`）通常只能管住**同一个进程内部**的线程
- Redis 分布式锁要解决的是：**多个服务实例、多个进程、甚至多台机器之间，谁能先处理同一份共享资源**

比如：

- 多个实例同时抢同一个订单
- 多个消费者同时处理同一条任务
- 多个节点同时执行同一个定时任务

这时大家都会去竞争同一个 Redis key，例如：

```bash
SET lock:order:1001 unique_value NX EX 30
```

谁先设置成功，谁就拿到锁；其他实例因为 `NX` 失败，就知道锁已经被别人持有。  
所以它本质上不是“单机锁”，而是：

> **把锁状态放在分布式系统共享可见的位置上，让所有节点对同一把锁达成共识。**

`unique_value` 的作用也很关键：

- 它表示“这把锁是谁拿到的”
- 解锁时不能直接 `DEL`，而要先校验 value 还是不是自己的
- 否则就可能把别人后来重新拿到的锁误删掉

所以更完整的理解是：

> Redis 分布式锁 = 共享存储中的互斥标记 + 原子加锁 + 带身份校验的安全解锁。

**基本实现：**

原始格式先记住：

```bash
SET key value NX EX seconds
SET key value NX PX milliseconds
EVAL script numkeys key [key ...] arg [arg ...]
```

示例：

```bash
# 加锁（SET NX EX）
SET lock_key unique_value NX EX 30
# NX: 不存在才设置
# EX 30: 30秒过期（防止死锁）

# 解锁（Lua脚本保证原子性）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**Go实现：**

```go
func TryLock(key string, value string, expire time.Duration) bool {
    return redis.SetNX(ctx, key, value, expire).Val()
}

func Unlock(key string, value string) bool {
    script := `
    if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
    else
        return 0
    end`
    return redis.Eval(ctx, script, []string{key}, value).Val().(int64) == 1
}
```

**问题与解决：**
| 问题 | 解决方案 |
|------|---------|
| 锁过期业务未完成 | 看门狗自动续期 |
| 主从切换锁丢失 | Redlock算法 |
| 不可重入 | 记录持有者+计数 |

### 23. Redisson分布式锁是怎么实现的？

**答案：**

**核心特性：**

1. **可重入**：记录线程ID和重入次数
2. **自动续期**：看门狗机制（默认30秒，10秒检查一次）
3. **公平锁**：支持等待队列

**加锁Lua脚本逻辑：**

```lua
-- KEYS[1]: 锁名
-- ARGV[1]: 过期时间
-- ARGV[2]: 线程标识 (UUID + ThreadId)

if redis.call('exists', KEYS[1]) == 0 then
    -- 锁不存在，获取锁
    redis.call('hset', KEYS[1], ARGV[2], 1)
    redis.call('pexpire', KEYS[1], ARGV[1])
    return nil
end
if redis.call('hexists', KEYS[1], ARGV[2]) == 1 then
    -- 重入
    redis.call('hincrby', KEYS[1], ARGV[2], 1)
    redis.call('pexpire', KEYS[1], ARGV[1])
    return nil
end
-- 获取失败，返回剩余时间
return redis.call('pttl', KEYS[1])
```

**看门狗续期：**

```java
// 后台线程定时续期
scheduleExpirationRenewal() {
    // 每 lockWatchdogTimeout/3 检查一次（默认10秒）
    // 如果锁还被持有，续期到 lockWatchdogTimeout（默认30秒）
}
```

### 24. Redis 分布式锁在什么情况下会死锁或失效？

**答案：**

- 最典型的问题是“加锁成功了，但没设置过期时间”或者“设置过期时间和加锁不是原子操作”，这样服务一挂就可能死锁。
- 第二类问题是业务执行时间太长，锁已经过期并被别人抢走，但当前线程还以为锁在自己手里，最后把别人的锁删掉了。
- 第三类问题出现在主从切换：主节点刚加完锁还没来得及同步到从节点就挂了，新主节点上可能根本没有这把锁。
- 面试里比较稳的回答是：用 `SET key value NX PX` 保证加锁原子性，value 放唯一标识，释放时用 Lua 校验“是不是自己的锁”，必要时配合续期；如果对一致性要求极高，还要考虑多副本法定多数或其他更强方案。

### 25. Redis 如何实现 CAS？

**答案：**

- Redis 没有像数据库那样直接提供通用 CAS 关键字，但可以通过 `WATCH + MULTI/EXEC` 做乐观锁风格的条件更新。
- 基本思路是：
  - 先 `WATCH key`
  - 读旧值并判断
  - 开启事务写入
  - 如果中间 key 被别人改过，`EXEC` 会失败
- 更复杂的场景里，通常会把“读取 + 判断 + 更新”封进 Lua 脚本，保证整个过程原子执行。
- 所以面试里抓住一句话就够：**Redis CAS 的本质是条件更新 + 冲突检测 + 原子执行。**

### 26. Redis 分布式锁的超时时间怎么评估？

**答案：**

- 这题不能拍脑袋说一个固定值，核心看：
  - 业务执行平均耗时
  - 业务执行最长耗时
  - 网络抖动和 GC / STW 毛刺
  - 是否有续期机制

**最稳的思路是：**

- 如果没有续期机制：
  - TTL 至少要覆盖“正常业务执行时间 + 一定安全余量”
- 如果有续期机制（比如 Redisson WatchDog）：
  - 初始 TTL 可以相对保守一些
  - 但仍然不能短到业务刚开始就频繁续租

**为什么不能太短：**

- 业务还没做完，锁先过期
- 别人抢到锁后，两个线程可能同时操作同一份数据

**为什么不能太长：**

- 真挂掉时，别人得等很久
- 锁恢复速度太慢

**面试里推荐这样答：**

> Redis 锁的超时时间不是固定值，而是要按业务执行时间来估。太短会导致业务没做完锁先过期，太长又会让故障恢复很慢。通常我会按“正常耗时 + 安全余量”来定，如果业务时间波动大，再配合续期机制，而不是拍脑袋写个 30 秒就结束。

### 27. Lua脚本为什么能保证原子性？

**答案：**

**原因：Redis是单线程执行命令**

1. **Lua脚本整体执行**：Redis会将整个Lua脚本作为一个命令执行
2. **不会被打断**：执行期间不会执行其他客户端命令
3. **类似事务**：但比MULTI/EXEC更强，支持条件判断

**与MULTI/EXEC对比：**
| 特性 | Lua脚本 | MULTI/EXEC |
|------|---------|-----------|
| 条件判断 | ✓ 支持 | ✗ 不支持 |
| 中间结果 | ✓ 可获取 | ✗ 命令排队 |
| 原子性 | 完全原子 | watch乐观锁 |
| 网络开销 | 一次往返 | 多次往返 |

**示例：原子性的CAS操作**

```lua
-- 在MULTI/EXEC中无法实现的条件更新
local current = redis.call('get', KEYS[1])
if current == ARGV[1] then
    redis.call('set', KEYS[1], ARGV[2])
    return 1
end
return 0
```

### 28. Redis保证incr命令原子性的原理是什么？

**答案：**

**原理：单线程 + 内存操作**

1. **单线程**：同一时刻只有一个命令在执行
2. **命令队列**：所有命令排队顺序执行
3. **不存在并发**：不需要锁

**INCR执行流程：**

```
Client A: INCR counter
Client B: INCR counter

Redis处理:
1. 从队列取出A的INCR
2. 读取counter=0，+1，写入counter=1，返回1
3. 从队列取出B的INCR
4. 读取counter=1，+1，写入counter=2，返回2
```

**即使是6.0多线程IO：**

- 多线程只处理网络IO
- 命令执行仍是单线程
- INCR操作仍然原子

### 29. Redis 锁和 ZooKeeper 锁怎么选？

**答案：**

这题别答成“谁绝对更强”，更稳的说法是：

> **Redis 锁更偏性能和易用性，ZooKeeper 锁更偏强协调语义。**

#### Redis 锁的特点

- 实现简单
- 性能高
- 延迟低
- 适合高并发、短临界区场景

但它的问题是：

- 过期时间不好估容易失效
- 主从切换时可能有锁丢失风险
- 更依赖业务方自己做好幂等和补偿

#### ZooKeeper 锁的特点

- 更偏强协调
- 临时节点 + 顺序节点语义清晰
- 更适合强一致协调场景

但它的问题是：

- 性能通常不如 Redis 锁
- 实现和运维成本更高
- 不适合拿来扛特别高频、极短临界区的热点锁竞争

#### 怎么选？

- 如果你要的是：
  
  - 高性能
  - 简单分布式互斥
  - 业务上能接受少量复杂性由应用兜底
  - 那通常优先 Redis 锁

- 如果你要的是：
  
  - 更强协调语义
  - 对锁的正确性要求极高
  - 分布式主从选举、协调控制这类场景
  - 那 ZooKeeper 更合适

#### 面试里推荐这样答

> Redis 锁更适合高性能、短临界区、业务层可做幂等补偿的场景；ZooKeeper 锁更适合对协调一致性要求更高的场景，比如分布式协调和选主。前者强在性能和简单，后者强在协调语义更稳，所以不是谁一定更好，而是场景不同。

## 六、部署、集群与客户端

### Redis、MySQL、Kafka 常见部署模式怎么对比？哪些模式是“主节点可写，从节点只读”？

**答案：**

这题很适合放在一起记，因为三个中间件都常见“单点 / 主从 / 高可用 / 集群”这些模式，但读写语义不完全一样。

#### 先给一个总表

| 系统    | 模式                         | 是否有主从角色 | 谁可写                    | 谁可读                        | 典型特点     |
| ----- | -------------------------- | ------- | ---------------------- | -------------------------- | -------- |
| Redis | 单机                         | 无       | 单节点可写                  | 单节点可读                      | 简单，没高可用  |
| Redis | 主从复制                       | 有       | **主节点写**               | **从节点一般只读**                | 读扩展、备份   |
| Redis | Sentinel                   | 有       | **主节点写**               | **从节点一般只读**                | 自动故障转移   |
| Redis | Cluster                    | 有多个主分片  | **每个 master 负责自己槽位的写** | 对应 master/slave 可读         | 分片扩容     |
| MySQL | 单机                         | 无       | 单节点可写                  | 单节点可读                      | 简单       |
| MySQL | 主从复制                       | 有       | **主库写**                | **从库通常读**                  | 读写分离     |
| MySQL | MGR / InnoDB Cluster（单主）   | 有       | **单主写**                | 其他节点读                      | 强一致更强    |
| Kafka | 单 Broker                   | 无明显主从   | Broker 可写可读            | 同一 Broker 可读               | 没高可用     |
| Kafka | 多 Broker + Leader/Follower | 有分区主从   | **分区 Leader 处理写**      | 通常从 Leader 读，Follower 主要同步 | 高吞吐、分区副本 |

#### Redis 怎么看？

- **单机**：一个节点自己读写
- **主从复制**：
  - 主节点负责写
  - 从节点主要同步主节点数据，常见用于读扩展
  - 从节点默认一般不建议业务写入
- **Sentinel**：
  - 本质还是一主多从
  - 只是多了哨兵帮你自动故障转移
  - 所以依然是：**主可写，从只读**
- **Cluster**：
  - 不是一个全局主，而是**多个 master 分片**
  - 每个 master 负责一部分 slot（槽位）
  - 所以可以理解成：**每个 master 对自己负责的槽位可写，从节点主要做复制和故障转移**

#### MySQL 怎么看？

- **单机**：自己读写
- **主从复制**：
  - 主库写
  - 从库一般承担读
  - 经典读写分离模式
- **MGR / InnoDB Cluster（单主）**：
  - 通常一个主节点负责写
  - 其他节点读
  - 比传统异步主从的一致性能力更强

#### Kafka 怎么看？

Kafka 虽然也有 Leader / Follower，但它和 Redis/MySQL 那种“整库主从”不完全一样。  
它是：

- **按 Partition（分区）来选 Leader**
- 某个分区的写请求，只能发给这个分区的 Leader
- Follower 主要负责复制同步

所以更准确地说：

> **Kafka 不是“整个集群一个主”，而是“每个分区各有自己的 Leader”。**

#### 面试里怎么答更稳？

> Redis、MySQL、Kafka 都有单点和高可用模式，但主从语义不完全一样。Redis 主从和 Sentinel 本质上是一主多从，通常主节点可写、从节点只读；Redis Cluster 是多主分片，每个 master 负责自己槽位的写。MySQL 经典主从也是主库写、从库读；更复杂的 MGR/InnoDB Cluster 通常也常见单主写。Kafka 则是按分区选 Leader，每个 Partition 的写都发给自己的 Leader，Follower 主要负责复制，所以它更像“分区级主从”，不是整库一个主。

### 30. Redis的部署方式有哪些？各自的优缺点？

**答案：**

| 部署方式           | 优点       | 缺点          | 适用场景 |
| -------------- | -------- | ----------- | ---- |
| **单机**         | 简单       | 无高可用、容量受限   | 开发测试 |
| **主从复制**       | 读扩展、数据备份 | 手动故障转移      | 读多写少 |
| **Sentinel哨兵** | 自动故障转移   | 写不可扩展、资源浪费  | 中小规模 |
| **Cluster集群**  | 水平扩展、高可用 | 架构复杂、部分功能受限 | 大规模  |

**主从复制：**

```
Master (写) ─────→ Slave1 (读)
           └────→ Slave2 (读)
```

**Sentinel哨兵：**

```
    Sentinel1   Sentinel2   Sentinel3
         ↓          ↓          ↓
        监控      监控       监控
         ↓          ↓          ↓
    ┌─────────────────────────────┐
    │  Master ←→ Slave1 ←→ Slave2 │
    └─────────────────────────────┘
```

**Cluster集群：**

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ Master1  │  │ Master2  │  │ Master3  │
│ (0-5460) │  │(5461-10922)│ │(10923-16383)│
│    ↓     │  │    ↓     │  │    ↓     │
│ Slave1   │  │ Slave2   │  │ Slave3   │
└──────────┘  └──────────┘  └──────────┘
```

缓存一致性问题，本质上是 **MySQL 和 Redis 双写时，数据库和缓存可能出现短暂不一致**。

面试里可以这样简洁回答：

- **常用方案是 Cache Aside**：读请求先查缓存，没命中再查数据库并回填；写请求先更新数据库，再删除缓存。

- **为什么删除缓存，不是更新缓存**：因为更新缓存更复杂，容易在并发下覆盖脏数据；删除缓存更简单，下一次读会自动回源重建。

- **它解决不了强一致，只能做到最终一致**：典型风险是删缓存失败，或者并发读把旧数据重新写回缓存。

- **常见优化手段**：延迟双删、删除失败重试、MQ 异步补偿、监听 binlog 删除缓存，再配合 TTL 兜底。

- **如果业务特别敏感**：比如余额、库存这类场景，优先保证数据库事务和并发控制，缓存主要承担读性能优化，不依赖缓存保证强一致。

### 31. Jedis 和 Lettuce 有什么区别？

**答案：**

- `Jedis` 更偏同步阻塞模型，使用方式直观，但单个连接对象通常不建议多线程共享，所以实际项目里经常会配连接池。
- `Lettuce` 基于 Netty，支持异步、响应式和更好的并发复用，很多场景下一个连接可以被多个线程安全地使用。
- 如果项目只是简单同步调用，`Jedis` 上手更直接；如果并发高、链路长，或者你本身就是异步/reactive 风格，`Lettuce` 往往更合适。
- 面试里一句话概括就是：`Jedis` 更简单，`Lettuce` 更现代、并发能力更强。

### 32. Redis Sentinel 和 Cluster 在故障恢复上有什么区别？

**答案：**

- Sentinel 解决的是“高可用”，本质上还是一主多从。主节点挂了以后，哨兵会选一个从节点升主，然后通知客户端切换。
- Cluster 解决的是“高可用 + 水平扩容”，它把 16384 个槽位分散到多个主节点上。某个主节点挂了，只会影响它负责的那部分槽位，恢复时也是这个分片内部完成故障转移。
- 所以可以简单记：Sentinel 更像单分片高可用，Cluster 是分片后的高可用。
- 如果数据量和写压力还没到要分片的程度，Sentinel 更简单；如果已经要横向扩容，一般就会走 Cluster。

### 33. Redis 选主机制是什么？Sentinel 和 Cluster 分别怎么选主？

**答案：**

- 这题别只答“投票”，要先区分场景：
  - `Sentinel` 是一主多从的故障转移；
  - `Cluster` 是分片集群里某个主分片故障后的故障转移。
- `Sentinel` 的大致流程是：
  1. 多个哨兵先判断主节点是否主观下线；
  2. 达到法定票数后认定客观下线；
  3. 哨兵之间通过 Raft 风格的选举先选出一个 leader sentinel；
  4. 由 leader sentinel 挑一个从节点提升为新主。
- `Sentinel` 挑从节点时一般会看三件事：
  - `replica-priority`，优先级越小越优先
  - 复制偏移量，越接近原主越好
  - run id 等稳定排序条件
- `Cluster` 的故障转移是分片内完成的：
  - 某个 master 挂掉后，它的从节点会发起竞选；
  - 集群里其他 master 给它投票；
  - 拿到多数 master 的票后，该 slave 晋升为新 master，并接管原来的槽位。
- 一句话记忆：
- `Sentinel` 是“哨兵先选 leader，再由 leader 选新主”；
- `Cluster` 是“故障分片的从节点自己竞选，其他 master 投票”。

### 34. Redis 三种部署模式怎么对比？单机、Sentinel、Cluster 分别适合什么场景？

**答案：**

- `单机`：最简单，没有高可用，也不做分片，适合本地开发、测试环境或低要求的小场景。
- `Sentinel`：本质上还是一主多从，解决的是**高可用**问题，主节点挂了可以自动故障转移，但数据本身不分片。
- `Cluster`：既解决**高可用**，也解决**水平扩容**。数据会按 `16384` 个槽位分散到多个主节点上，每个主节点负责一部分槽位。
- 面试里一句话记：
  - `Sentinel` 更偏“高可用”
  - `Cluster` 更偏“高可用 + 大数据量 + 水平扩展”

**推荐回答：**

> Redis 常见部署可以分成单机、Sentinel 和 Cluster。单机最简单但没有高可用；Sentinel 是一主多从加自动故障转移，解决的是高可用问题，但数据不分片；Cluster 则是在高可用基础上再做分片，既能自动故障转移，也能水平扩容。所以如果只是想提高可用性，Sentinel 就够；如果数据量和写压力都上来了，一般就会走 Cluster。

### 35. Redis Cluster 为什么是 16384 个槽？MOVED 重定向又是怎么回事？

**答案：**

- Redis Cluster 会把 key 映射到 `16384` 个 hash slot 之一。
- 路由方式通常是：

```text
slot = CRC16(key) % 16384
```

- 每个 master 节点负责一部分槽位，所以客户端先算出 key 落在哪个槽，再找到负责这个槽的节点。
- 如果请求打到了错误节点，节点会返回 `MOVED`，告诉客户端这个槽现在该去哪个节点。
- 常见场景是：
  - 客户端路由表旧了
  - 集群扩容 / 缩容后槽位迁移了
- 所以 `MOVED` 不是“报错结束”，更像是：

> **你打错节点了，这个槽现在归别的节点管。**

**关键点：**

- 只有 `Cluster` 模式才有 slot 路由
- `Sentinel` 不分片，只做主从切换

### 36. Redis 为什么快？“单线程”到底指的是什么？

**答案：**

- Redis 快的原因通常答这几个：
  1. 纯内存操作
  2. 命令执行路径短，数据结构针对性强
  3. 单线程执行命令，避免锁竞争和上下文切换
  4. 网络层用 IO 多路复用（如 `epoll`）
- Redis 6.0 之后网络 IO 可以多线程，但**命令执行仍然是单线程**。
- 所以面试里别说成“Redis 完全单线程”，更稳的说法是：

> **Redis 单线程主要指命令执行单线程，不是整个 Redis 进程完全没有多线程。**

### 37. 什么是大 key？为什么大 key 危险？怎么处理？

**答案：**

- 所谓大 key，不是一个严格固定值，而是指：
  - 单个 key 占用内存特别大
  - 或者集合类 key 元素数特别多
- 常见风险：
  1. 删除慢，容易阻塞
  2. 网络传输大，响应抖动明显
  3. 主从复制压力大
  4. 集群迁移槽位时成本高
- 处理思路：
  - 先定位：`MEMORY USAGE`、采样分析
  - 再治理：拆 key、分片存、控制单 key 规模
  - 删除时优先考虑 `UNLINK`，避免像 `DEL` 那样同步阻塞

**推荐回答：**

> 大 key 指的是单个 key 对应的数据量特别大，比如超大字符串，或者元素特别多的 Hash、List、Set、ZSet。它的危险在于删除慢、网络传输大、复制成本高，还可能在集群迁移时拖慢整个节点。处理时一般先定位大 key，再做拆分和分片；如果要删除，优先考虑 `UNLINK` 这种异步回收方式，而不是直接 `DEL`。

### 38. 什么是热 key？为什么上了 Cluster 也还会有热 key？

**答案：**

- 热 key 指的是访问量特别集中到某一个 key 上。
- Redis Cluster 解决的是“多 key 分散到多节点”，但**一个 key 还是只能落到一个槽、一个分片**。
- 所以如果某个 key 极热，最终还是会把负责它的那一个节点打爆。
- 治理方式要先分场景：
  - 读热点：本地缓存、多级缓存、热点隔离
  - 写热点：拆 key / 分片计数 / 异步聚合
  - 强一致热点：限流、削峰、业务降级

### 39. Redis 做分布式锁怎么答更稳？RedLock 要不要提？

**答案：**

- 最基础的获取锁写法：

```bash
SET lock_key request_id NX EX 30
```

- `NX` 表示不存在才加锁，`EX` 表示过期时间，保证死锁时最终能自动释放。
- 释放锁不能直接 `DEL`，要先校验 value，避免删掉别人的锁，通常用 Lua 保证“校验 + 删除”原子性。
- 如果业务执行时间可能超过过期时间，要讲续期机制，比如 Redisson 的 watchdog。
- `RedLock` 可以提，但要知道它有争议。更稳的说法是：

> 大多数业务里，单 Redis 实例 / 主从高可用下的锁已经够用；RedLock 可以作为扩展知识提到，但工程上不一定默认采用。

### 40. Redis 的 fork + Copy On Write 是什么？为什么 RDB / AOF rewrite 会吃内存？

**答案：**

- Redis 做 `BGSAVE` 或 `AOF rewrite` 时，会 `fork` 一个子进程。
- `fork` 那一刻，父子进程**不会立刻复制整块内存**，而是共享物理页。
- 只有当父进程继续写某些页时，操作系统才会把这些页复制出来，这就是 `Copy On Write`。
- 所以如果这段时间写入很多，就会有大量页复制，内存占用明显升高，甚至接近翻倍。

**推荐回答：**

> Redis 在做 RDB 快照或 AOF rewrite 时，会 fork 子进程。fork 本身不是直接把内存全复制一遍，而是父子进程先共享页面，只有页被修改时才触发 Copy On Write 真正复制。所以如果快照期间写流量很大，会产生大量页复制，导致内存额外膨胀，这也是为什么 Redis 做持久化时要关注内存余量。

### 41. Redis Pub/Sub 和专业 MQ 有什么区别？什么时候不能拿 Pub/Sub 当消息队列？

**答案：**

- `Pub/Sub` 更像实时广播机制，不是强可靠消息队列。
- 它的特点是：
  - 订阅者在线时能立即收到
  - 订阅者离线时消息不会补发
  - 没有可靠 ACK、没有真正的消费位点管理
- 所以它更适合：
  - 缓存失效广播
  - 实时通知
  - 轻量事件广播
- 不适合：
  - 强可靠异步解耦
  - 离线补偿
  - 海量堆积和复杂消费组治理

**一句话总结：**

> Pub/Sub 更像“广播通知”，不是“可靠消息队列”；真要队列语义，优先看 Stream 或专业 MQ。
