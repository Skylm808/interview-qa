# 后端开发面经合集 - Kafka / gRPC / etcd（答案版）

> 答案表述参考 Kafka / gRPC / etcd 官方文档做了校准，尽量做到简洁、清晰、可解释

## Kafka

### 1. Kafka 的核心架构是什么？Topic、Partition、Broker 分别是什么？

**答案：**

- Kafka 的核心链路可以先这样理解：

```text
Producer -> Topic -> Partition -> Broker -> Consumer Group -> Consumer
```

- `Producer` 是生产者，负责把消息发送到 Kafka。
- `Topic` 是消息的逻辑主题，比如订单消息、支付消息、日志消息，可以把它理解成“消息类别”。
- `Partition` 是 Topic 的物理分片。一个 Topic 可以拆成多个 Partition，每个 Partition 本质上都是一条追加写日志。
- `Broker` 是 Kafka 集群里的节点服务器。Partition 会分布在不同 Broker 上存储，所以 Kafka 才能横向扩容。
- `Consumer` 是消费者，负责从 Kafka 拉取消息并处理。
- `Consumer Group` 是消费者组，用来做消费端的负载均衡。**同一个分区，在同一个消费组里，同一时刻只会被一个消费者消费。**
- `Offset` 这里最好分成两层理解：
  - `message offset`：消息在某个 Partition 里的位置编号，这是分区自己的；
  - `consumer group offset`：某个消费组在某个 Partition 上消费到哪里了，这是消费进度。

**这几个概念之间的关系，可以这样讲：**

- 一个 Topic 可以有多个 Partition，目的是提升吞吐和并行能力。
- 多个 Partition 会分布到多个 Broker 上，目的是做扩展和高可用。
- 生产者发消息时，最终是把消息写入某个 Topic 的某个 Partition。
- 消费者消费时，不是直接“消费 Topic 这个抽象概念”，而是消费 Topic 下的各个 Partition。
- Consumer Group 的意义是：把多个 Partition 分给组内多个消费者并行消费。

**为什么 Kafka 要这么设计？**

- 用 `Topic` 做业务隔离；
- 用 `Partition` 做并行和扩展；
- 用 `Broker` 做分布式存储；
- 用 `Consumer Group` 做消费端扩展和负载均衡；
- 用 `Offset` 做消费进度管理。

**消费者组还可以顺手这样理解：**

- 如果两个消费者属于 **同一个 Consumer Group**，那它们是“分工消费”，常用于提升消费能力。
- 如果两个消费者属于 **不同的 Consumer Group**，那它们是“各自独立消费一份消息”，常用于一个 Topic 被多个下游系统分别使用。
- 所以 Kafka 天然支持“一份消息被多个系统订阅”，也支持“一个系统内部多实例并行消费”。

**这里可以举一个很典型的业务例子：**

- 假设有一个 Topic 叫 `order_created`，里面放的是“订单创建成功”的消息。
- 这个 Topic 下有一个分区 `Partition-0`，其中一条消息是：

```text
offset 100 -> order_id=1001
```

- 这时有三个下游系统都想消费它：
  - 库存系统：`stock-group`
  - 积分系统：`points-group`
  - 数据分析系统：`analytics-group`
- 那么这三个 **不同的 Consumer Group**，都可以各自消费到 `offset 100` 这条消息。
- 也就是说，**同一个分区里的同一条消息，可以被多个不同消费者组分别消费**；它们互不冲突，因为每个组维护的是自己的消费进度。

**但如果是同一个 Consumer Group 内部，就不是这样了：**

- 假设 `Partition-0` 只属于 `order-group` 这个消费组。
- 如果 `order-group` 里有两个消费者 `consumer-1` 和 `consumer-2`，那这个分区同一时刻只会分给其中一个，比如只分给 `consumer-1`。
- `consumer-2` 不能同时再来消费这个分区，否则同组内就会重复消费，顺序也会乱。

**容易混淆的一点：Kafka 里其实有两种 offset：**

- 第一种是 **消息 offset**，它属于 Partition，用来标识消息在分区日志里的位置，比如 `0、1、2、3...`。
- 第二种是 **消费进度 offset**，它属于 Consumer Group，用来标识“这个组在这个分区上读到哪了”。
- 所以更准确地说，不是“offset 在消费者手里，而不在分区上”，而是：
  - 分区自己有消息 offset，负责定义顺序；
  - 消费组有消费位点，负责记录进度。
- 消费组提交的消费位点通常会存到 Kafka 的内部主题 `__consumer_offsets` 里。

**面试里比较完整但不啰嗦的回答可以这样说：**

> Kafka 本质上是一个分布式消息队列，也是一个分布式日志系统。生产者把消息发到 Topic，Topic 再拆成多个 Partition，每个 Partition 是一条有序追加日志，这些 Partition 分布在不同 Broker 上存储。消费者不是直接消费 Topic，而是按 Consumer Group 去消费各个 Partition，同一个分区在同一个消费组里只会分给一个消费者处理。这样 Kafka 就同时具备了业务隔离、水平扩展、高吞吐和消费端并行能力。

### 2. Kafka 为什么吞吐量高？

**答案：**

- 第一，顺序写磁盘。Kafka 本质上是追加写日志，比随机写友好很多。
- 第二，充分利用操作系统页缓存，不是每次都直接落到物理磁盘。
- 第三，批量发送、批量拉取，减少了网络和系统调用开销。
- 第四，零拷贝等优化让数据传输链路更短。
- 所以 Kafka 的高吞吐不是靠某一个点，而是 **顺序 IO + 批处理 + 页缓存 + 网络优化** 一起堆出来的。

### 3. Kafka 里 Partition、Replica、Leader、Follower、ISR 分别是什么？

**答案：**

- `Partition` 是分片，解决扩展性问题。
- `Replica` 是副本，解决高可用问题。
- 一个分区在同一时刻只有一个 `Leader` 对外提供读写，其他副本是 `Follower`，负责跟随同步。
- `ISR` 全称是 `In-Sync Replicas`，表示“当前和 Leader 保持同步、状态足够健康”的副本集合。
- 这里要注意，**ISR 不是所有副本的全集**。如果某个 Follower 同步太慢、延迟太大，它可能暂时掉出 ISR。
- 所以当我们说 `acks=all` 时，等的不是“所有副本都确认”，而是“当前 ISR 里的副本确认”。
- 面试里常见追问是：**真正决定可用性的不是副本数本身，而是 ISR 里还有没有足够健康的副本。**

### 4. 生产者发送消息时，Kafka 怎么决定消息进哪个分区？

**答案：**

- 如果消息显式指定了 partition，那就直接进指定分区。
- 如果没指定 partition，但指定了 key，通常会根据 key 做哈希，让同一个 key 尽量落到同一个分区。
- 如果 key 也没有，通常会走轮询或 sticky 分区策略，尽量把批量发送做得更高效。
- 所以常见面试回答就是：**指定分区优先；否则 key 决定路由；再否则走默认分区器策略。**

**如果继续追问：哈希是不是一定不会冲突？为什么这里不用 MD5 / SHA-256？**

- 先说结论：**哈希一定可能冲突。**
- 只要输入空间远大于输出空间，就不可能做到绝对无冲突。区别只在于：
  - 冲突概率高不高
  - 工程上能不能接受
- 像 `MD5`、`SHA-256` 这种更多是**密码学哈希**，强调抗碰撞、抗篡改和安全性。
- Kafka 按 key 选分区，本质只是做**路由分发**，目标是：
  - 同一个 key 尽量稳定落到同一个分区
  - 算得快
  - 分布尽量均匀
- 所以这里更关注的是**速度和分布性**，不是密码学安全，没必要上 `MD5` / `SHA-256` 这种更重的哈希。
- 另外要注意，即使不同 key 哈希后落到同一个分区，这也不叫“数据错了”，只是它们被路由到了同一个桶里。Kafka 只要求“同 key 稳定路由”，不要求“不同 key 绝对分开”。

**面试里可以这样答：**

> Kafka 按 key 选分区时，本质是做路由，不是做安全校验，所以不需要 MD5 这类密码学哈希。哈希本身一定可能冲突，但这里工程上能接受，因为目标只是让同一个 key 稳定落到同一个分区，并尽量把数据分散得比较均匀。只要分区器足够快、分布性足够好，就能满足需求。

### 5. Kafka 怎么保证消息不丢？

**答案：**

- 这题不能只答 Broker，最好按 **生产者、Broker、消费者** 三段来讲。
- 核心思路是：**发的时候别假成功，存的时候别单点，消费的时候别没处理完就提交 offset。**

**1）生产者侧：保证“真的发成功了”**

- 关键配置是 `acks=all`。这里的 Leader 指的是**目标 Partition 的 Leader 副本**。Producer 会先把消息发给这个 Partition 的 Leader。
- 如果配置了 `acks=all`，Leader 不能只在自己写成功后就立刻返回，而是要等 **当前 ISR 里的副本** 也同步完成后，再给 Producer 返回成功。
- 这里一定要注意：`acks=all` 等的是 **ISR**，不是“所有副本的绝对全集”。如果某个 Follower 太慢、已经掉出 ISR，它不会参与这次确认。
- 只说 `acks=all` 还不够，通常还要配合 `retries`，应对网络抖动这类临时失败。
- 但一重试就可能重复发，所以还要开启幂等性 `enable.idempotence=true`，避免因为重试造成重复写入。

**2）Broker 侧：保证“存下来后不轻易丢”**

- Kafka 靠多副本机制保证高可用。一个 Partition 不只一份数据，Leader 挂了还能从 Follower 里选新的 Leader。
- 这里有个很关键的点是 `min.insync.replicas`。它的意思不是“总副本数至少几个”，而是：**当生产者用 `acks=all` 写入时，ISR 里最少要有几个副本可用，这次写入才允许成功。**
- 如果只配了 `acks=all`，但 ISR 里只剩 Leader 一个节点，那 `acks=all` 实际上就会退化得很危险，因为它等来等去也只等到了 Leader 自己。
- 所以高分回答通常会补一句：**`acks=all` 要和 `min.insync.replicas >= 2` 一起看**。这样当 ISR 副本数不够时，Broker 会直接拒绝写入，宁可报错，也不冒险接收一条可能马上丢失的数据。

**举个最常见的例子：**

- 假设一个 Partition 有 3 个副本：`1 个 Leader + 2 个 Follower`
- 此时配置：
  - `acks=all`
  - `replication.factor=3`
  - `min.insync.replicas=2`
- 如果当前 ISR 里有 3 个副本，写入当然可以成功。
- 如果有一个 Follower 掉线了，但 ISR 里还剩 `Leader + 1 个 Follower`，也就是 2 个副本，那仍然可以写。
- 但如果再坏一点，只剩下 Leader 自己还在 ISR 里，那这时写入就应该被拒绝，而不是假装成功。

**所以这三个概念要分清：**

- `replication.factor`：这个 Partition 一共配了几份副本
- `ISR`：这些副本里，当前真正健康、同步得上的那几份
- `min.insync.replicas`：当你用 `acks=all` 写入时，ISR 至少得剩几份，这次写入才允许成功

**3）消费者侧：保证“处理成功后再提交进度”**

- 如果消费者一拉到消息就自动提交 offset，但本地业务还没执行完就崩了，Kafka 会以为这条消息已经消费过了，这才是真正可怕的“业务丢消息”。
- 所以核心做法是关闭自动提交 `enable.auto.commit=false`，改成手动提交。
- 只有当本地业务逻辑真的执行成功，比如数据库事务提交成功了，再去提交 offset。
- 这里最好再补一句：为了应对重复消费，消费逻辑本身通常还要做幂等。

**所以更完整的总结是：**

- 生产者侧：`acks=all + retries + idempotence`
- Broker 侧：`replication + min.insync.replicas`
- 消费者侧：`手动提交 offset + 消费幂等`

**面试里可以直接这样答：**

> Kafka 保证消息不丢，不能只看 Broker，而要做端到端保障。生产者侧我会配置 `acks=all`，配合重试和幂等，保证消息真的写成功；Broker 侧依靠多副本机制，并结合 `min.insync.replicas`，避免副本不够时还错误地接收写入；消费者侧关闭自动提交，只有本地业务真正处理成功后再手动提交 offset。再往前走一步，消费逻辑本身还要做幂等。严格来说，工程上更准确的说法不是“绝对 0 丢失”，而是尽量做到端到端可靠和可恢复。

### 6. Kafka 怎么保证消息有序？

**答案：**

- Kafka 只能保证 **单分区内有序**，不能天然保证一个 Topic 所有分区的全局有序。
- 如果业务要求同一类消息严格有序，常见做法是让同一个业务 key 始终进入同一个分区。
- 消费端也要注意：一个分区最好只被一个消费者线程按顺序处理，否则应用层自己也可能打乱顺序。
- 所以面试时最好直接说：**Kafka 能保证的是单分区顺序，不是全局顺序。**

### 7. Consumer Group 和 Rebalance 是什么？为什么大家都怕 Rebalance？

**答案：**

- `Consumer Group` 是 Kafka 的消费组机制：一个分区同一时刻只会被组内一个消费者消费。
- 同组内是“分工关系”，不同组之间是“各自独立消费”。所以一个 Topic 的同一条消息，既可以被一个系统内部多实例并行处理，也可以被多个下游系统分别订阅。
- 举个例子：`order_created` 这个 Topic 里的同一条消息，可以同时被 `stock-group`、`points-group`、`analytics-group` 三个组分别消费；但在 `stock-group` 内部，这条分区消息同一时刻只能分给一个消费者实例处理。
- `Rebalance` 就是消费组成员变化、订阅变化、分区变化时，Kafka 重新分配分区归属。
- 大家怕 Rebalance，是因为它发生时通常会带来短暂停顿，严重时会影响消费延迟，甚至引发重复消费。
- 所以工程上一般会尽量减少不必要的频繁重平衡，比如控制消费者抖动、合理设置心跳和超时参数。

### 8. Kafka 出现消息堆积时，怎么排查和处理？

**答案：**

- 我会先判断是“生产太快了”还是“消费变慢了”。
- 如果是消费变慢，还要继续拆：是消费者挂了、下游接口慢了、单条消息太重了，还是分区数不够。
- 临时止血常见做法有：扩消费者、扩分区、批量消费、隔离慢下游、必要时降级非核心逻辑。
- 根因治理则是：优化单条消息处理耗时、提高幂等和重试能力、避免坏消息把整个消费链路拖死。

### 9. MQ 比 Kafka 好在哪？什么时候 MQ 比 Kafka 更合适？

**答案：**

- 这题不要答成“谁绝对更强”，而要答成：**Kafka 更偏高吞吐日志流和大数据管道；传统 MQ（常见如 RabbitMQ、RocketMQ）更偏业务消息投递。**

**MQ 相比 Kafka 更适合的点：**

1. **消息模型更贴近业务**
   
   - 传统 MQ 通常是“生产者 -> 队列 -> 消费者”模型，语义直观，业务同学更容易理解。
   - 对于下单、发券、发短信、发邮件、异步通知这类业务消息，队列模型往往更顺手。

2. **路由能力通常更灵活**
   
   - 比如 RabbitMQ 有 Exchange，可以做直连、广播、主题路由。
   - 当业务需要按规则把消息投递给不同下游时，传统 MQ 往往比 Kafka 的 Topic/Partition 模型更灵活。

3. **延迟队列、死信队列、优先级等特性更偏业务场景**
   
   - 例如订单超时取消、失败消息进死信队列、部分消息优先处理，这些是传统 MQ 的强项。
   - Kafka 也能做一部分，但通常需要额外设计 Topic 或应用层补逻辑，没有传统 MQ 那么开箱即用。

4. **细粒度确认和消费控制更自然**
   
   - 传统 MQ 往往更强调单条消息确认、重回队列、消费失败重试等能力。
   - 对“单条消息必须明确处理成功/失败”的业务链路，传统 MQ 往往更符合直觉。

5. **对低吞吐但强业务语义的系统更省心**
   
   - 如果系统量级不大，但要求消息投递语义清晰、路由复杂、业务控制丰富，传统 MQ 往往更省开发和运维心智。

**Kafka 更擅长的场景：**

- 高吞吐日志采集
- 埋点、行为流、监控流
- 大数据管道、流式计算
- 一个消息要被多个下游系统长期重复消费

**面试里推荐这样答：**

> 不能简单说 MQ 一定比 Kafka 好，更多是场景差异。Kafka 更偏分布式日志和高吞吐数据流，适合日志采集、埋点、大数据链路；而传统 MQ，像 RabbitMQ 或 RocketMQ，更偏业务消息系统。它们在灵活路由、延迟消息、死信队列、单条确认、失败重试这些业务特性上通常更强，也更贴近日常业务开发。所以如果是订单超时关闭、短信通知、异步任务编排这类场景，我会优先考虑传统 MQ；如果是海量日志、行为流、实时数据管道，我会优先考虑 Kafka。

## gRPC

### 10. gRPC 是什么？它和普通 REST API 的核心区别是什么？

**答案：**

- gRPC 是一种现代 RPC 框架，常见实现是基于 `HTTP/2 + Protocol Buffers`。
- REST 更偏资源风格，很多时候对外暴露 HTTP/JSON 接口；gRPC 更偏服务调用，接口定义通常更强、更明确。
- gRPC 在服务间调用里很常见，因为它序列化更紧凑、协议能力更强。
- 所以一句话说：**REST 更适合开放接口，gRPC 更适合内部高性能服务通信。**

---

### 11. gRPC 为什么通常比 HTTP + JSON 更适合服务间通信？

**答案：**

- 第一，Protobuf 比 JSON 更紧凑，序列化和反序列化开销通常更小。
- 第二，gRPC 基于 HTTP/2，天然支持多路复用，连接利用率更高。
- 第三，接口通过 `.proto` 明确定义，跨语言协作和代码生成都更规范。
- 但这不是说 gRPC 一定全面碾压 HTTP/JSON，而是说它在 **内部服务调用** 这类场景里通常更合适。

---

### 12. Protobuf 为什么适合拿来做 RPC 接口定义？

**答案：**

- 它既是序列化格式，也是接口描述语言的一部分。
- 你可以先在 `.proto` 里定义消息结构和服务方法，再自动生成多语言代码。
- 它的二进制编码更紧凑，字段演进也更规范，比较适合长期维护的接口。
- 所以 Protobuf 的价值不只是“比 JSON 小”，更重要的是 **强约束、可生成、跨语言一致**。

---

### 13. gRPC 的四种调用模式分别是什么？

**答案：**

- `Unary RPC`：最常见，一次请求对应一次响应。
- `Server Streaming RPC`：客户端发一次请求，服务端返回一个流。
- `Client Streaming RPC`：客户端发一个流，服务端最终回一个结果。
- `Bidirectional Streaming RPC`：双向都是流，双方可以持续收发。
- 面试里如果想说得更稳，可以补一句：**流式 RPC 很适合实时推送、增量返回、长连接交互。**

---

### 14. gRPC 为什么依赖 HTTP/2？HTTP/2 给了它哪些能力？

**答案：**

- HTTP/2 支持多路复用，所以多个请求可以共用一条连接，减少队头阻塞问题。
- 它还支持头部压缩和双向流，这些能力都很适合 RPC 场景。
- gRPC 的流式调用，本质上也是建立在 HTTP/2 的流模型之上。
- 所以可以简单理解成：**HTTP/2 提供了连接层和传输层能力，gRPC 在上面补了接口定义、序列化和调用语义。**

---

### 15. gRPC 里超时、取消、重试这些能力一般怎么理解？

**答案：**

- 超时本质上是在告诉下游：这次调用最多等多久，避免请求无限挂住。
- 取消一般用于调用方已经不需要结果了，或者上游请求已经结束，希望把取消信号继续往下传。
- 重试不能乱开，要先看接口是否幂等，否则可能把一次请求执行成多次。
- 所以工程上常见原则是：**超时一定要有，取消要能传递，重试要建立在幂等前提下。**

---

## etcd

### 16. etcd 是什么？为什么很多系统，尤其是 Kubernetes，会用它？

**答案：**

- etcd 是一个分布式、强一致的键值存储系统。
- 它很适合保存配置、服务注册信息、选举状态这类“数据量不一定大，但一致性要求高”的元数据。
- Kubernetes 用它，是因为集群状态必须有一个可靠、强一致的存储底座。
- 所以一句话概括：**etcd 不适合拿大量业务数据来硬扛，但特别适合做分布式系统的状态中枢。**

---

### 17. etcd 为什么常被说成是 CP 系统？这里的 CP 和 CAP 又是什么意思？

**答案：**

先把缩写讲清楚：

- `CAP` 指的是分布式系统里常说的三个维度：
  - `C` = `Consistency`，**一致性**
  - `A` = `Availability`，**可用性**
  - `P` = `Partition Tolerance`，**分区容错性**

#### CAP 分别是什么意思？

##### 1）Consistency（一致性）

一致性的意思是：

> **只要写成功了，后续所有节点对外读到的应该是同一个最新结果。**

比如：

- 你把 key 从 `x=1` 改成 `x=2`
- 如果系统说“写成功”
- 那后面无论你读哪个健康节点，都应该看到 `x=2`

##### 2）Availability（可用性）

可用性的意思是：

> **每个请求都能得到响应。**

注意这里强调的是“有响应”，不一定保证这个响应一定是最新值。  
所以某些系统为了保证 A，可能会返回旧数据，也不愿意直接拒绝请求。

##### 3）Partition Tolerance（分区容错性）

分区容错性的意思是：

> **当集群节点之间网络断开、消息丢失、出现脑裂式分区时，系统仍然要继续运行。**

现实分布式系统里，网络分区是必须面对的，所以 `P` 基本上是没法放弃的。

---

#### 那为什么 etcd 常被说成 CP？

因为 etcd 在网络分区这种异常场景下，会更优先保证：

- `C`：一致性
- `P`：分区容错性

而不会为了所有节点都继续可写，去牺牲一致性。

也就是说：

- etcd 底层基于 Raft
- 写请求必须经过 Leader
- 并且要得到**多数派确认**才算成功

这意味着：

- **多数派分区**：还能继续选主、继续写
- **少数派分区**：虽然节点可能还活着，但因为拿不到多数派确认，就不能继续随便写

这时候系统牺牲掉的，就是一部分 `A`（Availability，可用性）。

所以 etcd 被叫做 CP，本质上是在说：

> **它宁可在少数派上拒绝写请求，也不愿意让不同分区各自写出不一致的数据。**

---

#### 为什么分布式系统里总在说 CAP？

因为当发生网络分区时，你通常没法同时把三件事都做到极致：

- 既保证绝对一致
- 又保证所有节点始终可写
- 还要求系统在分区下继续运行

所以现实里更常见的是：

- 要么更偏 `CP`
- 要么更偏 `AP`

#### etcd 为什么不是 AP？

因为 etcd 的定位是：

- 服务注册发现
- 配置中心
- 分布式协调
- 元数据存储

这些场景里，**一致性通常比“所有节点随时都能写”更重要**。  
如果配置中心出现不一致、锁状态出现分叉、元数据左右不统一，后果往往比“少数派暂时不可写”更严重。

---

#### 面试里推荐这样答

> CAP 里的 C 是 Consistency，一致性；A 是 Availability，可用性；P 是 Partition Tolerance，分区容错性。etcd 常被说成 CP，是因为它在网络分区场景下更优先保证一致性和分区容错性。由于 etcd 底层基于 Raft，写请求必须经过 Leader 并得到多数派确认，所以少数派分区即使还活着，也不能继续随便写。它宁可牺牲一部分可用性，也要先守住一致性。

---

### 17A. etcd 的底层原理为什么总会讲到 Raft？Raft 选举和日志复制到底是怎么工作的？

**答案：**

先给结论：

> etcd 之所以能做到强一致，核心就在于它底层用的是 **Raft 一致性协议**。Raft 的目标，就是让多个节点对“同一份日志”达成一致，而不是让每个节点各自随便接收写入。

你可以把 Raft 理解成三件大事：

1. **先选一个 Leader**（统一写入口）
2. **再复制日志**（让多数节点看到同样的变更）
3. **最后提交并应用**（让所有节点按同样顺序执行）

---

#### 一、Raft 为什么一定要先选主？

如果一个 3 节点集群里，所有节点都能同时独立接收写请求，就很容易出现：

- A 节点先写了一条值
- B 节点也写了一条不同的值
- 两边互相都觉得自己是对的

这会导致数据冲突，系统很难收敛到一个统一结果。  
所以 Raft 的第一原则就是：

> **同一时刻只允许一个 Leader 对外统一处理写请求。**

这样所有写入先进入同一个入口，日志顺序才有可能统一。

---

#### 二、Raft 里有哪几种角色？

最常见的三个角色：

- `Follower`：默认状态，平时主要接收 Leader 的心跳和日志复制
- `Candidate`：怀疑 Leader 挂了之后，临时出来参加竞选的节点
- `Leader`：当前集群唯一对外处理写请求的节点

你可以把它粗略理解成：

- Follower = 普通成员
- Candidate = 竞选人
- Leader = 当前领导者

---

#### 三、Raft 选主（Leader Election）到底怎么发生？

这是面试官最喜欢追问的地方。

##### 1）初始状态

集群刚启动时，大家通常先都是 `Follower`。

每个 Follower 都会维护一个**选举超时时间（election timeout）**。  
如果在这个时间内，它一直能收到 Leader 的心跳，就继续当 Follower。

##### 2）Leader 挂了会怎样？

如果某个 Follower 在一段时间内一直没收到 Leader 心跳，它会怀疑：

> Leader 可能挂了，或者网络出问题了。

于是它会：

1. 把自己状态切成 `Candidate`
2. 把自己的 `term`（任期）加 1
3. 先投自己一票
4. 给其他节点发 `RequestVote`（拉票请求）

##### 3）其他节点什么时候会投票给它？

其他节点不会无脑投票，通常要满足两个条件：

1. **当前 term 不能比自己旧**
2. **候选人的日志不能比自己落后太多**

第二条特别关键。  
Raft 不是“谁先喊谁当 Leader”，而是尽量保证：

> **最后当选 Leader 的节点，日志不能太旧。**

这样才能避免“一个很落后的节点当了 Leader，把新数据覆盖掉”。

##### 4）什么时候当选成功？

当 Candidate 拿到**多数派（majority）**投票后，就会变成 Leader。

比如 3 节点集群：

- 至少要 2 票

5 节点集群：

- 至少要 3 票

所以多数派的意义就是：

> **任何两个多数派集合一定有交集。**

这能帮助系统保证不会同时出现两个“合法 Leader”长期并存。

##### 5）为什么选举超时要随机？

如果大家超时时间完全一样，就可能同时变成 Candidate，同时互相抢票，导致反复平票。

所以 Raft 会把选举超时做成一个随机区间，比如：

```text
150ms ~ 300ms
```

这样更容易让某个节点先超时、先发起选举、先拿到票。

---

#### 四、Term（任期）到底是什么？

`term` 可以理解成：

> **第几届领导周期**

每次发起新一轮选举，term 就会增加。  
节点会用 term 来判断消息新旧：

- term 更大的消息，通常更新
- term 更小的消息，通常过期

所以当一个旧 Leader 恢复后，如果它发现别人 term 比自己大，它就不能继续自称 Leader，而是要退回 Follower。

这也是 Raft 避免“脑裂后旧 Leader 继续乱写”的关键机制之一。

---

#### 五、Leader 选出来以后，写请求怎么处理？

Leader 当选后，客户端写请求一般都发给 Leader。  
Leader 收到一个写请求，不会马上说“成功”，而是会走这几步：

1. **先把这次变更追加成一条日志**
2. **把日志复制给 Followers**
3. **等多数派确认收到**
4. **这条日志才算 committed（已提交）**
5. **再把它应用到状态机**
6. **最后返回客户端成功**

所以 etcd 的强一致，本质上依赖的是：

> **写请求必须先经过 Leader，并且要过多数派确认，才算真正生效。**

---

#### 六、状态机应用（State Machine Apply）是什么意思？

Raft 不只是把日志“存下来”，还要求各节点最终按相同顺序去执行这些日志。

比如：

```text
log1: put a=1
log2: put b=2
log3: delete a
```

只要所有节点都按：

```text
log1 -> log2 -> log3
```

这个顺序去 apply（应用）到自己的状态机里，那么最终数据状态就会一致。

所以 Raft 一致性的核心不是“大家大概差不多”，而是：

- **日志顺序一致**
- **提交点一致**
- **应用顺序一致**

---

#### 七、心跳（Heartbeat）为什么重要？

Leader 平时会定期发心跳给 Followers。  
心跳的作用不只是“报平安”，还有两个意义：

1. 告诉大家：
   
   - 我这个 Leader 还活着
   - 你们别急着重新选举

2. 顺便推动日志复制进度
   
   - 有时心跳和日志复制可以复用同一类 `AppendEntries` 消息

所以你可以这样理解：

> **心跳是维持 Leader 地位和维持集群稳定节奏的基础机制。**

---

#### 八、为什么 Raft 能保证一致性？

因为它抓住了几个非常关键的约束：

1. **同一时刻只允许一个 Leader 统一处理写请求**
2. **日志必须复制到多数派，才能提交**
3. **新 Leader 不能比多数派手里的已提交日志更旧**
4. **所有节点最终都按同样顺序应用日志**

所以更适合面试的说法是：

> Raft 不是让每个节点都自己决定结果，而是先统一领导者，再统一日志顺序，再统一提交规则。

---

#### 九、面试官最爱追问的几个点

##### 1）为什么一定是“多数派”？

因为多数派能保证交集。  
只要一条日志已经被多数派确认，下一任 Leader 想合法当选，就不太可能完全看不到这条日志。

##### 2）为什么 etcd 少数派不能继续写？

因为少数派拿不到多数派确认。  
如果还允许它继续写，就可能和多数派分叉，破坏强一致。

##### 3）为什么 etcd 偏 CP？

因为它在网络分区时宁可让少数派不可写，也不愿意让两边都继续接受写请求。

---

#### 十、面试里推荐回答

> etcd 底层用的是 Raft 一致性协议。Raft 的核心思路是：先选出一个 Leader，统一处理写请求；如果 Follower 长时间收不到心跳，就会把 term 加一，变成 Candidate 发起投票，拿到多数派后成为新 Leader。Leader 收到写请求后，会先把变更写成日志，再复制给 Followers；只有多数派确认后，这条日志才算提交成功，随后各节点再按相同顺序把日志应用到状态机里。Raft 之所以能保证一致性，本质上靠的是 Leader 统一入口、多数派提交、term 约束和日志顺序一致。

**一句话总结：**

> etcd 强一致的根本原因，不是它“存的是 key-value”，而是它底层的写入和选主过程都受 Raft 的 Leader、term、日志复制和多数派提交规则约束。

---

### 18. etcd 的读写流程大致是怎样的？

**答案：**

- 写请求一般由 Leader 处理，Leader 会把日志复制给多数派节点，提交成功后才真正生效。
- 读请求如果要强一致，通常也需要通过 Leader 或经过一致性校验的方式返回。
- 如果只是本地读，延迟会更低，但一致性语义要看具体模式。
- 面试里不需要把 Raft 细节背得太死，抓住一句话就够：**写入要多数派确认，强一致读要保证读到的是最新提交结果。**

---

### 19. etcd 的 Watch 机制是什么？适合什么场景？

**答案：**

- Watch 就是对某个 key 或某个前缀持续订阅变更事件。
- 当数据发生变化时，客户端可以持续收到通知，而不是自己不停轮询。
- 它很适合做配置变更推送、服务上下线感知、任务状态监听这类场景。
- 所以 Watch 的核心价值是：**把“主动轮询”变成“事件驱动”。**

---

### 20. etcd 的 Lease 是什么？为什么服务注册常和 Lease 一起用？

**答案：**

- Lease 可以理解成“带租期的生命令牌”。
- 一个 key 绑定了 Lease 后，只要客户端持续续租，这个 key 就一直有效；如果客户端挂了、续租停了，key 到期后会自动失效。
- 这特别适合做服务注册，因为服务实例宕机后，注册信息可以自动清理，不需要人工扫垃圾数据。
- 所以常见说法是：**Lease 解决的是“临时状态如何自动过期”的问题。**

---

### 21. etcd、ZooKeeper、Redis 都能做注册中心或分布式协调，它们怎么选？

**答案：**

- 如果你最看重强一致、监听机制、和云原生生态配合，etcd 很常见。
- ZooKeeper 也是经典方案，生态成熟，但在很多新项目里，etcd 更常被优先考虑。
- Redis 做注册和锁的优势是简单、性能高，但它更偏缓存和高性能数据访问，不是天然为强一致协调而设计的。
- 所以面试里别答成“谁绝对最好”，更稳的说法是：**看你更看重强一致、生态、性能，还是实现复杂度。**

---

### 22. gRPC 服务注册与发现一般怎么做？

**答案：**

- gRPC 本身不强制你一定用哪种注册中心，但在工程上通常会配合 `etcd / Consul / Nacos` 这类组件做服务注册发现。
- 一个完整流程一般是：
  - 服务启动时把自己的 `IP + Port + Metadata` 注册到注册中心
  - 同时上报心跳或续租
  - 客户端调用前从注册中心获取可用实例列表
  - 再按轮询、随机、权重、一致性哈希等策略选一个实例发起 RPC
- 如果服务挂了，注册中心会通过超时、Lease 失效或主动注销把它摘掉，客户端后续更新列表后就不会再打到坏节点。
- 面试里推荐这样答：**注册发现解决的是“调用谁”，但不保证“调用一定成功”，线上还要配合超时、重试、熔断和幂等一起设计。**

---

### 23. 业务里为什么要用 MQ？什么时候会想到 Kafka？

**答案：**

- 我不会答成“为了用中间件而用”，而是会先看业务是不是有这几个问题：
  
  - 主链路太重，需要异步化
  - 上下游耦合太紧，需要解耦
  - 突发流量太大，需要削峰填谷
  - 任务不能在进程内直接丢，需要持久化

- 如果这些问题都比较明显，就会想到 MQ。

**Kafka 更适合什么时候：**

- 吞吐要求高
- 消息量大
- 更偏日志流、埋点流、异步处理流
- 消费端可能有多个下游系统各自消费

**为什么不是直接开 goroutine：**

- goroutine 适合轻量异步
- 但进程一挂，内存里的任务就没了
- Kafka 能把任务持久化下来，服务重启后还能继续消费

**面试里推荐这样答：**

> 业务里用 MQ，核心通常是为了解耦、异步化、削峰和持久化。Kafka 更适合高吞吐、分布式日志流和多个消费组并行消费的场景；如果只是单机内的轻量异步，goroutine 也许够用，但一旦任务比较重、不能接受丢任务，消息队列就更合适。  

---

### 24. 生产者发送 Kafka 失败了怎么办？

**答案：**

- 先不要只答“重试”，更完整的思路是分层看：
1. **先保证发送可靠性**
   
   - `acks=all`
   - 开启重试
   - 开启幂等 `enable.idempotence=true`

2. **失败时做分类**
   
   - 网络抖动：可重试
   - Broker 暂时不可用：可重试
   - 参数错误 / 消息非法：通常不是重试能解决的

3. **重试也要有边界**
   
   - 不能无限重试
   - 避免把下游压垮

4. **业务上还要兜底**
   
   - 本地消息表
   - 补偿任务
   - 告警和死信处理

**面试里推荐这样答：**

> 生产者发送 Kafka 失败时，我会先用 `acks=all + retries + idempotence` 提高发送可靠性。失败后要先区分是临时性故障还是参数类错误，临时性故障可以有限重试；如果消息对业务特别关键，还要用本地消息表或补偿任务兜底，不能把“发送失败”完全丢给应用日志。
