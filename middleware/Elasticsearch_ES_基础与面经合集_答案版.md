## 一、先建立最小认知地图

### 1. ES 到底是什么？

你可以先这样理解：

> **Elasticsearch 是一个分布式搜索和分析引擎，本质上擅长“海量数据的全文检索、过滤、聚合和近实时查询”。**

它通常适合这些场景：

- 搜索框检索（商品、文章、文档）
- 日志检索（ELK/Elastic Stack）
- 多条件过滤 + 聚合分析
- 地理位置搜索
- RAG 检索层的一部分（尤其是关键词检索 / 混合检索）

如果你只记一句：

```text
MySQL 更擅长事务型存储和精确查询
ES 更擅长搜索、全文检索、过滤、聚合
```

---

### 2. ES 为什么搜索快？

最核心的原因是：

> **它不是拿一篇篇文档从头扫，而是先把“词 -> 出现在哪些文档里”建好索引。**

这就是它最著名的底层结构：

- **倒排索引（Inverted Index）**

---

### 3. 什么是倒排索引？

先对比一下：

#### 正排思路

```text
文档1 -> 包含哪些词
文档2 -> 包含哪些词
文档3 -> 包含哪些词
```

#### 倒排思路

```text
词A -> 出现在哪些文档
词B -> 出现在哪些文档
词C -> 出现在哪些文档
```

比如：

```text
Doc1: "redis cache"
Doc2: "mysql redis"
Doc3: "es search"
```

倒排索引会更像：

```text
redis -> [Doc1, Doc2]
cache -> [Doc1]
mysql -> [Doc2]
search -> [Doc3]
```

所以当用户搜 `redis` 时，不用扫所有文档，直接拿 `redis -> [Doc1, Doc2]` 就行。

一句话：

> **倒排索引解决的是“根据词快速找到文档”。**

---

### 4. ES 的几个核心概念

| 概念           | 一句话理解                     |
| ------------ | ------------------------- |
| **Index**    | 索引，可以理解成一类数据的逻辑集合         |
| **Document** | 文档，ES 中最小的数据单元，通常是一条 JSON |
| **Field**    | 文档里的字段                    |
| **Node**     | 一个 ES 节点 / 实例             |
| **Cluster**  | 多个节点组成的集群                 |
| **Shard**    | 分片，索引被拆开的物理数据块            |
| **Replica**  | 副本分片，用来高可用和提高读吞吐          |
| **Mapping**  | 映射，定义字段类型和索引方式            |
| **Analyzer** | 分词器，把文本拆成词项               |

一句话理解分片：

> **一个 Index 会被拆成多个 Shard，Shard 再分布到不同 Node 上，这样 ES 才能横向扩展。**

---

### 5. text 和 keyword 的区别是什么？

这是 ES 面试里最经典的问题之一。

#### `text`

- 用于全文检索
- 会经过 analyzer 分词
- 适合文章正文、标题、描述

#### `keyword`

- 不分词
- 适合精确匹配、过滤、排序、聚合
- 适合手机号、状态、标签、订单号、用户名

你可以记成：

> **text 用来搜内容，keyword 用来做精确值匹配。**

比如：

- 商品标题：`text`
- 商品类目：`keyword`
- 用户 ID：`keyword`

---

### 6. DocValues 是什么？

这是一个经常被问但很多人讲不清的点。

DocValues 是：

> **ES 在索引时额外构建的一种列式存储结构，用来支持排序、聚合、脚本访问。**

它和倒排索引关注点不一样：

- 倒排索引：**词 -> 文档**，方便搜索
- DocValues：**文档 -> 字段值**，方便排序 / 聚合

所以你可以这样答：

> 倒排索引擅长检索，DocValues 擅长排序和聚合。

---

### 7. ES 为什么说“近实时”（Near Real-Time）？

因为写入文档后，不一定立刻就能被搜索到。  
中间会涉及一个：

- **refresh**

ES 默认会周期性 refresh（常见理解大约 1 秒级），refresh 之后新数据才对搜索可见。  
所以它叫“近实时”，不是严格实时。

你可以记成：

> **写成功不等于立刻可搜，search 可见性通常取决于 refresh。**

## 二、ES 的写入、查询和删除到底怎么理解

### 1. 写入流程大概是什么？

非常简化地理解：

1. 客户端发写请求
2. 路由到某个主分片（Primary Shard）
3. 主分片先处理并写入
4. 再同步到副本分片（Replica Shard）
5. 返回结果

底层常会提到：

- segment
- translog（transaction log，事务日志）
- refresh
- flush

粗略记忆：

- **translog**：保证崩溃恢复
- **refresh**：让新数据变得可搜索
- **flush**：更重的持久化动作

---

### 2. 搜索流程大概是什么？

常见会分成两段：

#### Query Phase

- 每个相关分片先各自查
- 找出本分片里最匹配的 TopN

#### Fetch Phase

- 协调节点汇总各分片结果
- 再去拿真正需要返回的文档内容

一句话：

> **先各分片算“谁最相关”，再统一取回真正文档。**

---

### 3. 删除和更新为什么不是真“原地改”？

ES 底层基于 Lucene，很多时候：

- 删除更像“打删除标记”
- 更新常常是“删旧文档 + 写新文档”

所以更新频繁时，写放大和 segment merge 成本会更明显。

## 三、面试高频题（基础版）

### 1. Elasticsearch 是什么？和 MySQL 有什么区别？

**答案：**

Elasticsearch 是一个分布式搜索和分析引擎，底层依赖倒排索引，特别适合全文检索、过滤和聚合。MySQL 更偏事务型数据存储和精确查询，适合订单、账户、关系型业务。一般不会拿 ES 替代 MySQL 做事务主库，而是让 ES 承担搜索和分析能力。

---

### 2. 什么是倒排索引？BM25 又是怎么和倒排索引配合工作的？

**答案：**

先记一句最重要的话：

> **倒排索引解决的是“怎么快速找到包含这些词的文档”，BM25 解决的是“这些文档里谁更相关、该排前面”。**

很多人会把 ES 搜索理解成“查到就行”，但实际上它至少分两步：

1. **先靠倒排索引快速召回候选文档**
2. **再靠 BM25 做相关性排序**

---

#### 1）什么是倒排索引？

倒排索引不是“文档里有哪些词”，而是：

> **某个词出现在哪些文档里。**

比如有三篇文档：

```text
Doc1: redis cache
Doc2: mysql redis
Doc3: es search
```

如果按“正排”思维看，会是：

```text
Doc1 -> redis, cache
Doc2 -> mysql, redis
Doc3 -> es, search
```

而倒排索引会更像：

```text
redis  -> [Doc1, Doc2]
cache  -> [Doc1]
mysql  -> [Doc2]
search -> [Doc3]
```

这样用户搜 `redis` 时，不需要扫所有文档，直接就能拿到：

```text
redis -> [Doc1, Doc2]
```

所以倒排索引的核心价值是：

> **根据词快速找到文档候选集。**

---

#### 2）倒排索引只解决“找到”，不解决“排谁前面”

如果用户搜：

```text
redis cache
```

倒排索引可能先找到很多候选文档：

- Doc1 包含 redis 和 cache
- Doc2 只包含 redis
- Doc4 也包含 redis，但出现次数很多

这时问题就变成：

> 这些候选文档里，谁更相关？

这一步就不是倒排索引本身能解决的，而要靠相关性打分模型。  
ES 默认最经典的全文相关性模型之一，就是：

- **BM25**

---

#### 3）BM25 是什么？

BM25 可以理解成：

> **一种“关键词相关性打分公式”，用来判断哪篇文档更像用户想要的结果。**

它关注的不是简单“包含没包含”，而是会综合考虑：

1. **词频（Term Frequency）**
   
   - 某个词在文档里出现得越合理，相关性通常越高

2. **逆文档频率（IDF，Inverse Document Frequency）**
   
   - 一个词如果全库到处都有，比如“的”“是”，区分度就低
   - 一个词如果比较少见，比如专业术语，区分度就高

3. **文档长度归一化**
   
   - 太长的文档天然更容易“碰巧包含很多词”
   - 所以要避免长文档因为字多就不公平占优

所以 BM25 的核心思想是：

> **词出现了多少、这个词稀不稀有、文档是不是过长，这三件事要一起看。**

---

#### 4）你可以怎么把倒排索引和 BM25 连起来理解？

你可以把 ES 的关键词检索流程理解成：

```text
query
  -> 分词器切词
  -> 倒排索引找候选文档
  -> BM25 给候选文档打分
  -> 按分数排序返回
```

比如搜：

```text
java channel goroutine
```

可能流程就是：

1. 先分词
2. 从倒排索引里找出包含这些词的文档
3. BM25 再判断：
   - 哪篇文档同时覆盖词更多
   - 哪篇文档词更集中
   - 哪篇文档更像用户想要的答案

所以：

- **倒排索引负责召回**
- **BM25 负责排序**

---

#### 5）为什么“用分词器 + 倒排索引 + BM25”是一整套？

因为这三者其实是连着的：

##### 分词器（Analyzer）

负责把文本拆成词项

##### 倒排索引

负责建立：

```text
词 -> 文档列表
```

##### BM25

负责在候选文档之间算相关性分数

所以面试时如果别人问：

> ES 为什么搜索快？

你不要只答“倒排索引”，更完整的说法是：

> **ES 先通过分词和倒排索引快速召回候选文档，再通过 BM25 做全文相关性排序。**

---

#### 6）面试里推荐这样答

> 倒排索引的核心作用是根据词快速找到包含这些词的文档候选集，它解决的是“怎么高效召回”；BM25 则是在这些候选文档之上做相关性打分，综合考虑词频、词的稀有程度以及文档长度，解决的是“谁更相关、应该排前面”。所以 ES 的全文检索不是只有倒排索引，还包括了倒排索引召回和 BM25 排序这两层。

一句压缩版：

> **倒排索引负责找，BM25 负责排。**

---

### 3. text 和 keyword 的区别？

**答案：**

`text` 会分词，适合全文检索；`keyword` 不分词，适合精确匹配、聚合和排序。面试里一句话最好记成：**text 用来搜内容，keyword 用来做精确值判断。**

---

### 4. DocValues 是什么？为什么排序和聚合会提到它？

**答案：**

DocValues 是一种列式存储结构，适合按文档读取字段值，所以很适合做排序、聚合和脚本访问。倒排索引更适合“根据词找文档”，DocValues 更适合“根据文档取字段值”。

---

### 5. query 和 filter 的区别？match 和 term 又是什么关系？

**答案：**

先把最容易混的层次讲清楚：

- `query` 和 `filter` 是**查询语义层面**的区分
- `match`、`term`、`terms` 这些是 **Query DSL（查询 DSL）里的具体查询子句**

所以：

> **match 和 term 不是字段属性，而是 ES Query DSL 里的查询类型；text、keyword、long、boolean 才是字段类型。**

---

#### 1）query 和 filter 的区别

- `query`：既做匹配，也关心相关性评分 `_score`
- `filter`：只判断是否满足条件，不算相关性分数，通常更适合精确过滤

如果面试官问优化，常见答法是：

> 能用 filter 的条件尽量用 filter，因为它不参与打分，语义更清晰，也通常更适合缓存和过滤场景。

---

#### 2）match 和 term 是同一维度吗？

算是同一层级的“查询类型”，但语义完全不同。

更准确地说：

- 它们都属于 ES Query DSL 里的 **query clause（查询子句）**
- 但用途完全不同

你可以这样记：

| 查询类型           | 适用字段                           | 是否分词         | 常见用途        |
| -------------- | ------------------------------ | ------------ | ----------- |
| `match`        | `text`                         | 会分词          | 正文检索 / 全文匹配 |
| `match_phrase` | `text`                         | 会分词，且更看重短语顺序 | 短语匹配 / 精排增强 |
| `term`         | `keyword` / `long` / `boolean` | 不分词          | 精确过滤        |
| `terms`        | `keyword` 等                    | 不分词          | 多值精确过滤      |

所以：

> **match 和 term 不是同一种算法，但它们是同一层级的查询子句。**

---

#### 3）match 是什么？

`match` 主要用于：

- `text` 字段
- 全文检索
- 分词后做相关性匹配

例如：

```json
{
  "match": {
    "text_content": "java channel goroutine"
  }
}
```

它的特点是：

- 会走分词器
- 会走 BM25 相关性评分
- 不是字面全等匹配

所以更准确的说法是：

> **match 是基于分词和 BM25 的全文检索查询。**

---

#### 4）term 是什么？

`term` 主要用于：

- `keyword`
- `long`
- `boolean`
- 不希望被分词的字段

例如：

```json
{
  "term": {
    "user_id": 123
  }
}
```

它的特点是：

- 不分词
- 不走全文相关性匹配
- 更像“值就等于它”

所以：

> **term 是精确匹配查询，不是全文检索。**

### 6. 前缀树（Trie）和 FST 有什么区别？为什么 ES 会用 FST？

**答案：**

- `Trie`（前缀树）可以理解成“每个字符一个节点”，公共前缀会共享，所以很适合做前缀匹配。
- 但普通 Trie 的问题是节点多、指针多，内存开销比较大。
- `FST`（`Finite State Transducer`，有限状态转换器）本质上是对 Trie 的进一步压缩，不只是前缀共享，很多后缀路径也能复用。
- 所以：
  - Trie：结构直观，适合理解前缀匹配
  - FST：更省内存，更适合大规模词典驻内存

**推荐回答：**

> Trie 是前缀树，核心特点是公共前缀共享，但普通 Trie 节点多、指针多，内存容易膨胀。FST 可以看成对 Trie 的进一步压缩，它把公共前缀甚至部分后缀路径都复用起来，所以更适合存大规模词典。ES 底层做词典索引时更偏向用 FST，而不是用一个朴素的 Trie。

### 7. FST 是不是就是 ES 的模糊搜索？

**答案：**

- 不是。
- `FST` 本身不是“模糊搜索算法”，而是**倒排索引里的内存词典结构**。
- 它主要作用是：
  - 存 term dictionary
  - 支持前缀 / 通配 / 自动补全这类词项查找加速
- 模糊搜索本身通常还会涉及：
  - 编辑距离
  - Levenshtein Automaton
  - 再和 FST 表示的词典做匹配

**一句话记忆：**

> `FST` 不是模糊搜索本身，而是高效存储和查找 term 的底层结构；模糊查询是在它之上再做自动机匹配。

### 8. FST 的全称是什么？FST 里存的值是什么？

**答案：**

- `FST` 全称是 `Finite State Transducer`，中文通常叫**有限状态转换器**。
- 在 ES / Lucene 的语境里，它常用于表示 **term dictionary**。
- 词项本身作为路径，终点关联的值通常不是 BM25 分数，而更偏向：
  - posting list 的地址 / 偏移信息
  - 或与该 term 相关的索引定位信息

**关键点：**

> FST 负责“把 term 高效组织起来并能快速定位后续索引数据”，它不是直接拿来存 BM25 分数的。

### 9. Term Dictionary 为什么不用哈希表？

**答案：**

- 哈希表很适合精确查找，但不适合：
  - 前缀查询
  - 模糊匹配
  - 通配符匹配
  - 有序遍历
- 搜索引擎对词典的要求不只是“查到一个词”，还包括：
  - 找某个前缀下的所有词
  - 和自动机做交集
  - 按词典顺序遍历
- FST 正好适合这些需求，而且内存压缩效果也更好。

**推荐回答：**

> Term Dictionary 不直接用哈希表，是因为搜索引擎不只是做精确查找，还要支持前缀、模糊、通配和有序遍历。哈希表精确查找很快，但缺乏这种模式匹配能力；FST 既能压缩词典内存，又能支持前缀和自动机匹配，所以更适合搜索引擎词典场景。

### 10. AND / OR 和 BM25 是什么关系？

**答案：**

- `AND / OR` 更偏布尔筛选逻辑，决定的是：
  - 哪些文档能进入候选集
- `BM25` 决定的是：
  - 候选文档里谁更相关、谁排前面

所以它们不是替代关系，而是两步：

1. 先通过布尔逻辑筛出候选文档
2. 再通过 BM25 给候选文档排序

**一句话记忆：**

> `AND / OR` 解决“哪些文档进来”，`BM25` 解决“进来的文档怎么排”。  

- `text_content` 是 `text` 类型
  - 所以通常用 `match`
- `org_tag` 是 `keyword` 类型
  - 所以通常用 `term` / `terms`

---

#### 7）面试里推荐这样答

> ES 查询本质上就是 JSON 形式的 Query DSL。`match` 和 `term` 都属于 Query DSL 里的查询子句，算是同一层级，但用途不同。`match` 主要用于 `text` 类型字段，会经过分词并结合 BM25 做全文检索；`term` 主要用于 `keyword`、`long`、`boolean` 这类字段，不分词，适合精确过滤。所以在项目里，正文 `text_content` 一般用 `match`，权限字段、状态字段、用户 ID 这类更适合用 `term` 或 `terms`。

一句压缩版：

> **ES 查询是 JSON DSL；match 和 term 是同层级的查询类型，不是字段属性，前者做全文检索，后者做精确匹配。**

---

### 6. ES 支持事务吗？

**答案：**

不像 MySQL 那种 ACID 事务，ES 不适合作为强事务主库。它更偏搜索和分析系统，强调分布式可用、检索能力和近实时索引，而不是强事务一致性。

---

### 7. ES 为什么是近实时，不是强实时？

**答案：**

因为写入后，数据要等 refresh 之后才对搜索可见。写成功和“立刻可搜”不是同一个时刻，所以 ES 更准确地说是 Near Real-Time。

---

### 8. ES 的分片（Shard）和副本（Replica）有什么作用？

**答案：**

- `Shard` 用来水平拆分数据，解决单机存不下、单机扛不住的问题
- `Replica` 用来高可用和扩展读能力

一句话：

> 分片解决扩展，副本解决高可用和读吞吐。

---

### 9. 主分片（Primary Shard）和副本分片（Replica Shard）是什么关系？

**答案：**

写请求通常先到主分片，再同步到副本分片。主分片负责主写路径，副本分片负责冗余和读扩展。某个节点挂掉时，只要副本还在，集群还能做故障恢复。

---

### 10. ES 写入为什么会提到 translog？

**答案：**

Lucene 的 commit 很重，不可能每写一条文档就 commit 一次。所以 ES 会先把操作写入 translog，用来崩溃恢复。这样即使 refresh / flush 之前发生故障，也能靠 translog 恢复最近的写入。

---

### 11. refresh 和 flush 的区别是什么？

**答案：**

- `refresh`：让新数据对搜索可见
- `flush`：做更重的持久化动作，并滚动 translog

面试里不要把两者混成一个概念。最稳的说法是：

> refresh 解决“可搜索”，flush 更偏“持久化与恢复管理”。

---

### 11A. kNN、ANN、HNSW 分别是什么？它们是什么关系？

**答案：**

这三个概念在向量检索里经常一起出现，但它们不是同一层意思。

先把全称写清楚：

- `kNN` = `k-Nearest Neighbors`，**k 近邻搜索**
- `ANN` = `Approximate Nearest Neighbor`，**近似最近邻搜索**
- `HNSW` = `Hierarchical Navigable Small World`，**分层可导航小世界图**

一句话先区分：

> **kNN 说的是“要找什么”——找最近的 K 个向量；ANN 说的是“怎么更快地找”——允许近似；HNSW 是实现 ANN 的一种经典算法。**

---

#### 1）kNN 是什么？

`kNN` 的目标很直接：

> **给你一个查询向量，找出距离它最近的 K 个向量。**

这里的“最近”通常会基于某种相似度 / 距离度量，比如：

- 余弦相似度（cosine similarity）
- 欧式距离（L2 distance）
- 点积（dot product）

所以 `kNN` 更像一个问题定义：

> **我想找最相似的 K 个结果。**

如果数据量很小，最直接的方法就是：

- 把所有向量都算一遍距离
- 再挑 TopK

这个叫“精确 kNN”或“暴力搜索”。

---

#### 2）ANN 是什么？

如果向量数据非常大，比如：

- 100 万
- 1000 万
- 1 亿

那每次都全量算距离，代价就太高了。  
这时就会引入 `ANN`：

> **Approximate Nearest Neighbor，近似最近邻搜索。**

它的核心思想是：

- 不追求每次都绝对精确地找到理论最优最近邻
- 换取更高的查询速度和更低的资源消耗

所以 ANN 的 trade-off 是：

- **速度更快**
- **吞吐更高**
- **内存和 CPU 更省**
- 但结果可能不是数学意义上绝对最优的最近邻

所以：

> **kNN 是目标，ANN 是工程化优化思路。**

---

#### 3）HNSW 是什么？

`HNSW` 是一种非常常见的 ANN 算法。

它的名字是：

- `Hierarchical`：分层的
- `Navigable`：可导航的
- `Small World`：小世界图

你可以把它粗略理解成：

> **先把向量组织成一个多层图结构，搜索时从高层快速跳到目标附近，再在低层细找。**

这样做的好处是：

- 不用把所有向量都暴力比一遍
- 能很快缩小搜索范围
- 在大规模向量检索里效果很好

所以 HNSW 不是“kNN 的同义词”，也不是“ANN 的总称”，而是：

> **ANN 的一种具体实现算法。**

---

#### 4）除了 HNSW，还有哪些常见的 ANN 实现？

如果面试官继续追问，不需要一口气报很多名字，讲 2~3 个最常见的就够了。

##### （1）IVF

`IVF` = `Inverted File Index`

核心思路是：

> **先把向量分到很多桶（簇）里，查询时先找到最相关的几个桶，再只在这些桶里搜索。**

你可以理解成：

- 不全库暴力搜
- 先缩小候选范围
- 再在候选桶里细查

所以 IVF 更像：

> **先分桶，再检索。**

##### （2）PQ

`PQ` = `Product Quantization`

核心思路是：

> **把高维向量切成多个子空间，再分别做量化压缩。**

它的价值主要是：

- 更省内存
- 距离计算更快

代价是：

- 会有一定精度损失

所以 PQ 更像：

> **用压缩换空间和速度。**

##### （3）IVF + PQ

这是工业界非常常见的组合。

思路就是：

- 先用 IVF 缩小搜索范围
- 再用 PQ 压缩和加速

可以理解成：

> **先分桶，再压缩。**

所以很多大规模向量检索系统里，如果不是纯图索引路线，IVF + PQ 也是非常常见的一种方案。

面试里可以这样总结：

> 除了 HNSW，常见的 ANN 实现还可以讲 IVF、PQ 和 IVF+PQ。HNSW 更偏图索引，IVF 更偏先分桶缩小范围，PQ 更偏用量化压缩换速度和内存，IVF+PQ 则是把分桶和压缩组合起来。

---

#### 5）三者关系怎么记最稳？

你可以这样记：

```text
kNN  = 我想找最近的 K 个
ANN  = 我想更快地近似找到它们
HNSW = 我具体采用的一种 ANN 算法
```

或者再换一种说法：

- `kNN`：问题定义
- `ANN`：工程路线
- `HNSW`：具体算法实现

---

#### 6）在 ES 里它们怎么对应？

如果你在 ES 里做向量检索，一般说“knn 查询”，其实往往背后已经不是纯暴力精确搜索，而是：

- 前端 / 查询层看起来叫 `knn`
- 底层可能使用的是 `HNSW` 这类 ANN 索引结构

所以更准确地说：

> **ES 暴露给你的接口层可能叫 kNN 查询，但底层为了提速，很可能依赖 HNSW 这种 ANN 算法。**

---

#### 7）面试里推荐这样答

> kNN 的全称是 k-Nearest Neighbors，指的是找最近的 K 个向量；ANN 的全称是 Approximate Nearest Neighbor，指的是为了提升大规模向量检索性能，允许做近似搜索；HNSW 的全称是 Hierarchical Navigable Small World，是一种常见的 ANN 索引算法。三者关系可以理解成：kNN 是检索目标，ANN 是近似加速思路，HNSW 是实现 ANN 的具体算法。

一句压缩版：

> **kNN 是“找什么”，ANN 是“怎么更快找”，HNSW 是“具体怎么实现 ANN”。**

---

### 12. ES 深度分页为什么慢？

**答案：**

因为深度分页时，各个分片都要先取出更大范围的候选结果，再由协调节点做汇总排序。比如你取第 10001~10010 条，每个分片前面那一大段结果也得先参与排序，成本很高。  
所以深分页常见优化是：

- `search_after`
- scroll（更偏离线导出）
- 避免用户真的翻到很深

---

### 13. ES 为什么不适合做高频更新场景？

**答案：**

因为底层很多更新不是原地修改，而更像“删旧写新”，再叠加 segment merge，会带来较明显的写放大和资源消耗。所以 ES 更适合读多、搜索多、写入相对批量或日志型的场景。

---

### 14. text 字段为什么不适合直接聚合和排序？

**答案：**

因为 `text` 会被分词，它面向的是全文检索，不适合直接拿来做精确排序和聚合。工程上更常见的做法是给同一个字段配一个 `.keyword` 子字段，用来做聚合和排序。

---

### 15. ES 查询流程为什么常说 Query Then Fetch？

**答案：**

因为搜索通常先由各分片本地算出候选 TopN（Query 阶段），再由协调节点汇总，并回拉真正需要返回的文档内容（Fetch 阶段）。这是分布式搜索场景下兼顾效率和结果排序的一种典型流程。

---

## 四、ES 面试常见延伸题

### 1. ES 和数据库联合使用时，为什么常讲 MySQL + ES 双写一致性？

**答案：**

因为很多系统把 MySQL 当事实来源，把 ES 当搜索副本。这样一来就会有经典问题：

- MySQL 更新了
- ES 什么时候同步？
- 同步失败怎么办？

工程上常见做法：

- 业务双写（简单但风险高）
- MySQL binlog + Canal / CDC + MQ 异步同步 ES（更常见）

所以这题和 Redis 缓存一致性其实很像，本质上都是：

> **主存储变了，搜索副本怎么追上来。**

---

### 2. ES 为什么适合日志检索？

**答案：**

因为日志的典型特点是：

- 写入多
- 查询多
- 搜索条件复杂
- 经常要按字段过滤、按时间聚合、按关键词搜索

这正好是 ES 擅长的方向。尤其是日志系统里：

- 全文检索
- 范围过滤
- 聚合统计
- 时间序列查询

都很常见。

---

### 3. ES 在 RAG 里一般承担什么角色？

**答案：**

在 RAG 里，ES 常见角色是：

- 做关键词召回
- 做结构化过滤
- 做混合检索中的一部分
- 结合 BM25 做传统文本检索

所以 ES 在 RAG 场景里不一定单独扛全部检索，但它非常适合作为：

> **关键词检索 / 条件过滤 / 混合召回中的稳定基础层。**
