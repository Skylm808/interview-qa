## 一、先讲清楚：Agent 开发和传统后端开发到底差在哪

### 1. 你的理解方向是对的，但要纠偏

你说的这句：

> 传统网关层是请求进来后，网关分发给不同服务；现在是网关分发给不同需求的 agent 去做服务。

**这个理解方向是对的，但不够完整。**

更准确的说法是：

- **传统后端**：网关把请求转到某个确定的业务服务，后面的调用链基本是**人写死的**，例如 `order-service -> inventory-service -> payment-service`。
- **Agent 开发**：网关后面仍然有传统服务，但在服务之上多了一层**任务理解 / 规划 / 工具选择 / 记忆 / 评估**的“智能编排层”。
- 所以 Agent **不是把微服务替掉**，而是把原本由人手写在代码里的那部分“决策逻辑”和“流程编排逻辑”部分上移，交给一个**可规划、可试错、可回看轨迹**的执行体来完成。

一句话：

> **传统后端更像“固定流程机”，Agent 后端更像“带策略和工具系统的任务执行机”。**

---

### 2. 传统后端 vs Agent 开发：核心差异表

| 维度   | 传统后端开发         | Agent 开发                        |
| ---- | -------------- | ------------------------------- |
| 核心对象 | API、服务、数据库、缓存  | 任务、目标、工具、记忆、评估                  |
| 调度方式 | 人工写死流程         | 模型/编排层动态决策                      |
| 输入形式 | 结构化请求          | 自然语言 + 半结构化上下文                  |
| 输出形式 | 确定性 JSON/状态变更  | 文本、结构化结果、工具动作、长链路任务结果           |
| 系统重点 | 高并发、高可用、一致性、性能 | 除了传统重点，还要解决规划质量、幻觉、工具可靠性、轨迹可观测  |
| 状态管理 | Session、DB、缓存  | 会话状态 + 短期记忆 + 长期记忆 + 工具执行状态     |
| 错误类型 | 超时、重试、幂等、脏数据   | 还多了错规划、乱调工具、上下文漂移、幻觉、循环调用       |
| 验证方式 | 单测/集测/压测       | 还要做 eval、trace、人工验收、bad case 回放 |
| 架构边界 | 服务边界清晰、接口强约束   | 工具边界 + Prompt 约束 + 状态机/图编排共同约束  |

---

### 3. 两类系统的典型链路

#### 传统后端链路

```text
用户请求
  -> API Gateway
  -> 某个业务服务
  -> 调用下游服务 / DB / Cache / MQ
  -> 返回结果
```

#### Agent 后端链路

```text
用户请求
  -> API Gateway
  -> Agent 接入层
  -> 意图识别 / 路由
  -> Planner / Orchestrator
  -> 选择：直接回答 / RAG / Tool Call / Multi-Agent
  -> 执行工具 / 调服务 / 查知识库 / 写状态
  -> Critic / Guardrail / 校验
  -> 返回结果 + 轨迹日志 + 评估指标
```

注意：

- **底下那些 DB、Redis、MQ、业务服务依然还在。**
- Agent 更像是这些能力之上的“**智能编排层**”。
- 因此真正成熟的 Agent 开发岗位，通常仍然要求你有扎实的 **Java / Go / Python / MySQL / Redis / MQ / 分布式 / 观测性** 基础，而不是只会调模型 API。

---

### 4. 面试里怎么回答“为什么 Agent 开发和传统后端不一样？”

你可以直接这样答：

> 传统后端开发的核心是把确定性的业务流程工程化，接口、状态流转、异常处理大多是开发者预先写死的；而 Agent 开发是在这个基础上增加了一层以大模型为核心的任务编排能力。它不仅要做普通后端的服务治理、高可用和性能优化，还要解决任务规划、工具选择、长短期记忆、上下文压缩、轨迹可观测和自动化评估等问题。我的理解是，Agent 不是替代微服务，而是在微服务之上增加一个面向目标执行的智能控制层。

这段答法，面试官一般会觉得：**你没有把 Agent 说成“套壳聊天机器人”**，也没有把它说成“完全推翻传统后端”，是比较成熟的理解。

### Q1：什么是 AI Agent？它和普通聊天模型有什么区别？

**来源倾向：滴滴、腾讯、阿里、字节**

**推荐答案：**

AI Agent 是一种**以目标驱动**的智能系统，不只是根据输入生成一句回复，而是会围绕任务目标去做理解、规划、调用工具、读取外部信息、根据执行结果调整下一步动作，直到任务完成。

它通常包含五层：

1. **模型层**：负责语言理解和生成；
2. **规划层**：负责任务拆解和决策；
3. **工具层**：负责调 API、查数据库、读文件、执行外部动作；
4. **记忆层**：负责短期上下文和长期经验沉淀；
5. **控制/校验层**：负责重试、终止条件、安全约束和结果验证。

和普通聊天模型比，普通模型更偏“一问一答”；Agent 更强调“**任务闭环**”。比如分析异常、拉取日志、调用工具生成报告、执行多步流程，这些都属于 Agent 更擅长的范畴。

一句话总结：

> **LLM 更像会说话的大脑，Agent 是能围绕目标持续做事的执行系统。**

---

### Q2：Agent 和传统后端系统相比，新增了哪些核心工程问题？

**来源倾向：滴滴、阿里、腾讯、美团 AI 后端**

**推荐答案：**

我会把新增问题概括成四类：

1. **规划正确性**：模型不一定总能做出对的决策，可能出现乱拆任务、错误工具选择、逻辑塌缩；
2. **状态管理**：除了普通后端状态，还多了短期记忆、长期记忆、工具执行状态、轨迹状态；
3. **可观测与评估**：传统后端只看耗时、QPS、错误率不够，还要看答案质量、工具成功率、Hallucination Rate、任务完成率；
4. **安全与约束**：要限制高风险工具调用、保证参数合法、避免 prompt injection 和越权执行。

所以 Agent 开发比传统后端多的，不是“会调模型接口”这么简单，而是多了一套**任务编排 + 状态治理 + 质量评估**体系。

---

### Q3：Agent 和 RAG 的区别是什么？

**来源倾向：滴滴、阿里、美团、字节**

**推荐答案：**

RAG 的核心是**检索增强生成**，主要解决的是模型知识不足、知识过期、领域知识缺失的问题。它本质上仍然是“**先查资料，再回答**”的问答范式。

Agent 的范围更大，它不只是查资料，还会：

- 判断任务类型；
- 决定要不要检索；
- 决定要不要调用工具；
- 处理多步任务；
- 根据中间结果继续调整策略。

所以我会这样概括：

> **RAG 是 Agent 常用的一种能力组件，但 Agent 不等于 RAG。**

一个成熟 Agent 里，RAG 更像“知识获取模块”，而 Agent 是“决策 + 执行 + 反馈”的总系统。

### Q4：一个完整的 Agent 架构你会怎么设计？

**来源倾向：阿里、滴滴、腾讯、字节**

**推荐答案：**

我会按分层设计：

1. **接入层**：接 HTTP / SSE / WebSocket 请求，做鉴权、限流、租户隔离；
2. **编排层**：做意图识别、任务拆解、Prompt 模板管理、工具路由；
3. **检索层**：做 chunk、embedding、混合召回、rerank、元数据过滤；
4. **工具层**：统一封装 DB、搜索、浏览器、外部 API、业务服务；
5. **记忆层**：维护会话态、摘要记忆、长期经验库；
6. **观测层**：记录 trace、token、耗时、工具成功率、answer quality；
7. **安全层**：做人审确认、参数校验、越权控制、敏感工具防误触。

如果更工程化，我会补一个**结果校验层**：对工具参数、工具返回、最终答案做结构校验和一致性校验，因为 Agent 真正难的地方不是“能不能规划”，而是“规划完能不能稳定执行”。

---

### Q5：为什么很多 Agent 选择 ReAct，而不是纯工作流？

**来源倾向：腾讯、阿里、通用 Agent 岗**

**推荐答案：**

ReAct 的价值在于把“**Reasoning**”和“**Acting**”结合起来：

- 先想下一步做什么；
- 再调用工具；
- 再根据 Observation 修正下一步。

纯工作流适合边界明确、路径稳定的任务，比如审批流、固定报表生成；但一旦任务有歧义、需要动态补信息、需要根据中间结果改计划，纯工作流会非常僵硬。

我更认可的工程实践不是“全 Agent”或“全 Workflow”，而是：

> **用 Workflow 管住大框架，用 ReAct 给 Agent 局部决策权。**

这样既保留灵活性，又不会失控。

### Q6：长短期记忆怎么设计？Session 和 User ID 怎么绑定？

**来源倾向：腾讯、字节、淘天**

**推荐答案：**

我一般把记忆分三层：

1. **短期记忆**：当前会话窗口中的消息历史和中间结果；
2. **摘要记忆**：把长会话压缩成结构化摘要，例如用户偏好、已确认事实、当前任务状态；
3. **长期记忆**：跨会话沉淀到向量库或结构化存储里的经验、画像、历史轨迹。

绑定策略上：

- **Session ID** 负责一次会话内的上下文连续；
- **User ID** 负责跨会话的长期画像和长期记忆召回；
- 两者不能混为一谈，否则会出现会话污染和用户串话。

工程上我会注意三点：

- 会话态尽量轻，别把整段长历史都塞上下文；
- 长期记忆必须带时间戳、来源和过期策略；
- 召回长期记忆前要做权限和用户维度校验。

### Q7：Multi-Agent 应该怎么拆？为什么要这么拆？

**来源倾向：腾讯、阿里、滴滴、牛客 Agent 专题**

**推荐答案：**

Multi-Agent 不应该为了“看起来高级”而拆，拆分标准应该是：

1. **职责是否稳定**：比如 Planner、Retriever、Executor、Critic 职责边界清楚；
2. **工具集合是否不同**：不同 Agent 访问不同工具，安全边界更清晰；
3. **上下文是否可局部化**：某个子任务只需要局部上下文，就适合独立 Agent；
4. **是否存在并行收益**：比如代码生成、代码审查、测试生成可并行。

我通常不建议按“功能点越多越好”去拆，因为拆得太细会带来通信成本、上下文同步成本和循环风险。

一句话：

> **Multi-Agent 的核心不是多，而是让每个 Agent 的职责、工具权限和状态边界足够清晰。**

### Q8：MCP、Function Call、Skill 的区别是什么？

**来源倾向：腾讯、阿里、字节、小红书 AI 全栈**

**推荐答案：**

这三个概念很容易混，是因为它们经常一起出现，但其实它们分属**不同层**。

先给最稳的分层答案：

- **MCP**：解决“能力怎么标准化接进来”
- **Function Call**：解决“模型这一次具体怎么调用某个能力”
- **Skill**：解决“拿到这些能力后，按什么方法完成一类任务”

一句话压缩版：

> **MCP 负责把能力接进来，Function Call 负责真的去调，Skill 负责规定怎么用这些能力完成任务。**

---

#### 1）先分别看这三个概念

##### MCP（Model Context Protocol）

你可以把 MCP 理解成：

> **模型接入外部能力的规范协议。**

它关注的是：

- 文件系统怎么接进来
- 数据库怎么接进来
- 浏览器怎么接进来
- 搜索、日志、文档资源怎么接进来
- 工具能力怎么被标准化暴露给模型

所以 MCP 更像：

> **接入层 / 协议层**

也就是你说的：

> “MCP 是接入能力的规范协议”

这个理解是对的。

---

##### Function Call

Function Call 更像：

> **模型发起一次结构化调用的动作。**

比如模型决定调用：

```json
{
  "name": "get_weather",
  "arguments": {
    "city": "Beijing"
  }
}
```

这里关心的是：

- 调哪个函数
- 参数是什么
- 返回结果是什么

所以 Function Call 解决的是：

> **“这次具体怎么调”**

它更像：

> **调用层 / 动作层**

---

##### Skill

Skill 更像：

> **一套任务方法、SOP（标准操作流程）或工作流模板。**

它关注的是：

- 先做什么
- 后做什么
- 什么情况下调用什么工具
- 输出格式是什么
- 遇到失败怎么办

所以 Skill 解决的是：

> **“这类任务该怎么做”**

它更像：

> **方法层 / 工作流层**

---

#### 2）为什么你会觉得它们很像？

因为现实里它们经常一起配合：

1. **MCP** 把外部能力接进来
2. 模型通过 **Function Call** 真的去调用
3. 再按 **Skill** 的方法把整个任务做完

所以它们是：

> **协作关系，不是互斥关系。**

---

#### 3）用一个最容易懂的类比

你可以把它们类比成“公司里做事”：

##### MCP

像是：

> 公司把财务、法务、人事、运维这些部门都接进了统一通讯录和工单系统。

也就是：

- 现在有哪些能力可用
- 这些能力怎么统一接进来

##### Function Call

像是：

> 你真的发起了一张工单。

比如：

- 查报销
- 重启服务
- 查假期

也就是：

- 这次具体调哪个能力
- 参数是什么

##### Skill

像是：

> 公司教你的“这类事情标准流程怎么做”。

比如“新人入职流程”：

1. 先开账号
2. 再领电脑
3. 再配权限
4. 再登记信息

也就是：

- 一套可复用的任务方法

---

#### 4）再用 AI Coding 的例子看一遍

假设用户说：

> 帮我读日志，然后总结根因

##### MCP 在做什么？

把这些能力接给模型：

- 文件系统
- shell
- 日志搜索
- 浏览器
- 文档库

##### Function Call 在做什么？

模型真的发起调用：

```json
{
  "name": "read_file",
  "arguments": {
    "path": "app.log"
  }
}
```

或者：

```json
{
  "name": "search_logs",
  "arguments": {
    "keyword": "ERROR"
  }
}
```

##### Skill 在做什么？

Skill 会规定：

1. 先看最近错误
2. 再看上下文堆栈
3. 再分类根因
4. 最后输出“现象 / 根因 / 修复建议”

所以：

- MCP 决定“能力怎么接进来”
- Function Call 决定“这次具体怎么调”
- Skill 决定“整类任务怎么做”

---

#### 5）面试里最稳的答法

> MCP 更像模型接入外部工具和资源的标准协议，解决的是“能力怎么标准化暴露给模型”；Function Call 解决的是“模型这一次具体怎么调用某个函数或工具”；Skill 则更像高层任务方法，规定一类任务的步骤、约束和输出格式。简单说，MCP 是接入层，Function Call 是调用层，Skill 是工作流层。

如果面试官继续追问 Skill 的价值，可以补一句：

> Skill 的核心价值，不是多一个概念，而是把一类高频任务的 Prompt、步骤、工具选择、输出格式和约束沉淀下来，降低重复设计成本，提高执行一致性。

### Q9：LangChain、LangGraph、Spring AI 分别适合什么场景？

**来源倾向：腾讯、字节**

**推荐答案：**

我的理解是：

- **LangChain** 更偏能力封装，适合快速把模型、检索、工具串起来；
- **LangGraph** 更偏有状态的 Agent 编排，适合做状态机、条件边、循环、长链路任务；
- **Spring AI** 更适合 Java 生态和企业后台，把模型调用、Prompt、向量库、工具调用更自然地接进 Spring 体系。

如果做 PoC，LangChain 上手快；
如果做真正的 Agent 编排，LangGraph 更强；
如果团队本来就是 Java / Spring 技术栈，Spring AI 的落地成本更低。

所以选型不该只看流行度，而应该看：

1. 团队语言栈；
2. 是否需要强状态管理；
3. 是否需要企业级集成和治理。

### Q10：可插拔 RAG 和多租户多模型热切换要怎么做？

**来源倾向：腾讯、字节**

**推荐答案：**

我会把“模型配置”和“RAG 配置”抽象成可版本化配置中心，而不是写死在代码里。

例如：

- 每个租户绑定自己的模型配置、Embedding 模型、向量库、reranker、Prompt 模板；
- 编排层通过工厂模式或策略模式按租户和场景加载配置；
- 变更配置时优先走配置热更新，不直接重启服务；
- 对关键配置加版本号和回滚能力，避免热更新造成线上污染。

热更新的关键不是“能不能改”，而是要回答三个问题：

1. **切换是否租户隔离？**
2. **旧请求是否受影响？**
3. **回滚是否可控？**

所以我会用“读写分离配置 + 版本号 + 灰度发布 + trace 验证”去保证切换安全。

### Q11：RAG 系统怎么设计？怎么评估？

**来源倾向：阿里、美团、滴滴、字节**

**推荐答案：**

RAG 设计我一般分离线和在线两段：

#### 离线段

- 文档清洗
- chunk 切分
- 去重
- embedding 计算
- 向量入库
- 元数据索引构建

#### 在线段

- query 理解
- 混合召回（向量 + BM25）
- rerank
- 上下文拼接
- 生成回答
- 回答后校验

评估我不会只说“看准确率”，而会拆成三层：

1. **检索层**：Recall@K、MRR、Hit Rate；
2. **回答层**：Faithfulness、Answer Relevance、Completeness；
3. **业务层**：任务完成率、人工满意度、bad case 分布。

如果面试官追问怎么优化，我会说：

- 先优化 chunk 和 metadata；
- 再做混合召回和 rerank；
- 最后补 query rewrite / multi-query / self-rag；
- 不要上来就狂改 Prompt。

### Q12：怎么控制 Agent 工具调用的可靠性和安全性？

**来源倾向：字节、小红书 AI 全栈、腾讯、阿里**

**推荐答案：**

我会分三层做：

#### 1）语法层可靠性

- JSON Schema / 强类型约束
- 参数枚举和必填项校验
- Tool 输出结构化约束

#### 2）逻辑层可靠性

- 调用前校验参数是否合法
- 调用后校验返回是否完整
- 失败重试 / fallback / degrade
- 高风险工具必须 human-in-the-loop

#### 3）安全层控制

- 工具最小权限原则
- 操作白名单 / 黑名单
- 审计日志
- 防 Prompt Injection：把外部文档和系统指令做隔离，不允许文档内容覆盖系统约束

如果问“MCP update 方法如何保证安全”，我会强调：

> update 不能让模型直接携带任意字段落库，必须经过 schema 校验、权限校验、字段白名单过滤和审计记录。

### Q13：为什么很多公司会问 SSE？SSE 在 Agent 系统里怎么设计？

**来源倾向：字节、小红书 AI 全栈**

**推荐答案：**

因为 Agent / AI Coding 场景里，用户体验往往依赖**流式输出**，而 SSE 是实现成本较低、适合服务端单向推流的方案。

我在后端会把 SSE 事件拆成几类：

- `message_delta`：正文流式 token；
- `tool_call_start`：开始调用工具；
- `tool_call_result`：工具返回摘要；
- `state_update`：Planner 状态变化；
- `done`：本轮结束；
- `error`：错误事件。

这样前端不只是看到“字一个个出来”，还能看到**Agent 当前处于什么阶段**。

工程上要注意：

- 心跳保活；
- 客户端断连检测；
- 取消请求；
- tool call 阶段的中间事件格式一致；
- 日志 trace id 要贯穿整条 SSE 链路。

### Q14：为什么要做 FSM / 状态机？

**来源倾向：腾讯 AI 应用开发**

**推荐答案：**

因为纯 LLM 驱动的 Agent 如果没有显式状态，很容易出现：

- 多轮对话阶段错乱；
- 工具重复调用；
- 任务未结束却继续循环；
- 用户澄清前就强行下结论。

FSM 的价值是把“允许发生什么、下一步可以走到哪”写清楚。

比如我会把对话分成：

- `INIT`
- `INTENT_RECOGNIZED`
- `NEED_CLARIFY`
- `RAG_EXECUTING`
- `TOOL_EXECUTING`
- `ANSWER_GENERATING`
- `DONE`
- `FAILED`

这样做以后，系统的可解释性、回放能力和终止控制都会强很多。

### Q15：为什么有些场景要用 Agent，而不是用传统技术直接写死？

**来源倾向：滴滴、阿里、腾讯、美团 AI 后端**

**推荐答案：**

不是所有场景都该上 Agent。我的判断标准是看三个维度：

1. **任务是否开放式**：路径不固定、需要动态补信息、需要临场决策；
2. **输入是否模糊**：用户表达不完整、存在歧义；
3. **工具组合是否动态**：同一目标下可能有多条执行路径。

如果是固定流程、高稳定、高审计要求的任务，传统规则流往往更稳；
如果是复杂问答、问题诊断、研发辅助、电商导购、跨工具任务执行，这类动态决策需求强，Agent 更合适。

所以标准答案不是“Agent 更先进所以都该用”，而是：

> **对不确定性高、流程动态、工具组合复杂的任务，Agent 更有优势；对确定性强、审计严格的任务，传统系统更稳。**

### Q16：主动澄清（clarification）逻辑怎么设计？

**来源倾向：淘天 AI Agent / 电商导购场景**

**推荐答案：**

主动澄清的核心是判断“当前信息是否足以支持低风险决策”。

我一般会让系统先打一个**置信度分数**，从三个角度看：

1. 意图是否明确；
2. 参数是否齐全；
3. 直接执行的风险是否可接受。

如果缺参数、歧义大、执行后果敏感，就应该反问；
如果历史画像很强、歧义很小、执行可回滚，就可以结合历史直接推断。

一个比较稳的面试答法是：

> 我不会把澄清逻辑只交给 Prompt，我会引入结构化判定：意图置信度、槽位完整度、风险等级三项联合决定是否反问。

---

### Q17：多工具调用、3 个以上工具链路时，怎么压低端到端延迟？

**来源倾向：阿里、淘天、快手**

**推荐答案：**

我会从四个方向优化：

1. **DAG 并行化**：无依赖的工具并行执行，不必串行；
2. **轻重模型分层**：分类、路由先走小模型，复杂总结再走大模型；
3. **缓存与结果复用**：高频工具结果做 TTL 缓存；
4. **减少无效规划**：减少 planner 的来回思考次数，避免长链路循环。

如果场景允许，我还会做：

- 预取（predictive fetch）
- 工具超时熔断
- fallback 策略
- RAG 召回 topK 自适应

一句话：

> **延迟优化本质上不是单点优化，而是把“规划次数、工具串行度、模型成本、缓存命中率”一起做系统 trade-off。**

### Q17.5：KV Cache 是什么？它算不算 AI Coding / Agent 开发八股？

**来源倾向：字节 AI Infra、阿里 AI Infra、推理服务、Agent 平台、LLM 应用后端**

**推荐答案：**

**算，而且是“偏 AI 基础设施 / 大模型推理 / Agent 平台性能优化”的八股。**

如果岗位只是普通业务后端，`KV Cache` 不一定高频；但如果岗位开始提这些关键词，它就很可能会被问：

- LLM 推理优化
- AI Infra
- AI Gateway
- Agent 平台
- 大模型服务化
- 长上下文 / 多轮对话
- 降低首 token 延迟和生成延迟

一句话先记住：

> **KV Cache 就是在 Transformer 推理阶段，把已经算过的 Key / Value 结果缓存起来，下一轮生成新 token 时直接复用，避免把历史 token 全量重复计算一遍。**

#### 为什么叫 KV Cache？

因为在 Transformer 的注意力（Attention）计算里，会把输入表示变换成三类向量：

- `Q`：`Query`
- `K`：`Key`
- `V`：`Value`

在自回归生成（autoregressive generation）场景里，比如模型一个 token 一个 token 往外吐字：

- 历史 token 的 `K` 和 `V` 一旦算过，后面其实不用反复重算
- 新来的 token 只需要算自己的 `Q`，再结合历史缓存的 `K/V` 做注意力计算

所以会把历史的 `K/V` 存起来，这就是 `KV Cache`。

#### 不用 KV Cache 会发生什么？

假设模型已经生成了 100 个 token，现在要生成第 101 个 token：

- **不用 KV Cache**：前 100 个 token 的注意力相关计算又要重新参与一遍
- **用了 KV Cache**：前 100 个 token 的 `K/V` 直接复用，只算新 token 相关部分

所以 KV Cache 的核心价值就是：

> **减少重复计算，降低推理时延，提高吞吐。**

#### 一个最容易懂的例子

你可以把它类比成“开会做会议纪要”：

- 如果每来一个新同事，你都要把前面 2 小时讨论内容重新现场推导一遍，这就很慢
- 更合理的方法是，把前面内容记成纪要，后面直接翻纪要继续讨论

这里：

- 历史上下文 = 前面已经说过的话
- `KV Cache` = 这份“已经整理好的会议纪要”

模型后续生成时，不需要把历史上下文从头重新推导一次，而是基于缓存继续往下生成。

#### 它和 Agent / AI Coding 有什么关系？

它和 Agent 的关系主要体现在**性能**而不是“业务逻辑”。

比如一个 Agent 系统常常有这些特点：

- 多轮对话
- 上下文越来越长
- 频繁调用模型
- 需要流式输出
- 对首 token 延迟很敏感

这时如果没有 KV Cache：

- 每轮生成都要重复处理大量历史上下文
- 延迟会明显升高
- GPU 显存和算力消耗也更大

所以你可以这样理解：

> `KV Cache` 不是 Agent 逻辑层的概念，但它是 Agent 系统底层推理性能优化里非常重要的一环。

#### KV Cache 的代价是什么？

这也是面试里常见追问。

虽然它能提速，但也有成本：

1. **占显存 / 内存**  
   上下文越长、batch 越大、层数越多，KV Cache 占用越大。

2. **长上下文时缓存会越来越重**  
   对超长对话、超长 prompt，KV Cache 可能成为显存瓶颈。

3. **多用户并发场景更难管理**  
   推理服务需要管理不同请求的 cache 生命周期。

所以工程上常见关注点是：

- cache 复用
- cache 淘汰
- prefix cache（前缀缓存）
- paged attention / 分页式 cache 管理
- 显存与吞吐 trade-off

#### 面试里怎么答最稳？

> KV Cache 是 Transformer 推理优化里的一个核心概念。模型在自回归生成时，会把历史 token 已经算过的 Key 和 Value 缓存起来，后续生成新 token 时直接复用，而不是把整段历史重新计算一遍。它的主要价值是降低推理延迟、提高吞吐，但代价是会额外占用显存，所以在长上下文、多轮对话和高并发推理服务里，KV Cache 管理本身也是一个重要工程问题。

#### 如果面试官问：这算 Agent 八股吗？

你可以这样答：

> 如果岗位更偏业务 Agent 编排，KV Cache 不一定是最核心八股；但如果岗位涉及 AI Coding、Agent 平台、推理服务、AI Infra 或长上下文优化，那它就是很典型的高频问题。因为 Agent 系统虽然表面上看是工具编排和任务执行，但底层模型调用的时延、吞吐、显存成本，都会直接受 KV Cache 影响。

### Q18：如何做 Agent 的评估和 AgentOps？

**来源倾向：快手、阿里、腾讯、通用大模型应用岗**

**推荐答案：**

我会把评估拆成三层：

#### 1）离线评测

- 工具调用成功率
- 任务完成率
- 规划正确率
- answer faithfulness
- hallucination rate

#### 2）线上观测

- 首 token 延迟
- 端到端耗时
- 每轮工具调用次数
- fallback 比例
- 人工接管率

#### 3）bad case 闭环

- 错误类型分类：检索错、规划错、参数错、幻觉、超时；
- 典型失败轨迹回放；
- 新数据回灌评测集；
- A/B test 验证优化是否真的有效。

我会强调：

> Agent 系统不能只看“能跑通”，必须能追踪“为什么成功、为什么失败、失败后如何稳定复现并修复”。

## 四、TDD 为什么会成为 AI Coding 面试里的加分点

### 1. 先把缩写和术语讲清楚

为了避免面试时一堆缩写说得太“黑话”，这部分先统一解释：

| 缩写 / 术语     | 全称                           | 中文意思 / 你可以怎么理解           |
| ----------- | ---------------------------- | ------------------------ |
| `TDD`       | `Test-Driven Development`    | 测试驱动开发，先写失败测试，再写最小实现     |
| `DSL`       | `Domain-Specific Language`   | 领域特定语言，面向某类业务的专用描述语言     |
| `YAML`      | `YAML Ain't Markup Language` | 一种常见配置格式，常用来写工作流、配置文件    |
| `JSON`      | `JavaScript Object Notation` | 一种常见结构化数据格式              |
| `DAG`       | `Directed Acyclic Graph`     | 有向无环图，常用于表达任务依赖关系        |
| `schema`    | -                            | 结构约束 / 数据格式定义，比如字段类型、必填项 |
| `parser`    | -                            | 解析器，把输入文本解析成内部结构         |
| `compiler`  | -                            | 编译器，把输入规则/DSL 转成可执行计划    |
| `validator` | -                            | 校验器，负责检查输入/输出是否合法        |
| `router`    | -                            | 路由器，决定请求下一步该走哪个分支 / 工具   |
| `fallback`  | -                            | 降级 / 兜底策略，主流程失败后切换备用方案   |
| `eval`      | `evaluation`                 | 评测，通常指对模型或系统效果的系统性评估     |
| `trace`     | -                            | 调用链追踪 / 运行轨迹，用来观察执行过程    |

你在面试里如果要说这些词，最好别只丢缩写，建议顺手补一句中文意思。这样会显得你表达更清晰，也更照顾面试官理解节奏。

---

### 2. TDD 不只是写测试，而是一种“约束实现顺序”的工程方法

在 AI Coding / Agent / 平台开发面试里，面试官其实很在意两件事：

1. 你是不是**会控范围**；
2. 你是不是**能证明自己写出来的功能真的成立**。

TDD（`Test-Driven Development`，测试驱动开发）之所以容易加分，是因为它天然回答了这两个问题。

它的核心不是“测试写得多”，而是：

> **先写一个失败的测试，用测试明确需求边界；再只写最小实现让测试通过；最后再重构。**

所以面试里讲 TDD，不要讲成“我喜欢先写单测”，而要讲成：

> **我用 TDD 来拆需求、控范围、防回归、证明功能成立。**

---

### 3. 你可以主动讲的 4 个加分点（带例子）

#### 加分点 1：它帮助我拆需求

你可以这样说：

> TDD 对我最大的帮助之一，是它强迫我先把需求拆成一个个可验证的行为，而不是上来就写一大段实现。比如做一个 Agent Compiler、Validator 或 Workflow Engine 时，我不会一口气把整条链路都写完，而是先定义一个最小行为：输入一段 DSL，能不能先编译通过；然后再测运行时校验；再测边界条件；最后再测完整链路。这样需求会被拆成很多清晰的小步。

**更好理解的例子：**

假设你要做一个“把工作流 DSL 编译成执行计划”的模块，不要一开始就写完整编译器，而是按 TDD 拆成：

1. 先写测试：只有一个节点的 DSL 能不能被正确解析；
2. 再写测试：两个节点之间的依赖能不能被正确识别；
3. 再写测试：遇到不存在的依赖节点能不能抛出错误；
4. 最后再写测试：完整 workflow 能不能编译成 DAG。

这样你不是在“硬写编译器”，而是在一步步把能力长出来。

为什么这段话加分：

- 面试官会感觉你**不是“想到哪写到哪”**；
- 你有能力把复杂系统拆成最小可验证单元；
- 这正是 AI Coding 场景里很重要的能力，因为很多任务本身就是“长链路、多状态、容易失控”。

---

#### 加分点 2：它帮助我控制范围，避免一开始就过度设计

你可以这样说：

> 我会先让测试定义当前阶段真正需要的最小能力，然后实现刚好能通过这个测试的代码，而不是一开始就把未来可能用到的抽象、插件机制、配置中心、并行调度全设计进去。这样能显著降低过度设计的风险。

**更好理解的例子：**

比如你在做 Tool Router（工具路由器），第一版目标只是：

- 用户问“查订单”，系统走 `order_tool`
- 用户问“查天气”，系统走 `weather_tool`

那你第一版只需要：

- 一个最简单的分类规则；
- 两个测试；
- 一个最小 router 实现。

这时就**不要**一上来加：

- 多模型路由；
- 置信度重排；
- 动态插件热更新；
- 多租户策略隔离。

TDD 会逼着你只实现“当前测试要求的最小能力”，这就是它对**控范围**最直接的帮助。

---

#### 加分点 3：它帮助我避免回归

你可以这样说：

> 在 AI Coding 场景里，很多系统都不是一次性代码，而是会不断迭代，比如我后面可能还要继续改 compiler、validator、tool router、prompt parser 或 state machine。如果前面没有测试兜底，后续改动特别容易把旧能力悄悄破坏掉。TDD 的好处就是一开始就把关键行为固化成测试，后面每次改动都能快速发现回归问题。

**更好理解的例子：**

比如你后面为了支持新工具，把 tool schema 从：

```json
{ "name": "search", "query": "xxx" }
```

改成：

```json
{ "tool_name": "search", "args": { "query": "xxx" } }
```

如果没有测试，很可能老工具全挂了你都没注意到。  
如果前面已经有：

- schema 校验测试
- 历史 tool call 兼容测试
- 错误输入测试

那你一改结构，回归立刻就能暴露出来。

---

#### 加分点 4：它帮助我证明功能真的成立

你可以直接说：

> TDD 还有一个很重要的价值是，它让我能证明这个功能不是“我觉得对”，而是“测试先失败过，再被我修到通过”。我会刻意强调这个过程，因为只有先看到失败，才能证明这个测试真的测到了目标行为，而不是写了个永远为绿的摆设测试。

**更好理解的例子：**

比如你说“我的 validator 可以拦截非法 tool 参数”，那你最好能讲出这个过程：

1. 先写一个失败测试：缺少必填参数 `query` 时，应该返回错误；
2. 跑测试，看到它先失败；
3. 再实现 validator；
4. 再跑测试，看到它变绿。

这样你的表达就从：

- “我觉得这个功能是对的”

变成：

- “我用先失败后通过的测试证明这个功能是成立的”

这会让面试官更信服。

---

### 4. 在 AI Coding 场景里，TDD 特别适合哪些模块？

#### 1）Compiler / Parser / DSL 解释器

比如：

- 把自然语言或 DSL 编译成执行计划；
- 把 YAML / JSON workflow 编译成内部 DAG；
- 把技能定义编译成 tool schema。

**举例：**

```yaml
steps:
  - id: search
    tool: web_search
  - id: summarize
    depends_on: [search]
    tool: llm_summary
```

你可以先写测试验证：

- `search` 节点能否被识别；
- `summarize` 是否正确依赖 `search`；
- 如果 `depends_on` 指向不存在节点，是否报错。

这类模块非常适合 TDD，因为：

- 输入输出清晰；
- 规则边界明确；
- 适合从最小 case 一点点长出来。

#### 2）Validator / Guardrail / Schema 校验器

比如：

- 校验 tool 参数是否合法；
- 校验输出 JSON 是否符合 schema；
- 校验 prompt 渲染结果是否缺字段。

**举例：**

你定义一个 tool schema：

```json
{
  "name": "search",
  "required": ["query"]
}
```

那就可以先写测试：

- 带 `query` 时通过；
- 缺 `query` 时失败；
- `query` 为空字符串时失败。

这种模块特别适合先写失败测试，因为测试本身就能天然覆盖大量边界条件。

#### 3）Agent 状态机 / Workflow Engine

比如：

- `INIT -> NEED_CLARIFY -> TOOL_EXECUTING -> DONE` 的状态流转；
- 某些状态不允许跳转；
- 某些超时要进入 fallback 状态。

**举例：**

可以先写测试：

- `INIT` 状态下允许进入 `NEED_CLARIFY`；
- `DONE` 状态不允许再回到 `TOOL_EXECUTING`；
- 工具超时后应进入 `FAILED` 或 `FALLBACK`。

这类逻辑如果不用测试先卡住，很容易越写越乱。

#### 4）Tool Router / Planner 的规则层

虽然大模型本身不太适合做传统 TDD，但**规则化的外围层**非常适合：

- 什么情况下走 RAG；
- 什么情况下澄清；
- 哪些工具允许并行；
- 哪些工具必须人工确认。

**举例：**

- 用户问题缺少关键槽位时，应该进入澄清分支；
- 高风险操作（删库、发消息、支付）必须走人工确认；
- 查询类工具可以并行，写操作工具不能并行。

这些判断逻辑本质上都是可测试的。

### 5. 面试里怎么把 TDD 和 AI Coding 结合起来，形成加分表达？

下面这段你可以直接讲，比较像成熟候选人的说法：

> 我觉得 TDD 在 AI Coding 场景里特别有价值，因为 AI 系统表面上看很“智能”，但真正落地时，很多核心模块其实仍然是确定性工程，比如 compiler、validator、状态机、tool schema 校验、路由规则和 fallback 逻辑。这些地方如果先写测试，再写最小实现，会比先堆完整链路更稳。它一方面能帮我拆需求、控范围，另一方面也能在后续频繁迭代 prompt、工具和编排逻辑时防止回归。所以我会优先把“确定性部分”做成测试驱动，把模型本身的不确定部分交给 eval 和 trace 去兜底。

这段话的加分点在于：

- 你没有把 TDD 生搬硬套到“大模型输出本身”；
- 你准确地区分了：
  - **确定性模块** 用 TDD；
  - **非确定性模型行为** 用 eval、trace、人工验收；
- 这会让面试官觉得你对 AI 工程化理解很成熟。

---

### 6. 一个更强的回答：TDD + Eval 的组合思维

这是你在 AI 岗位面试里可以拉开差距的地方。

你可以这样说：

> 我不会把所有 AI 系统问题都用 TDD 解决，因为模型生成本身存在概率性，不适合照搬传统单测思路。我的做法是：对确定性模块，比如 parser、validator、router、state machine，用 TDD 去保证行为正确；对非确定性的模型输出，用离线 eval、线上 trace、bad case 回灌和人工抽样去做质量控制。也就是说，我会把 TDD 和 Eval 分工使用，而不是混在一起。

这段话非常加分，因为它体现你知道：

- TDD 不是万能；
- Eval 也不是万能；
- 真正成熟的是**把两种验证机制分层组合**。

一句话总结：

> **确定性模块靠 TDD，概率性行为靠 Eval。**

---

### 7. 面试里你可以直接背的 TDD 加分版表达

#### 简洁版

> TDD 对我最大的价值是四点：第一，它帮助我拆需求；第二，它帮助我控制范围，只实现当前测试要求的最小能力，避免过度设计；第三，它帮助我避免回归，后面继续改 compiler、validator 或 router 时，老测试会一直兜底；第四，它帮助我证明功能是真的成立，不是我“觉得代码对”，而是测试先失败过，再被我修到通过。

#### AI Coding 结合版

> 在 AI Coding 场景里，我会优先把确定性模块做成 TDD，比如 parser、validator、tool schema 校验、状态机和路由规则，因为这些地方最适合测试先行。对于模型本身的不确定输出，我不会强行套传统单测，而是配合 eval、trace 和 bad case 回放去做质量控制。这样整体系统既有工程可控性，又能适应模型的不确定性。

#### 面试官最容易给好印象的一版

> 我理解 TDD 的价值不只是“先写测试”，而是用测试来定义需求边界、限制实现范围、持续防回归，并证明功能真实成立。在 AI Coding 里，这种方法特别适合处理那些确定性模块，因为 AI 系统最怕的不是代码写不出来，而是边界慢慢失控、修改后悄悄回归。所以我会把 TDD 当成一种工程约束手段，而不是一个仪式化动作。

## 五、你可以直接背的“高分总结话术”

### 1. 一句话讲 Agent 开发

> Agent 开发不是简单把大模型接到业务里，而是在传统后端之上增加一层任务规划、工具调用、记忆管理和质量评估的智能编排能力。

### 2. 一句话讲 Agent 和传统后端差异

> 传统后端解决确定性流程工程化，Agent 开发解决不确定性任务执行工程化。

### 3. 一句话讲 RAG 和 Agent

> RAG 是 Agent 的知识获取组件，Agent 是围绕目标做决策和执行的总系统。

### 4. 一句话讲 MCP / Skill / Function Call

> MCP 是能力接入协议，Function Call 是具体调用动作，Skill 是高层任务方法论 / 工作流模板。

### 5. 一句话讲 Agent 岗为什么仍然要后端基础

> 因为 Agent 上层再智能，底下仍然要落到服务调用、缓存、数据库、并发、观测、安全和发布治理这些传统工程能力上。

## 七、最后的准备建议

如果你准备的是 **Agent 开发 / AI 应用开发 / AI Coding 后端**，我建议按下面优先级准备：

1. **先打牢传统后端基础**：Java/Go、MySQL、Redis、MQ、并发、网络；

如果面试官问得很深，真正拉开差距的通常不是“你知道哪些名词”，而是：
