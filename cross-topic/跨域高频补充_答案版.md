## 1. 文件唯一标识常见有哪些？CRC32、MD5、SHA-1、SHA-256 分别怎么理解？

**答案：**

先说结论：

- `CRC32`：更偏**错误检测**
- `MD5 / SHA-1 / SHA-256`：更偏**哈希摘要**
- 真要安全场景，优先 `SHA-256`
- 真要高性能校验或分桶，常见会用 `CRC32 / xxHash`

### `CRC32`

- 本质是循环冗余校验码
- 输出是 `32 bit`
- 特点是计算快、适合发现传输中的随机错误
- 不适合安全场景，因为太容易碰撞，也不抗篡改

### `MD5`

- 输出 `128 bit`
- 是一种哈希摘要算法
- 现在已经不安全，不适合签名、密码存储等安全用途
- 但在一些历史系统里仍会看到它被用于“文件去重”或“快速校验”

### `SHA-1`

- 输出 `160 bit`
- 比 MD5 更强，但也已经不再推荐用于安全场景

### `SHA-256`

- 输出 `256 bit`
- 目前仍是主流安全哈希之一
- 适合完整性校验、签名体系、内容寻址等更严肃场景

**推荐回答：**

> 如果只是做高性能错误检测，比如网络传输、分片校验，我会想到 CRC32；如果是一般性的内容摘要，可以想到 MD5 或 SHA 系列；但如果是安全场景，比如签名、抗篡改、抗碰撞，就应该优先用 SHA-256，而不是 MD5。

## 2. TCP 用的是 CRC32 吗？为什么不是？

**答案：**

- TCP/IP 协议栈里常见的是**校验和（checksum）**，不是直接用 `CRC32`
- TCP 头部和数据部分会计算一个 `16 bit` 的校验和
- 它的目标是：
  - 发现传输过程中的比特错误
  - 成本低，易于在协议栈里实现

### TCP 校验和和 CRC32 的区别

#### TCP 校验和

- 更偏轻量级错误检测
- 通常用反码求和
- 计算简单，适合集成到协议处理中

#### CRC32

- 更偏强一点的链路层错误检测
- 常见于以太网帧的 `FCS`
- 更擅长检测某些模式的突发错误

所以可以这样理解：

> **底层链路层经常用 CRC32 这种强一些的错误检测，上层 TCP 则用自己的 16 位校验和。**

**推荐回答：**

> TCP 不是直接用 CRC32。TCP/IP 协议里更常见的是 16 位校验和，主要用于检测传输中的位错误；而 CRC32 更常见于数据链路层，比如以太网帧校验。可以理解成：链路层和传输层各自有不同的错误检测机制，互相补充。

## 3. MD5 是哈希吗？碰撞怎么理解？常见解决方法是什么？

**答案：**

- 是，`MD5` 是一种哈希摘要算法
- 但它已经不适合安全场景

### 什么是碰撞

碰撞指的是：

> **两个不同输入，得到同一个哈希结果。**

这件事在理论上一定可能发生，因为：

- 输入空间无限大
- 输出空间有限

### 碰撞分两类理解

#### 1）自然碰撞

- 纯粹因为概率问题出现
- 在足够大的输入规模下迟早会发生

#### 2）人为构造碰撞

- 攻击者故意制造两个不同输入得到同样哈希
- 这就是安全问题

MD5 最大的问题是：**人为构造碰撞已经不够难了。**

### 面试里如果问“碰撞怎么解决”

要分场景回答。

#### 如果是安全场景

- 不要继续用 MD5
- 改用 `SHA-256` 或更安全的方案

#### 如果是工程去重 / 哈希表场景

- 允许哈希先做快速筛选
- 再做二次校验

例如文件去重：

1. 先比较 `MD5`
2. 如果 MD5 一样，再比较：
   - 文件大小
   - 甚至直接比内容

#### 如果是哈希表

- 用链地址法、开放寻址法等冲突解决机制

**推荐回答：**

> MD5 是哈希摘要算法，但它已经不适合安全场景。碰撞就是两个不同输入得到相同哈希值，这在理论上一定可能发生。工程上要看场景：如果是安全问题，就不要再用 MD5，改用 SHA-256；如果是文件去重，可以先用哈希做快速筛选，再做二次校验；如果是哈希表，则通过链地址法或开放寻址法处理冲突。

## 4. 倒排索引和 B+ 树有什么区别？为什么实际架构里 MySQL 和 ES 经常一起用？

**答案：**

先说一句最重要的话：

> **B+ 树更适合结构化数据的精确查询、范围查询和排序；倒排索引更适合全文检索。**

### B+ 树

- 常见于 MySQL / InnoDB
- 适合：
  - 等值查询
  - 范围查询
  - 排序
  - 结构化字段过滤

### 倒排索引

- 常见于 ES / Lucene
- 核心是：

```text
词 -> 出现在哪些文档
```

- 适合：
  - 全文检索
  - 关键词匹配
  - 相关性召回

### 为什么两者经常一起用

因为它们解决的是不同问题：

- `MySQL`：存业务真数据，保证事务、一致性、精确查询
- `ES`：负责搜索和检索体验

**推荐回答：**

> 倒排索引和 B+ 树不是谁替代谁，而是各自服务不同查询模式。B+ 树更适合结构化数据的精确查、范围查和排序，所以 MySQL 用它；倒排索引更适合根据词快速召回文档，所以 ES 用它。真实架构里通常是 MySQL 存数据，ES 做搜索，两者配合使用。

## 5. Buffer Pool 和 OS 页缓存（Page Cache）有什么区别？

**答案：**

这题很容易混，面试里一定要分开。

### Buffer Pool

- 是 **InnoDB 自己管理的缓存**
- 主要缓存：
  - 数据页
  - 索引页
  - 脏页
- 目标是：
  - 加速数据库页访问
  - 配合 redo log / 刷盘机制工作

### OS 页缓存（Page Cache）

- 是 **操作系统内核管理的文件缓存**
- 更通用，不只服务 MySQL
- 普通文件读写通常都会先经过 Page Cache

### 为什么很多人会把两者混在一起

因为它们看起来都像“内存缓存”，而且都和磁盘 IO 有关。

但核心区别是：

- Buffer Pool：数据库内部缓存
- Page Cache：操作系统文件缓存

### 为什么 MySQL / InnoDB 常提 `O_DIRECT`

因为如果数据库自己已经有 Buffer Pool，再让 OS 用 Page Cache 缓一次，就可能出现“双重缓存”。

所以很多生产环境会：

- 用 `O_DIRECT`
- 尽量绕过 Page Cache
- 让 InnoDB 主要依赖自己的 Buffer Pool

**推荐回答：**

> Buffer Pool 是 InnoDB 自己管理的页缓存，主要缓存数据页和索引页；Page Cache 是操作系统对文件 IO 的通用缓存。两者都能减少磁盘访问，但层次不同。生产里常提 O_DIRECT，就是为了尽量绕开 OS 页缓存，避免和 InnoDB 的 Buffer Pool 形成双重缓存。

## 6. 哈希在后端系统里常见有哪些应用？

**答案：**

### 1）分桶 / 路由

- Kafka 按 key 路由分区
- Redis Cluster 按 key 映射槽位

### 2）快速查找

- 哈希表 / HashMap
- Redis Hash / dict

### 3）去重与内容摘要

- 文件去重
- 内容指纹

### 4）错误检测

- CRC32 做校验

### 5）一致性哈希

- 负载均衡
- 缓存路由

**一句话记忆：**

> 哈希最常见的价值就是：快速定位、均匀分布、内容摘要和错误检测。
