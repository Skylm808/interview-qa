## 场景题 / 系统设计

### 1. 高并发秒杀怎么设计？

**面试官真正想听什么：**

- 你能不能先抓住目标，而不是上来就背 Redis 锁
- 你知不知道秒杀的核心矛盾是“瞬时高流量”和“极少库存”
- 你会不会讲削峰、库存控制、异步化、补偿闭环

**先定目标：**

- 不能超卖
- 主链路不能被打垮
- 用户结果要尽快返回
- 失败链路要能收口，不能把库存卡死

**典型链路：**

1. 活动页静态化，尽量把流量挡在后端之前
2. 入口做限流、验证码、风控、黑名单、用户资格预校验
3. 库存不直接查数据库，而是先在 Redis 里做原子预扣
4. 预扣成功后写 MQ，异步创建订单
5. 订单支付成功再最终确认库存，未支付超时则回补库存
6. 全链路要有幂等和补偿任务

**为什么这样设计：**

- 如果每个请求都直接打数据库，数据库一定先挂
- Redis 预扣的价值不只是快，更关键是把“库存判断”放到一个高并发可承受的位置
- MQ 的价值不只是解耦，更重要的是削峰，把突发流量摊平

**关键难点：**

- Redis 预扣成功但 MQ 投递失败怎么办
- 订单创建成功但支付超时怎么办
- 重试导致重复下单怎么办
- 热点商品一个 key 被打爆怎么办

**对应处理：**

- 预扣和消息投递之间要有失败补偿，比如本地消息表、事务消息或重试队列
- 下单逻辑必须幂等，通常用用户 ID + 商品 ID 作为业务唯一约束
- 超时未支付要异步回补库存
- 对热点商品可以做本地缓存、分片库存、队列前置隔离

**面试里推荐这样答：**

> 秒杀系统我会先围绕四个目标设计：不超卖、不把主链路打垮、尽快返回用户结果、失败可补偿。整体链路上，入口先做静态化和限流，库存层通过 Redis 原子预扣避免直接打数据库，预扣成功后写 MQ 异步创建订单，再通过支付超时回补、幂等约束和补偿任务保证最终一致。这个题的关键不在于某个组件，而在于削峰、预扣、异步、补偿这四件事形成闭环。

### 2. 视频点赞系统怎么设计？

**面试官真正想听什么：**

- 你能不能把“用户行为”与“统计计数”拆开
- 你知不知道点赞这种场景的难点不在写库，而在热点和一致性
- 你会不会讲取消点赞、幂等、热点隔离
- 热点隔离：热点 key 隔离不一定只等于“单独放一个 Redis 节点”，本质是把超热点流量和普通流量拆开治理。常见做法包括独立 Redis 实例、独立热点分片、应用层特殊路由，以及在上层增加本地缓存，避免热点 key 把整套缓存系统打爆。

**先定语义：**

- 是否允许重复点赞
- 是否支持取消点赞
- 点赞数是否必须实时精确
- 用户端看到的“已点赞状态”和总数是否允许短暂不一致

**常见拆法：**

1. 用户行为链路  
   负责回答“这个用户到底点没点过赞”
2. 总数统计链路  
   负责回答“这条视频当前有多少赞”

**典型实现：**

- 用户行为可以落 Redis Set / Bitmap / DB 明细表 / MQ 异步落库
- 总数统计可以先写缓存，再异步汇总刷库
- 热门视频的总数不能只靠单 key 原地自增，容易打爆

**为什么不能简单地“点赞就 update count=count+1”：**

- 高频写直接打 DB，热点视频很容易形成行锁竞争
- 同时还要解决取消点赞、重复点击、重试幂等
- 点赞状态和点赞总数从本质上是两类数据

**关键难点：**

- 重复点赞怎么防
- 取消点赞怎么保证状态一致
- 热门视频单 key 热点怎么扛
- 总数展示和明细状态短时间不一致能否接受

**常见做法：**

- 用业务唯一键或状态机防止重复点赞
- 行为明细和计数异步解耦
- 热门视频做分片计数，最后聚合
- 前台展示允许短时间最终一致

**面试里推荐这样答：**

> 点赞系统我会拆成两条链路：一条是用户行为链路，解决‘这个用户是否点过赞’；另一条是计数链路，解决‘这条内容现在有多少赞’。因为这两个问题的关注点不一样，行为链路更强调状态正确性和防重，计数链路更强调高并发下的吞吐和热点抗压。
> 
> 行为链路上，我一般会先确定唯一关系，也就是 (user_id, target_id) 这一对。落地时可以根据规模选型：如果要强一致和可追溯，可以落 MySQL 明细表，并对 (user_id, target_id) 建唯一索引；如果更关注高性能判断用户有没有点过赞，可以在 Redis 里放 Set 或 Bitmap。这样用户点进详情页时，就能快速判断当前用户是否已点赞。
> 
> 计数链路上，我不会每次都直接 update DB 的点赞数字，因为热点内容会把数据库行打成瓶颈。更常见做法是先把计数写到 Redis，比如做原子自增；再通过异步任务或者 MQ，把增量批量刷回数据库。这样主链路快，数据库压力也更平滑。
> 
> 如果是热门视频，还要考虑热点 key 问题。单个 Redis key 虽然比数据库强很多，但极热点场景下仍可能成为瓶颈，所以可以做分片计数，比如把一个视频的点赞计数拆成多个 shard key，写入时按用户或随机路由到不同分片，读取时再聚合。这样可以把单 key 压力打散。
> 
> 另外，点赞和取消点赞不能只看计数，还要保证行为和计数一致。比如用户重复点赞，不能把计数重复加一；取消点赞时，也不能减成负数。所以行为侧必须有幂等判断，可以先判断用户状态，再决定是否更新计数。工程上常见做法是：先原子更新行为状态，再异步发计数变更事件，消费端做幂等汇总。
> 
> 如果面试官继续追问异常场景，我会补两点：第一，如果 Redis 计数更新成功但 MQ 投递失败，要有补偿机制，比如本地消息表或者重试；第二，如果缓存计数和数据库计数短暂不一致，我会接受最终一致，因为点赞数本身通常允许短暂延迟，只要用户自己的点赞状态尽量实时准确即可。

### 3. 群聊 / 聊天室怎么设计？

**面试官真正想听什么：**

- 你知不知道聊天系统的核心不是“发消息”，而是长连接、路由、顺序、离线消息
- 你会不会区分单聊、群聊、广播场景
- 你能不能讲清消息链路

**先定需求：**

- 是单聊还是群聊
- 是否要求强顺序
- 是否有离线消息
- 是否需要未读数、撤回、禁言、审核
- 延迟目标是多少

**核心链路建议这样讲：**

1. 长连接接入层  
   用户通过 WebSocket / 长连接网关接入
2. 连接管理层  
   维护用户 ID 和连接的映射关系
3. 路由层  
   按会话 ID 或群 ID 做路由，尽量让同一会话落到同一逻辑分片
4. 存储层  
   在线消息实时推送，离线消息持久化
5. 消息扩展能力  
   未读数、撤回、禁言、审核、历史拉取

**为什么按群 ID / 会话 ID 路由：**

- 主要是为了降低同一会话的乱序处理成本
- 如果消息到处乱飞，顺序、一致性和状态同步都会变复杂

**关键难点：**

- 海量连接如何维护
- 消息顺序如何保证
- 用户离线后如何补消息
- 未读数怎么统计
- 敏感词和审核链路放在哪

**一个更像后端的回答方式：**

- 在线用户走实时推送
- 不在线则写离线消息表或消息队列
- 同会话消息尽量经过同一路由分片
- 历史消息和实时推送分开考虑，不要混成一套

**面试里推荐这样答：**

> 聊天系统我会拆成长连接网关、连接管理、消息路由、消息存储和离线投递五层。核心难点不是能不能把消息发出去，而是顺序、离线补投递、未读数和海量连接管理。工程上我会按会话 ID 或群 ID 做路由，尽量把同一会话消息收敛到同一路径处理，这样更容易控制顺序和状态一致性。

### 4. 大文件上传怎么设计？

**面试官真正想听什么：**

- 你知不知道大文件上传不是“一个接口传完”那么简单
- 你能不能讲分片、断点续传、校验、合并和失败恢复
- 你有没有安全意识

**先定目标：**

- 上传过程可恢复
- 网络波动不需要整文件重传
- 服务端能感知上传进度
- 合并过程安全且幂等

**典型设计：**

1. 客户端把文件切成多个 chunk
2. 每个 chunk 带文件 ID、分片号、校验值上传
3. 服务端记录分片上传状态
4. 所有分片到齐后触发合并
5. 合并完成后做二次校验和异步后处理

**为什么要断点续传：**

- 大文件最怕中途中断
- 如果每次都重传全量，用户体验和带宽成本都很差

**关键难点：**

- 分片重复上传怎么处理
- 合并操作如何做到幂等
- 上传中断后怎么快速恢复
- 恶意文件、超大文件、病毒文件怎么管控

**工程补充点：**

- 上传状态最好落 DB 或稳定存储，不要只放内存
- 合并操作建议异步化
- 需要配合文件类型校验、大小限制、权限校验

**面试里推荐这样答：**

> 大文件上传我会设计成分片上传加断点续传。客户端按 chunk 上传，服务端记录每个 chunk 的状态和校验值，全部完成后异步合并。这样既能处理网络中断，也能避免整文件重传。这个题的关键点不只是分片本身，而是上传状态管理、合并幂等、失败恢复和安全校验。

### 5. 预约通知 / 延迟通知怎么设计？

**面试官真正想听什么：**

- 你会不会把“定时触发”和“通知发送”拆开
- 你知不知道通知系统的重点是可靠性，不是把消息发出去就算完
- 你能不能讲幂等、重试、取消预约

**先定目标：**

- 到点尽量准时触发
- 重复通知不能发生
- 失败后能重试
- 用户取消预约后不能误发

**核心链路：**

1. 用户预约时，先落预约记录
2. 同时把触发时间交给延迟任务系统
3. 到点后任务系统把事件投递给通知服务
4. 通知服务根据渠道发送站内信 / Push / 短信
5. 发送结果回写，失败重试，超过阈值进入死信或补偿

**延迟任务系统怎么选：**

- 量不大可以定时扫描
- 吞吐更高可以用时间轮
- 已有 MQ 体系时可用延迟队列

**关键难点：**

- 用户取消预约后任务如何失效
- 通知服务超时重试会不会重复发
- 多渠道通知如何保证状态可追踪

**工程回答要补：**

- 每条通知任务都要有唯一 ID，发送逻辑必须幂等
- “预约记录”和“任务状态”要能关联，方便取消和补偿

**面试里推荐这样答：**

> 预约通知我会拆成四段：预约存储、延迟触发、通知执行、失败补偿。用户发起预约时，先落一条预约记录，里面至少包含用户 ID、业务对象 ID、预约触发时间、通知渠道、任务状态和唯一任务 ID；同时把这个任务交给延迟任务系统，比如定时扫描、时间轮，或者基于 MQ 的延迟队列。到触发时间后，延迟系统不直接发通知，而是只负责把“该任务已到期”这个事件投递给通知服务。通知服务收到后，会先根据任务 ID 查状态，做一次幂等校验，确认这个任务没有被取消、没有发送成功过，然后再调用具体渠道去发站内信、Push 或短信。发送成功后回写发送结果和状态；如果发送失败，就按重试策略继续重试，超过阈值进入死信队列或补偿表，后面由人工或补偿任务处理。这样设计的关键点是把“什么时候触发”和“怎么发送”彻底解耦：前者解决定时问题，后者解决渠道可靠性问题。对于用户取消预约，我会把预约记录改成已取消，并让后续执行时再做一次状态校验，这样即使延迟任务已经出队，也不会误发。对于下游超时或重复投递，我会依赖任务唯一 ID 和发送幂等来保证‘最多只成功一次’，并把整个链路的状态都落库，保证后续可查、可重试、可补偿

### 6. 缓存一致性怎么保证？

**面试官真正想听什么：**

- 你知不知道这题没有银弹
- 你会不会先谈一致性目标，再谈方案
- 你能不能讲出极端情况下的补偿

**第一句话就要说：**

- 绝大多数互联网业务追求的是**最终一致**，不是强一致

**为什么这点重要：**

- 如果你一上来说“保证绝对一致”，面试官通常会继续追到你答不下去
- 正确做法是先问清楚业务能接受多大延迟、读旧值后果多严重

**常见方案：**

1. `cache-aside`
   - 读：先查缓存，miss 再查 DB，再回填
   - 写：先写 DB，再删缓存
2. 延迟双删
   - 适合处理并发读写窗口问题
3. MQ 异步补偿
4. 订阅 binlog / CDC 驱动缓存更新

**关键难点：**

- 删缓存失败怎么办
- 并发读写时会不会把旧值重新写回缓存
- 缓存和 DB 分别成功 / 失败的组合场景怎么收口

**更稳的回答方式：**

- 主方案是 cache-aside
- 再根据业务写频率、一致性要求和链路复杂度决定是否加延迟双删、MQ 补偿或 binlog 驱动
- 最后一定补一句：TTL 是兜底，不是主策略

**面试里推荐这样答：**

> 缓存一致性这题我会先确认业务要的是强一致还是最终一致。大部分互联网业务我会优先用 cache-aside，也就是先写库再删缓存；如果并发读写窗口比较明显，再加延迟双删、消息补偿或者 binlog 驱动更新。这个题的重点不是背某一个方案，而是你能不能说清楚一致性要求、失败场景和补偿策略。

### 7. 热点 key 怎么治理？

**面试官真正想听什么：**

- 你是不是知道热点 key 不只是 Redis 问题，而是整条链路问题
- 你会不会把热点识别、流量分散、缓存层次都讲出来

**热点 key 的本质：**

- 某个 key 的访问量远高于其他 key，导致单点成为瓶颈

**治理手段可以分四类讲：**

1. 提前识别  
   监控热点 key、热榜数据、活动流量
2. 多级缓存  
   本地缓存 + Redis + DB
3. 流量分散  
   key 分片、热点隔离、读副本
4. 流量治理  
   限流、降级、预热

**为什么本地缓存常常有效：**

- 很多热点流量其实是同一批数据被反复读取
- 如果每次都打 Redis，Redis 本身就会变成热点中心

**关键难点：**

- 本地缓存带来一致性问题
- key 分片后聚合更复杂
- 热点变化快，预热可能跟不上

**面试里推荐这样答：**

> 热点 key 治理我不会只盯 Redis。更完整的思路是：先通过监控识别热点，再用本地缓存和多级缓存兜住最热流量，必要时做 key 分片和热点隔离，最后配合限流和降级守住主链路。这个题的关键是把“热点识别、流量分散、缓存层次、失败兜底”连成一套。**

### 8. 多机房 / 多实例流量怎么均匀分配？

**面试官真正想听什么：**

- 你会不会区分“实例负载均衡”和“跨机房调度”
- 你知不知道“绝对均匀”不一定最优
- 你能不能讲出健康检查和故障切流

**先分两个层次：**

1. 单机房内实例调度
2. 多机房之间流量调度

**常见手段：**

- 四层 / 七层负载均衡
- 权重路由
- 就近接入
- 健康检查和自动摘除
- 一致性哈希
- 会话亲和

**为什么不一定追求绝对均匀：**

- 如果业务有本地缓存、会话状态或连接状态，过于随机的均匀分布反而会降低命中率
- 所以很多场景要在“均匀分布”和“局部粘性”之间平衡

**关键难点：**

- 机房故障切流时如何避免雪崩
- 跨机房延迟和数据一致性怎么权衡
- 有状态连接如何迁移

**面试里推荐这样答：**

> 多机房流量分配我会先分成实例级和机房级两层来看。实例层更关注负载均衡和健康检查，机房层更关注就近接入、权重调度和故障切流。如果业务有本地缓存或长连接，我不会一味追求绝对随机均匀，而是会适度保留会话亲和，在命中率和负载均衡之间做取舍。

### 9. 服务超时、网络波动、重试风暴怎么处理？

**面试官真正想听什么：**

- 你知不知道重试不是免费午餐
- 你会不会把超时、幂等、退避、熔断一起讲
- 你有没有“避免故障放大”的思维

**先讲原则：**

- 重试是恢复手段，不是默认加速器
- 没有边界的重试，往往会把小故障放大成大故障

**一套完整方案应该包括：**

1. 明确的超时控制  
   每一层调用都有边界
2. 幂等设计  
   不然重试本身就会造成副作用
3. 指数退避和抖动  
   避免一窝蜂重试
4. 熔断  
   下游明显异常时，不再持续压测它
5. 降级  
   关键功能保主链路，非关键功能先让步
6. 限流  
   防止系统在故障期间被进一步打穿

**关键难点：**

- 超时值设太小，误伤正常请求
- 超时值设太大，线程 / goroutine 被长时间占住
- 没幂等时重试会重复扣款、重复下单、重复发消息

**面试里推荐这样答：**

> 面对服务超时和网络波动，我不会只说“加重试”。更完整的思路是：先给每层调用设清晰超时边界，再保证请求幂等，然后用指数退避重试、熔断、限流和降级一起控制故障扩散。重试只能在“可重试 + 幂等 + 有边界”的前提下使用，否则它本身就会变成故障放大器。

### 10. 登录业务怎么保证通信安全和数据安全？

**面试官真正想听什么：**

- 你会不会区分通信安全、身份认证、数据存储、风控审计
- 你会不会答成“上 HTTPS 就完了”

**推荐拆成四层：**

1. 通信安全  
   HTTPS / TLS，避免窃听和中间人攻击
2. 身份认证  
   token / session / 签名 / 多因素认证
3. 数据安全  
   密码加盐强哈希，敏感字段加密与脱敏
4. 风控与审计  
   验证码、限流、设备指纹、异地登录提醒、审计日志

**密码存储一定要答到位：**

- 绝不能明文存储
- 也不建议简单做 `md5(password)`
- 更稳的做法是使用带盐的强哈希算法，比如：
  - `bcrypt`
  - `scrypt`
  - `Argon2`

**关键难点：**

- 登录接口本身就是暴露面最大的入口之一
- 除了密码，还要关注 token 泄露、撞库、暴力破解、会话劫持

**面试里推荐这样答：**

> 登录安全我会分成通信安全和数据安全两层来讲。通信层通过 HTTPS 保证链路不被窃听和篡改；数据层里，密码必须用带盐的强哈希算法存储，不能明文也不能简单 md5。除此之外，还要配合限流、验证码、设备指纹、异地登录提醒和审计日志。这个题的高分点是把“传输、认证、存储、风控、审计”连成一套，而不是只说一个 HTTPS。

### 11. CAP 理论怎么理解？不同场景下该选 CP 还是 AP？常见中间件有哪些？

**面试官真正想听什么：**

- 你知不知道 CAP 不是背定义，而是讲业务取舍
- 你会不会先说“P 基本不可回避”
- 你能不能把中间件选择和场景联系起来

**先把定义说清楚：**

- `C`：一致性，所有节点看到同一份最新数据
- `A`：可用性，每个请求都能在有限时间内返回
- `P`：分区容错，网络分区时系统仍然要继续工作

**高分点在这句：**

- 在分布式系统里，网络分区不可避免，所以很多时候是在 `P` 必选的前提下，在 `C` 和 `A` 之间做倾向性取舍

**什么时候偏 CP：**

- 配置中心
- 分布式锁
- 选主元信息
- 核心库存、金额、强一致元数据

**常见偏 CP 的中间件：**

- `etcd`
- `ZooKeeper`

**什么时候偏 AP：**

- 缓存
- 推荐
- 点赞、浏览量、非关键统计
- 可接受短时间旧值的读流量

**常见偏 AP 的系统或使用场景：**

- `Redis` 作为缓存时通常更偏 AP
- 各类最终一致消息驱动系统

**面试里推荐这样答：**

> 我理解 CAP 的关键不在于背定义，而在于先承认网络分区不可避免，所以很多系统是在 P 必选的前提下，在一致性和可用性之间做业务取舍。像配置中心、分布式锁、选主元信息更偏 CP；像缓存、推荐、点赞、浏览量这类允许短时旧值的场景更偏 AP。选中间件时，不是看它名义上属于哪一类，而是看业务不能接受什么。

### 12. 项目里并发很大、订单量突然上来时，你会怎么扛？

**面试官真正想听什么：**

- 你会不会先讲瓶颈分析，而不是一上来就说“加机器”
- 你能不能把流量入口、应用层、缓存层、数据库层拆开讲
- 你知不知道“先止血，再扩容，再重构”的思路

**先定目标：**

- 先保证核心链路不挂
- 先保写入成功，再谈复杂功能
- 避免局部热点拖垮整个系统

**回答顺序建议这样讲：**

1. **先定位瓶颈在哪一层**
   
   - 网关扛不住
   - 应用线程池 / goroutine 爆了
   - Redis 热 key
   - MySQL 慢 SQL / 锁竞争
   - MQ 积压

2. **入口先做止血**
   
   - 限流
   - 熔断
   - 降级
   - 非核心功能先关掉

3. **把重操作异步化**
   
   - 下单、发券、通知、统计尽量解耦到 MQ
   - 主链路只保最关键成功结果

4. **热点隔离**
   
   - 热门商品、热门用户、热点配置单独隔离
   - 防止单点热点把整个系统拖垮

5. **数据库层做兜底**
   
   - 补索引
   - 限制大事务
   - 读写分离
   - 必要时做分库分表

**trade-off 要主动补一句：**

- 不是所有问题都该靠数据库硬扛
- 也不是所有问题都能马上靠分库分表解决
- 很多时候第一步是削峰、限流、异步化，而不是直接做大手术

**面试里推荐这样答：**

> 如果订单量突然上来，我不会直接说加机器，而是先按层定位瓶颈：入口、应用、缓存、数据库、消息队列分别看。短期先靠限流、降级、异步化和热点隔离止血，优先保住下单主链路；中期再补数据库索引、优化慢 SQL、拆大事务；如果增长是长期的，再考虑分库分表和更大的架构调整。这个题的关键不只是扩容，而是知道先保什么、先挡什么、先拆什么。

### 13. 如果用户量本身很大，系统要怎么设计得更稳？

**面试官真正想听什么：**

- 你会不会把“高并发”和“高用户量”区分开
- 你知不知道读多写少和写多读多的治理方式不同
- 你会不会讲容量规划和分层设计

**先定问题：**

用户量大，不一定等于每时每刻并发都大。  
所以这题要先拆成：

- 总用户量大
- 在线用户量大
- 瞬时并发大

**系统设计思路：**

1. **分层承压**
   
   - 静态资源走 CDN
   - 热读请求优先走缓存
   - 重写操作尽量异步化

2. **存储分层**
   
   - 热数据和冷数据分开
   - 在线查询链路和历史归档链路分开

3. **服务拆分**
   
   - 核心链路和非核心链路拆开
   - 用户系统、订单系统、搜索系统不要混成一个大服务

4. **容量规划**
   
   - 不是“用户量 1000 万”就结束
   - 要继续估：
     - 日活
     - 峰值 QPS
     - 单用户请求频率
     - 数据增长速度

**高分点：**

- 真正大规模系统不是靠单点参数调优扛出来的
- 而是靠“缓存、异步、隔离、分层、容量规划”一起做

**面试里推荐这样答：**

> 用户量大时，我会先把“总量、在线量、瞬时并发”拆开看，因为它们对应的架构压力不一样。整体设计上会走分层承压：静态资源走 CDN，热点请求走缓存，重操作异步化；数据上做冷热分离，服务上做核心链路和非核心链路拆分。真正稳定的大规模系统，不是靠某一个数据库参数或线程池参数解决，而是靠分层、隔离和容量规划一起兜住。

### 14. 面试官让你讲“系统的数据量、设备量、上传频率、传输路径”，你怎么答更像工程师？

**面试官真正想听什么：**

- 你是不是只会说“数据量很大”
- 你能不能把量级、链路、瓶颈和优化点一起讲

**推荐回答结构：**

1. **先讲量级**
   
   - 日上传文件数
   - 平均文件大小
   - 峰值上传并发
   - 每天新增多少索引数据

2. **再讲链路**
   
   - 客户端上传
   - 服务端接收
   - 对象存储落盘
   - 元数据入库
   - 异步处理
   - 检索索引更新

3. **最后讲瓶颈**
   
   - 上传链路更吃带宽和对象存储
   - 处理链路更吃 CPU、Embedding 服务和 ES 写入
   - 检索链路更吃 ES 查询延迟和模型响应时间

**如果没有真实超大规模数据怎么办？**

- 不要编很夸张的数字
- 可以说当前量级有限，但设计时已经按“上传、处理、检索”三条链路拆开，预留了扩展点

**面试里推荐这样答：**

> 我不会只说“数据量很大”，而是会按量级、链路、瓶颈三层来讲。比如每天多少上传、平均文件大小、峰值并发大概多少；上传后经过对象存储、元数据入库、异步解析、向量化和索引更新；其中上传链路更吃带宽，处理链路更吃 CPU 和外部模型服务，检索链路更吃 ES 和生成延迟。这样讲面试官更容易判断你有没有真实做过容量分析。

### 15. 如果文件非常大、内存有限，你会怎么处理？

**面试官真正想听什么：**

- 你知不知道“不能把整个文件一次性读进内存”
- 你会不会讲分块、流式处理、外部排序这类思路

**核心原则：**

- 不要一次性全量读入内存
- 要按块处理
- 中间结果必要时落磁盘

**常见做法：**

1. **流式读取**
   
   - 一边读一边处理
   - 适合日志解析、文本扫描、哈希计算

2. **分块处理**
   
   - 把大文件拆成多个 chunk
   - 每块单独处理，再汇总结果

3. **外部算法**
   
   - 如果数据量远超内存，就用外部排序、分桶、临时文件归并

**高分点：**

- 你要主动说明：这类题本质上是“用磁盘换内存”
- 不要假装所有问题都能靠内存 map 解决

**面试里推荐这样答：**

> 文件特别大、内存有限时，核心原则就是不要整文件读进内存，而是改成流式读取和分块处理。如果问题需要全局排序或全局聚合，就要接受“用磁盘换内存”，采用外部排序、分桶或归并的思路。这个题考的不是 API，而是你有没有大数据量下的资源约束意识。

### 16. 500G 文件、内存只有 8G，这类海量数据题怎么回答才不慌？

**面试官真正想听什么：**

- 你能不能先判断“内存肯定不够”
- 你会不会讲外部排序、分桶、哈希拆分、归并
- 你能不能先说思路，再看具体题型

**万能回答框架：**

1. **先判断约束**
   
   - 500G 远超 8G，不能全量进内存

2. **再判断题型**
   
   - 如果是排序：外部排序
   - 如果是去重 / 聚合：分桶 + 局部处理
   - 如果是查 TopK：小根堆 + 分块扫描
   - 如果是匹配组合：哈希分片后局部求解

3. **最后讲工程细节**
   
   - 临时文件怎么落盘
   - 分桶规则怎么定
   - 如何保证最终归并正确

**举一个最通用的说法：**

> 这类海量文件题我会先承认“内存一定不够”，所以不能按常规算法直接做，而要转成外部算法。排序就做外部排序，先分块排序再归并；如果是查找或聚合，就先按哈希或范围做分桶，把大问题拆成多个内存可处理的小问题。核心思想是分而治之，用磁盘换内存。  

**面试里推荐这样答：**

> 500G 文件配 8G 内存，这种题第一反应不是想“怎么调参数塞进去”，而是立刻切换到外部算法思路。先按题型分类：排序就外部排序，聚合就分桶，TopK 就分块扫描配堆，匹配组合就先哈希拆分。高分点在于你先明确资源约束，再给出分治和落盘方案，而不是直接写普通内存算法。

### 17. 定时任务怎么设计，才能让面试官觉得你考虑得比较完整？

**面试官真正想听什么：**

- 你会不会把“触发”与“执行”拆开
- 你知不知道多实例部署下不能重复跑
- 你会不会讲幂等、重试和失败补偿

**先定目标：**

- 到点尽量触发
- 多实例下不要重复执行
- 执行失败能重试
- 任务状态可追踪

**一套完整思路可以这样讲：**

1. **任务定义层**
   
   - 任务 ID
   - 执行时间
   - 执行参数
   - 当前状态

2. **调度层**
   
   - 负责判断“什么时候该触发”
   - 可以是定时扫描、时间轮、延迟队列

3. **执行层**
   
   - 真正跑任务逻辑
   - 要做幂等，避免重复执行出副作用

4. **补偿层**
   
   - 失败重试
   - 超过重试次数进告警 / 死信 / 人工处理

**多实例下怎么避免重复跑：**

- 分布式锁
- 数据库状态抢占
- 调度中心分配

但无论哪种方案，都要补一句：

> 即使已经做了互斥，执行逻辑本身最好仍然幂等，因为线上永远可能出现重复触发或补偿重试。

**面试里推荐这样答：**

> 定时任务我会拆成任务定义、调度、执行、补偿四层。调度层负责“什么时候触发”，执行层负责“真正把事做掉”，两者不要耦死。多实例场景下要通过分布式锁、状态抢占或调度中心避免重复跑；同时任务执行本身也要幂等，失败要能重试和告警。这个题的重点不是定时器 API，而是可靠触发、避免重复和失败收口。
