### 1. 如何设计一个秒杀系统？如何限流、削峰？

**答案：**

**系统架构：**

```
                    CDN
                     ↓
客户端 → 负载均衡 → API网关(限流) → 秒杀服务 → Redis → 数据库
                                    ↓
                                 消息队列
                                    ↓
                                 订单服务
```

**核心设计：**

| 阶段      | 策略             |
| ------- | -------------- |
| **前端**  | 按钮置灰、验证码、答题    |
| **网关**  | 限流、IP黑名单       |
| **服务层** | 内存预减库存、队列削峰    |
| **数据库** | Redis预扣库存、异步落库 |

**限流方案：**

```go
// 1. 令牌桶
limiter := rate.NewLimiter(1000, 2000)  // 1000 QPS, 桶容量2000
if !limiter.Allow() {
    return "系统繁忙"
}

// 2. 滑动窗口
func slidingWindowLimit(key string, limit int, window time.Duration) bool {
    now := time.Now().UnixNano()
    pipe := redis.Pipeline()
    pipe.ZRemRangeByScore(key, "0", strconv.FormatInt(now-window.Nanoseconds(), 10))
    pipe.ZAdd(key, redis.Z{Score: float64(now), Member: now})
    pipe.ZCard(key)
    pipe.Expire(key, window)
    results, _ := pipe.Exec()
    count := results[2].(*redis.IntCmd).Val()
    return count <= int64(limit)
}
```

**削峰方案：**

```go

func seckill(userID, productID string) {
    // 1. 参数校验、限流

    // 2. Redis预扣库存

    if stock < 0 {
        redis.Incr("stock:" + productID)  // 回滚

    }

    // 3. 发送到消息队列

// 消费者异步创建订单
```

### 2. 朋友圈系统设计

**答案：**

**表结构设计：**

```sql
-- 用户表
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50)
);

-- 关注关系表
CREATE TABLE follows (
    user_id BIGINT,
    follower_id BIGINT,
    PRIMARY KEY (user_id, follower_id)
);

-- 动态表（Feed）
CREATE TABLE feeds (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    content TEXT,
    created_at DATETIME,
    INDEX idx_user_time (user_id, created_at)
);

-- 时间线表（推模式）
CREATE TABLE timelines (
    user_id BIGINT,
    feed_id BIGINT,
    created_at DATETIME,
    PRIMARY KEY (user_id, created_at, feed_id)
);
```

**推拉模式选择：**

| 模式       | 写放大       | 读放大       | 适用   |
| -------- | --------- | --------- | ---- |
| **推模式**  | 高（写入所有粉丝） | 低         | 粉丝少  |
| **拉模式**  | 低         | 高（聚合关注的人） | 粉丝多  |
| **推拉结合** | 中         | 中         | 实际使用 |

**推拉结合策略：**

```go
func publishFeed(userID string, feed Feed) {
    // 1. 写入动态表
    db.Insert("feeds", feed)

    // 2. 获取粉丝列表
    followers := getFollowers(userID)

    // 3. 活跃粉丝推送，非活跃拉取
    for _, f := range followers {
        if isActiveUser(f) {
            redis.ZAdd("timeline:"+f, feed.CreatedAt, feed.ID)
        }
    }
}

func getTimeline(userID string) []Feed {
    // 1. 从Redis获取推送的
    pushed := redis.ZRevRange("timeline:"+userID, 0, 20)

    // 2. 拉取关注的大V的动态
    bigVFeeds := pullBigVFeeds(userID)

    // 3. 合并排序
    return merge(pushed, bigVFeeds)
}
```

### 4. 多个consumer只让一个消费，如何实现？

**答案：**

**Consumer Group机制（Kafka示例）：**

```
生产者 → Topic(3个分区) → Consumer Group
                          ├─ Consumer1 (处理分区0)
                          ├─ Consumer2 (处理分区1)
                          └─ Consumer3 (处理分区2)
```

**关键点：**

1. 同一Consumer Group中的消费者不会消费同一分区
2. 每个分区只会被组内一个消费者消费
3. 消费者数量不应超过分区数量

**Go实现（使用Sarama）：**

```go
config := sarama.NewConfig()
config.Consumer.Group.Rebalance.Strategy = sarama.BalanceStrategyRoundRobin

group, _ := sarama.NewConsumerGroup(brokers, "my-group", config)

handler := &ConsumerGroupHandler{}
for {
    group.Consume(ctx, []string{"topic"}, handler)
}

type ConsumerGroupHandler struct{}

func (h *ConsumerGroupHandler) ConsumeClaim(session sarama.ConsumerGroupSession, claim sarama.ConsumerGroupClaim) error {
    for msg := range claim.Messages() {
        // 处理消息
        session.MarkMessage(msg, "")
    }
    return nil
}
```

### 5. 如何保证消息不被重复消费？如何保证消息的顺序性？

**答案：**

**防止重复消费（幂等性）：**

| 方案          | 实现               |
| ----------- | ---------------- |
| **数据库唯一键**  | INSERT时利用唯一索引去重  |
| **Redis去重** | SETNX消息ID，已存在则跳过 |
| **业务幂等**    | 状态机，已完成的不再处理     |

```go
func consume(msg Message) error {
    // 1. Redis去重
    ok, _ := redis.SetNX(ctx, "msg:"+msg.ID, "1", time.Hour).Result()
    if !ok {
        return nil // 已处理
    }

    // 2. 处理业务
    if err := process(msg); err != nil {
        redis.Del(ctx, "msg:"+msg.ID)  // 处理失败，删除
        return err
    }

    return nil
}
```

**保证顺序性：**

| 方案       | 说明             |
| -------- | -------------- |
| **单分区**  | 同一分区内有序（牺牲并行度） |
| **业务分区** | 相同业务ID发到同一分区   |
| **版本号**  | 消费时检查版本，乱序则等待  |

```go
// 业务分区
func produce(orderID string, msg []byte) {
    partition := hash(orderID) % partitionCount
    producer.Send(topic, partition, msg)
}

// 版本号机制
func consume(msg Message) error {
    for {
        currentVersion := getVersion(msg.ID)
        if msg.Version == currentVersion + 1 {
            process(msg)
            setVersion(msg.ID, msg.Version)
            return nil
        }
        time.Sleep(100 * time.Millisecond)  // 等待前序消息
    }
}
```

### 6. 分布式事务有哪些解决方案？

**答案：**

| 方案        | 原理                 | 优点   | 缺点      |
| --------- | ------------------ | ---- | ------- |
| **2PC**   | 协调者统一提交/回滚         | 强一致  | 同步阻塞、单点 |
| **3PC**   | 增加PreCommit阶段      | 减少阻塞 | 网络分区问题  |
| **TCC**   | Try-Confirm-Cancel | 业务控制 | 代码侵入大   |
| **Saga**  | 事件驱动补偿             | 长事务  | 隔离性差    |
| **本地消息表** | 消息+定时任务            | 简单可靠 | 延迟      |
| **最终一致性** | 消息队列               | 解耦   | 最终一致    |

**TCC示例：**

```go
// Try阶段：预留资源
func try(orderID string) error {
    // 冻结库存
    db.Exec("UPDATE stock SET frozen = frozen + 1 WHERE product_id = ?")
    // 冻结余额
    db.Exec("UPDATE account SET frozen = frozen + 100 WHERE user_id = ?")
    return nil
}

// Confirm阶段：确认执行
func confirm(orderID string) error {
    db.Exec("UPDATE stock SET quantity = quantity - 1, frozen = frozen - 1")
    db.Exec("UPDATE account SET balance = balance - 100, frozen = frozen - 100")
    return nil
}

// Cancel阶段：取消预留
func cancel(orderID string) error {
    db.Exec("UPDATE stock SET frozen = frozen - 1")
    db.Exec("UPDATE account SET frozen = frozen - 100")
    return nil
}
```

### 7. CAP定理是什么？两阶段提交、三阶段提交的区别？

**答案：**

**CAP定理：**
分布式系统不可能同时满足以下三个特性：

- **C（Consistency）**：一致性，所有节点数据相同
- **A（Availability）**：可用性，每个请求都能响应
- **P（Partition Tolerance）**：分区容错性，网络分区时仍能运行

**由于P必须保证，实际选择是CP或AP：**
| 选择 | 特点 | 系统 |
|------|------|------|
| **CP** | 保证一致性，可能不可用 | ZooKeeper、etcd、Redis Cluster |
| **AP** | 保证可用性，最终一致 | Cassandra、Eureka |

**2PC vs 3PC：**

| 特性       | 2PC              | 3PC                              |
| -------- | ---------------- | -------------------------------- |
| **阶段**   | Prepare → Commit | CanCommit → PreCommit → DoCommit |
| **阻塞**   | 协调者故障时阻塞         | 超时可继续                            |
| **数据一致** | 可能不一致            | 改善但未解决                           |

**2PC流程：**

```
协调者         参与者A    参与者B
   |──Prepare───>|           |
   |──Prepare────────────────>|
   |<──Yes───────|           |
   |<──Yes───────────────────-|
   |──Commit────>|           |
   |──Commit─────────────────>|
```

### 8. 如何设计分布式锁？

**答案：**

**实现方式对比：**

| 方式            | 优点      | 缺点     |
| ------------- | ------- | ------ |
| **数据库**       | 简单      | 性能差、单点 |
| **Redis**     | 高性能     | 主从切换丢锁 |
| **ZooKeeper** | 强一致、可靠  | 性能较差   |
| **etcd**      | 强一致、高性能 | 学习成本   |

**Redis实现（Redlock）：**

```go
// 单节点版本
func Lock(key, value string, ttl time.Duration) bool {
    return redis.SetNX(ctx, key, value, ttl).Val()
}

func Unlock(key, value string) bool {
    script := `
    if redis.call("GET", KEYS[1]) == ARGV[1] then
        return redis.call("DEL", KEYS[1])
    else
        return 0
    end`
    return redis.Eval(ctx, script, []string{key}, value).Val().(int64) == 1
}

// Redlock（多节点）
func RedlockLock(key, value string, ttl time.Duration) bool {
    n := len(redisClients)
    quorum := n/2 + 1
    successCount := 0

    for _, client := range redisClients {
        if client.SetNX(ctx, key, value, ttl).Val() {
            successCount++
        }
    }

    return successCount >= quorum
}
```

```
   ↓
```

### 10. DDD和三层架构对比？

**答案：**

**三层架构：**

```
表现层(Controller)
       ↓
业务层(Service)
       ↓
数据层(DAO/Repository)
```

**DDD架构：**

```
用户接口层(Controller/API)
       ↓
应用层(Application Service)
       ↓
领域层(Domain)  ←── 核心业务逻辑
       ↓
基础设施层(Infrastructure)
```

| 对比       | 三层架构     | DDD            |
| -------- | -------- | -------------- |
| **关注点**  | 技术分层     | 业务领域           |
| **业务逻辑** | Service层 | 领域层（实体、值对象）    |
| **数据库**  | 直接操作     | 通过Repository抽象 |
| **复杂度**  | 简单       | 较复杂            |
| **适用场景** | CRUD为主   | 复杂业务           |

**DDD核心概念：**

- **实体（Entity）**：有唯一标识的对象
- **值对象（Value Object）**：无标识，通过属性定义
- **聚合（Aggregate）**：一组相关对象的集合
- **聚合根（Aggregate Root）**：聚合的入口
- **领域服务（Domain Service）**：跨实体的业务逻辑
- **仓储（Repository）**：聚合的持久化抽象

### 11. 为什么使用JWT，不使用session和cookie？

我选择 JWT，不是因为它一定比 session 更高级，而是因为它更适合前后端分离和微服务场景。Session 模式下，服务端要维护会话状态，通常要有共享存储，比如 Redis；而 JWT 是无状态的，请求里带上 token 后，各服务只要验证签名就能识别用户身份，减少了中心化会话依赖。另一方面，浏览器跨域场景下 cookie 的配置会更麻烦，比如 SameSite、credentials、跨域传递这些问题，而 JWT 放在 Authorization 头里通常更直接。  
当然 JWT 也有代价，比如主动注销、封禁、立即失效没 session 那么方便，所以实际项目里一般会配合短期 access token、refresh token，必要时再加黑名单机制。”

### 12. 四种限流算法怎么选？

**答案：**

- 固定窗口最简单，但窗口切换点容易出现瞬时双倍流量。
- 滑动窗口更平滑，统计更准确，但实现和存储成本更高。
- 漏桶适合把流量整形成稳定速率，令牌桶适合“整体限速，但允许一定突发”，工程里最常见的通常是令牌桶。
- 所以这题最好的回答不是背定义，而是说清楚：**要不要允许突发、要不要绝对平滑、实现成本能不能接受**。

### 13. 消息队列出现消息堆积时，怎么排查和处理？

**答案：**

- 我会先判断是“生产突然变快了”，还是“消费变慢了”。如果是后者，还要继续拆：是消费者挂了、下游接口慢了，还是分区不够、单条消息太重。
- 排查时重点看 backlog、消费速率、异常重试、下游响应时间和消费者线程状态。
- 临时止血的手段通常有：扩消费者、扩分区、批量消费、非核心链路先降级，必要时把特别慢的下游先隔离出去。
- 根因治理则是：提升消费幂等、减少单条消息耗时、做好重试和死信队列，避免一条坏消息把整个消费链路拖死。

### 14. 如何保证消息不丢？为什么“只靠 MQ”做不到绝对不丢？

**答案：**

- 真正的不丢是端到端问题，至少要把生产者、Broker、消费者三段都考虑进去。
- 生产者侧要有发送确认和重试，Broker 侧要有持久化和副本，消费者侧要在业务真正处理成功后再确认，并且做好幂等。
- 但即便这样，也很难说“绝对 0 丢失”，因为还会有应用崩溃、网络分区、人为误操作这些问题。
- 所以工程上更准确的说法是：**尽量做到可恢复、可重试、可补偿**，而不是轻易承诺绝对不丢。

### 15. 高热活动或热点帖子流量暴涨时，如何防止主链路被打垮？如果二级消费者阻塞怎么办？

**答案：**

- 先做热点隔离：热点数据单独缓存、提前预热、静态化能静态化，入口再配合限流、排队、削峰，避免所有请求直接打到数据库。
- 核心链路和非核心链路一定要拆开。比如主流程只保证发帖/下单成功，积分、通知、推荐这些二级逻辑走异步，不要互相拖累。
- 如果二级消费者阻塞，我会优先保证主链路，把慢消费者独立 consumer group、独立 topic 或单独线程池，不让它反压核心业务。
- 面试里一句话概括就是：**主链路保核心成功，次链路异步化 + 隔离 + 可降级**。
